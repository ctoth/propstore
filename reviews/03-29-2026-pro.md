I reviewed the Python under `propstore/` and ignored docs, tests, and config as requested.



Overall, this is an ambitious codebase with real formal-methods intent. The strengths are clear: the core domain objects are reasonably modeled, the code is literature-aware, and the project is trying to unify Dung AFs, ASPIC+, bipolar reasoning, PrAFs, ATMS, and belief merging in one system. The weakest part is that several “formal” backends have drifted from the papers they cite, while the architecture makes that drift hard to localize. The single highest-risk area is `propstore/praf\_treedecomp.py`: I found correctness bugs there, not just style issues.



The most important findings:



1\. \*\*`compute\_tree\_decomposition()` can build an invalid tree decomposition.\*\*

&#x20;  In `propstore/praf\_treedecomp.py:156-222`, bags are built from an elimination order, but the parent-selection logic at `202-219` just connects each bag to the first later bag sharing \*any\* remaining vertex. That is not enough to guarantee running intersection.



&#x20;  I reproduced a concrete counterexample with arguments `{0,1,2,3}` and defeats:

&#x20;  `{(0,0), (0,3), (1,1), (1,2), (1,3), (2,2)}`.

&#x20;  The function returned bags:



&#x20;  \* `0: {0,1,3}`

&#x20;  \* `1: {1,2}`

&#x20;  \* `2: {0,1}`

&#x20;  \* `3: {0}`



&#x20;  with tree edges `0-1-2-3`. Vertex `0` appears in bags `0,2,3`, but the path from bag `0` to bag `2` passes through bag `1`, which does not contain `0`. So the connectedness condition fails. Since the module docstring and `TreeDecomposition` docstring both claim the usual tree-decomposition properties, this is a real semantic bug, not a documentation nit. Popescu \& Wallner explicitly require vertex coverage, edge coverage, and that bags containing a vertex induce a connected subtree; the current construction violates exactly that condition. (\[KR Conference Proceedings]\[1])



2\. \*\*The “exact DP” backend returns wrong probabilities.\*\*

&#x20;  In `propstore/praf\_treedecomp.py:514-645`, I differentially compared the DP result against `propstore.praf.\_compute\_exact\_enumeration(...)` on small grounded defeat-only PrAFs. I found mismatches.



&#x20;  One reproduced example on 5 arguments with defeats:

&#x20;  `{(0,2), (1,1), (1,2), (2,1), (3,0), (3,1), (3,3), (4,1), (4,2)}`

&#x20;  gave:



&#x20;  \* exact enumeration: `{'0': 0.375, '1': 0.117431640625, '2': 0.240478515625, '3': 0.25, '4': 0.5}`

&#x20;  \* DP: `{'0': 0.375, '1': 0.2540283203125, '2': 0.2099609375, '3': 0.25, '4': 0.5}`



&#x20;  The tree decomposition for that AF is also invalid, so bug #1 is at least one root cause. I did not prove it is the only cause, but it is enough to make the backend untrustworthy.



3\. \*\*`praf\_treedecomp.compute\_exact\_dp()` has a broken public fallback path.\*\*

&#x20;  `propstore/praf\_treedecomp.py:388-405` falls back to `\_compute\_brute\_force\_fallback()` when native DP is unsupported. But `\_compute\_brute\_force\_fallback()` calls `\_compute\_exact\_enumeration(praf, semantics)` without the required keyword-only arguments declared in `propstore/praf.py:1139-1145`.



&#x20;  Result: calling `compute\_exact\_dp(praf, "preferred")` raises

&#x20;  `TypeError: \_compute\_exact\_enumeration() missing 3 required keyword-only arguments`.



&#x20;  So the public function is not robust even outside the main dispatcher.



4\. \*\*Disconnected AFs are mishandled by the tree-decomposition utilities.\*\*

&#x20;  `compute\_tree\_decomposition()` can return a forest, not a tree, on disconnected inputs; then `to\_nice\_tree\_decomposition()` at `249-259` roots a BFS at `td.root` and silently ignores unreachable bags.



&#x20;  I reproduced a case where an isolated-argument bag existed in `td.bags` but no corresponding node ever appeared in the “nice” decomposition. The main PrAF path currently avoids this by splitting connected components first (`praf\_treedecomp.py:485-509`), but the TD utilities are public and claim to operate on an AF generically. As written, they are not correct standalone.



5\. \*\*`merge\_base()` can return the wrong ancestor.\*\*

&#x20;  In `propstore/repo/branch.py:131-182`, the algorithm alternates BFS from the two tips and returns the first overlap. That does not compute the nearest common ancestor.



&#x20;  Counterexample:



&#x20;  \* branch A tip = `5`

&#x20;  \* branch B tip = `4`

&#x20;  \* parents: `5 <- {0,4}`, `4 <- {0}`



&#x20;  The correct merge base is `4` because it is itself a common ancestor and is newer than `0`. The function returns `0`. This will poison any merge logic that assumes Git-like merge-base semantics.



6\. \*\*`max\_merge()` and `gmax\_merge()` crash on unhashable values.\*\*

&#x20;  In `propstore/repo/ic\_merge.py:78-91`, `\_unique\_values()` deduplicates with a `set`. That raises on lists, dicts, or other unhashable claim values.



&#x20;  Reproduced:



&#x20;  \* `sigma\_merge({'a': \[1], 'b': \[2]})` works

&#x20;  \* `max\_merge(...)` raises `TypeError: unhashable type: 'list'`

&#x20;  \* `gmax\_merge(...)` raises the same



&#x20;  Given how much of the rest of the system still passes raw Python data structures around, this is not a purely theoretical edge case.



7\. \*\*Backend/semantics handling is internally inconsistent.\*\*

&#x20;  `propstore/core/analyzers.py:512-521` rejects `semantics=="grounded"` when `attacks != defeats`, calling it ambiguous for hybrid claim graphs. But `propstore/structured\_argument.py:81-89` silently strips `attacks` down to `defeats` and proceeds with grounded reasoning anyway. So the meaning of “grounded” depends on which backend you picked.



&#x20;  There is also a CLI mismatch: `propstore/cli/worldline\_cmds.py:74-79` exposes `--reasoning-backend claim\_graph` and `--semantics grounded` as defaults, while `analyze\_claim\_graph()` may raise and recommends semantics (`legacy\_grounded`, `d-preferred`, `s-preferred`, `c-preferred`) that are not even available in that CLI option set. That is a user-facing bug, not just an internal inconsistency.



8\. \*\*The exact-DP routing is both disabled and internally contradictory.\*\*

&#x20;  In `propstore/praf.py:329-336`, `\_public\_exact\_dp\_enabled()` always returns `False`, so the main public PrAF pipeline will never actually use exact DP. Yet:



&#x20;  \* the strategy machinery still accepts `exact\_dp` and silently downgrades it (`655-680`);

&#x20;  \* `\_exact\_dp\_supports\_query()` claims `extension\_probability` is supported (`390-399`);

&#x20;  \* `\_compute\_exact\_dp()` immediately rejects anything except credulous argument acceptance (`1216-1233`).



&#x20;  That combination says “this backend exists, but is disabled, and if re-enabled its advertised query support is wrong.” It looks like half-finished feature gating rather than a stable interface.



There are also some broader architecture and code-quality issues.



The import graph is too tangled. A static pass over the package found one \*\*44-module strongly connected component\*\* spanning `world`, `core`, validation, repo, conflict detection, and even some CLI-facing code. That is a serious layering failure. It makes formal kernels harder to test in isolation and encourages “just import from somewhere nearby” coupling.



A concrete symptom is `propstore/\_\_init\_\_.py:3-14`, which eagerly imports `BoundWorld`, `WorldModel`, and related world machinery. `propstore/world/bound.py:11-22` pulls in `world.value\_resolver`, and `propstore/world/value\_resolver.py:9` imports `ast\_equiv`. So a normal `import propstore.dung` first executes package initialization and can fail with `ModuleNotFoundError: ast\_equiv`, even though `dung.py` itself has no such dependency. I reproduced exactly that import failure. This is not just a missing dependency issue; it is an avoidable package-root coupling problem.



The project has good dataclasses in some places (`ArgumentationFramework`, `ProbabilisticAF`, `StructuredArgument`), but many important APIs still degrade back to `dict\[str, Any]` and `list\[dict]`. Examples include `SharedAnalyzerInput` in `propstore/core/analyzers.py:50-58` and the resolver interfaces in `propstore/world/resolution.py:139-148` and `216-225`. That weakens invariants, makes refactors risky, and is part of why semantic mismatches slip through.



There is also visible duplication and stale abstraction. `propstore/structured\_argument.py:3-6,49-71` says the structured-projection backend now delegates to `aspic\_bridge.build\_aspic\_projection`, but `propstore/world/types.py:125-130` still documents `structured\_projection` and `aspic` as distinct backends, and `propstore/world/resolution.py:139-292` still carries two near-duplicate resolver paths. Right now that abstraction boundary is mostly ceremonial.



On the literature side, there are three notable divergences.



First, the Popescu \& Wallner integration is not faithful. Their setup relies on valid tree decompositions and a treewidth-parameterized DP over them; the paper also distinguishes the complexity of extension probability from argument acceptability. The code’s own top-level note in `praf\_treedecomp.py:12-16` admits that it stores full active edge sets and forgotten arguments in table keys, eliminating the intended asymptotic gain, while `compute\_exact\_dp()` still advertises the paper’s complexity at `371-386`. So even before the correctness bugs, this implementation is better described as \*inspired by\* the paper than as the paper’s algorithm. (\[KR Conference Proceedings]\[1])



Second, the ASPIC+ preference layer is only a heuristic approximation. `propstore/preference.py:1-17` explicitly says the metadata heuristic is not a full ASPIC+ Def. 19 / Defs. 20-21 ordering. `propstore/aspic\_bridge.py:341-390` then constructs preferences by component-wise domination of a fixed metadata vector and hardcodes `rule\_order=frozenset()`, ignoring `defeasible\_rules`. The UI still exposes `comparison` and `link` knobs, but the implementation is not doing full structured comparison over premises and defeasible rules in the sense suggested by the cited ASPIC+ literature. (\[arXiv]\[2])



Third, the IC-merge claims are too strong for the code that exists. `propstore/repo/ic\_merge.py:7-19` says Sigma/Max/GMax satisfy various IC postulates, but the same docstring also concedes that the implementation is adapted from belief-base merging to a scalar-value domain and does not implement the integrity constraint parameter `mu`. Since Konieczny \& Pino Pérez formulate merging explicitly in the presence of integrity constraints, those postulate claims need to be narrowed or re-proved for this adaptation. 



What I would fix first, in order:



1\. Quarantine `praf\_treedecomp` as experimental only. Keep it disabled publicly, fix `compute\_tree\_decomposition()`, add a decomposition validator, and add differential fuzzing against exact enumeration for every supported semantics/query combination.

2\. Replace `merge\_base()` with a real nearest-common-ancestor computation on the commit DAG.

3\. Centralize backend capability policy: one matrix for backend × semantics × graph kind × query kind, used by CLI, analyzers, and world resolution.

4\. Make `propstore/\_\_init\_\_.py` lazy and break the large import cycles. Formal kernels (`dung`, `bipolar`, `praf`, merge operators) should not depend on world/repo/CLI layers.

5\. Decide whether `structured\_projection` and `aspic` are genuinely distinct. Either separate them again or collapse them into one backend and remove the duplicate resolver/documentation surface.

6\. Tighten the data model by replacing raw dict flows with typed records or `TypedDict`s at subsystem boundaries.



The short version: strong ideas, but the code currently overstates the faithfulness and reliability of some of its formal backends. I would trust the deterministic Dung-style pieces much more than the current exact-DP PrAF path.
