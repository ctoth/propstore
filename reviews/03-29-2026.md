# Audit Summary

Biggest confirmed issues: the repository docs overstate the formal status of the git/merge story in a few places, `docs/git-backend.md` directly contradicts the actual two-parent merge implementation, and the PrAF/worldline docs drift from the real CLI surface. Biggest non-issues: the ASPIC+ bridge is not aspirational, the Z3-based conflict detector is materially real and well tested, and the layer boundary criticism against `WorldModel.from_path()` is outdated. Biggest surprises: the ATMS and ASPIC+ claims are generally better justified by property tests than the documentation suggests, while the merge path still has a schema-validity bug for conflict provenance (`branch_origin`).

# Findings by Issue

## A. Formal-Claim Overreach / Epistemic-Model Claims
- Status: PARTIALLY CONFIRMED
- Bottom line: the repo implements a real operational analogy, not a mere metaphor, but `README.md` and `docs/semantic-merge.md` still overstate several correspondences as if they were direct formal identifications.
- Evidence
  - `README.md:39-44` says propstore "formalizes git operations as belief revision" and names branches as independent epistemic states plus IC merge operators as render policies.
  - `docs/semantic-merge.md:8-16` maps branch -> branching-time epistemic fork, merge commit -> IC merging of belief bases, branch assumption -> DATMS agent belief space, and cross-branch attack -> PAF attack.
  - `propstore/repo/merge_classifier.py:8-10,160` explicitly says the Coste-Marquis relation is analogous and "structural, not a formal isomorphism."
  - `propstore/repo/ic_merge.py:11-18` explicitly admits the implementation adapts Konieczny & Pino Perez from propositional belief bases to a scalar-value domain and does not implement integrity constraint `mu`.
  - `propstore/repo/branch_reasoning.py:20-38,41-62,65-102` implements only a `kind="branch"` assumption, `CONFLICT`-driven nogoods, and symmetric `contradicts` stances.
  - Paper inspection: `papers/Bonanno_2007_AGMBeliefRevisionTemporalLogic/pngs/page-002.png` and `page-005.png` show BU is a branching-time property; `papers/Mason_1989_DATMSFrameworkDistributedAssumption/pngs/page-011.png` supports agent-indexed assumptions; `papers/Coste-Marquis_2007_MergingDung'sArgumentationSystems/pngs/page-005.png` and `page-006.png` define PAFs over attack statuses; `papers/Konieczny_2002_MergingInformationUnderConstraints/pngs/page-003.png`, `page-004.png`, `page-012.png`, `page-015.png`, `page-017.png` define IC merging over propositional profiles under `mu`.
- Analysis
  - `branch <-> epistemic state`: implemented operational analogy / constrained formalization. Branches do carry operationally separate knowledge states, but no code instantiates Bonanno's epistemic-temporal semantics.
  - `merge classification <-> PAF conflict relation`: structural analogy only. The code translates claim-level conflict classes into downstream attack/no-attack/ignorance behavior; it does not instantiate a PAF completion semantics.
  - `merge commit <-> IC merge`: overclaimed in current docs. The merge commit preserves both sources with provenance, which is useful as a merge substrate, but it is not itself the IC merge operator from Konieczny & Pino Perez.
  - `branch assumptions <-> DATMS agent-space assumptions`: merely an implemented operational analogy. The code uses branch-tagged assumptions and nogoods; it does not model DATMS agents or distributed message/update semantics.
- Recommended documentation wording
  - Replace "formalizes git operations as belief revision" with "implements a git-backed operational analogue of belief revision and belief merging."
  - Replace direct table identities with "models as", "treats as", or "adapts from" unless the code and tests actually enforce the cited postulates.
- Recommended code change, if any
  - No code change required to support the weaker wording.
- Confidence: High

## B. “No Merge Commits” Contradiction
- Status: CONFIRMED
- Bottom line: `docs/git-backend.md` contains a real contradiction. The code absolutely creates two-parent merge commits.
- Evidence
  - `docs/git-backend.md:12` says: "No merge commits. Strictly linear history per branch. Each commit has exactly one parent."
  - `propstore/repo/merge_commit.py:52-60` documents "Create a merge commit with two parents."
  - `propstore/repo/merge_commit.py:179-205` builds a Dulwich `Commit` and sets `commit.parents = [left_sha.encode("ascii"), right_sha.encode("ascii")]`.
  - `docs/semantic-merge.md:102-121` also describes merge commits as two-parent commits.
  - Paper inspection: `papers/Bonanno_2007_AGMBeliefRevisionTemporalLogic/pngs/page-005.png` sharpens why this matters, because BU is about unique predecessor history.
- Analysis
  - The only coherent reading is: history is linear within each branch between merges, but the global repository history is a DAG with integration merges.
  - That distinction is not what `docs/git-backend.md:12` says. As written, it is false.
- Recommended documentation wording
  - "Each ordinary branch commit has one parent. Integration uses two-parent merge commits, so per-branch evolution is linear but the repository history is a DAG."
- Recommended code change, if any
  - None; the docs should be fixed.
- Confidence: High

## C. IC Merge Operator Validity
- Status: PARTIALLY CONFIRMED
- Bottom line: the implementation preserves recognizable Sigma/Max/GMax aggregation patterns, but over an adapted scalar/categorical value domain, without `mu`, and without direct test coverage of the original IC postulates in their native formal setting.
- Evidence
  - `propstore/repo/ic_merge.py:1-18` states the adaptation from propositional belief bases and notes missing `mu`.
  - `propstore/repo/ic_merge.py:46-69,89-111,114-142` implements sum, max, and lexicographic sorted-distance aggregation.
  - `docs/semantic-merge.md:173-191` and `propstore/repo/ic_merge.py:52,125` claim IC0-IC8 satisfaction.
  - `tests/test_ic_merge.py` verifies useful proxy invariants: unanimity, majority behavior, syntax-independence under profile relabeling, arbitration insensitivity to multiplicity, and `gmax` refinement of `max`.
  - Paper inspection: `papers/Konieczny_2002_MergingInformationUnderConstraints/pngs/page-003.png` and `page-004.png` define IC0-IC8 and `mu`; `page-012.png`, `page-015.png`, and `page-017.png` define Sigma/Max/GMax on propositional interpretations.
- Analysis
  - C1: recognizable faithfulness is partial, not full. The aggregator shapes are faithful; the domain and distance function are adapted.
  - C2: the operators are defined over per-concept claim values, not propositional belief bases.
  - C3: there is no code-level analogue of integrity constraint `mu` yet.
  - C4: tests cover adapted invariants, not literal IC postulate proofs in the paper's setting.
  - C5: the code acknowledges the adaptation more clearly than the docs do.
  - C6: the papers weaken the current strongest documentation wording.
- Recommended documentation wording
  - "Implements Sigma/Max/GMax-inspired render-time aggregation over claim-value profiles, preserving the aggregation schemes while adapting the distance domain away from propositional belief bases. Integrity constraints (`mu`) are not yet implemented."
- Recommended code change, if any
  - Either implement a real constraint channel for `mu`, or stop claiming IC0-IC8 satisfaction as a literal property.
- Confidence: High

## E. ATMS Claims
- Status: PARTIALLY CONFIRMED
- Bottom line: the implementation is genuinely ATMS-style and more thoroughly tested than expected, but it is still a bounded exact-support engine, not a full de Kleer ATMS, and `docs/atms.md` overstates this with "complete map of the belief space."
- Evidence

  - `docs/atms.md:41` says the engine grants exact support only.
  - `docs/atms.md:214-228` explicitly says the backend is not a full de Kleer runtime manager, not AGM revision, and not ASP-based stability.
  - `propstore/world/atms.py:1-8` describes "ATMS-style exact-support propagation plus bounded replay."
  - `tests/test_labels_properties.py` property-tests label minimality, antichain behavior, commutativity/associativity, and nogood pruning.
  - `tests/test_atms_engine.py:959-998,1204-1266,1307-1429,1620-1766` directly tests exact-support-only behavior, bounded additive futures, why-out distinctions, relevance, stability, interventions, and CLI output.
  - Paper inspection: `papers/deKleer_1986_AssumptionBasedTMS/pngs/page-017.png` and `page-018.png` support label/nogood/minimal-environment language; they do not justify "complete map" for bounded future replay. `papers/Mason_1989_DATMSFrameworkDistributedAssumption/pngs/page-011.png` supports assumption-space reasoning, not a full distributed ATMS runtime here.
- Analysis
  - E1: yes, this is ATMS-style bounded replay rather than a full ATMS.
  - E2: labels, nogoods, minimal environments, and support propagation are real and test-backed.
  - E3: "exact support only" is enforced in code and tests, not just described.
  - E4: stability/relevance/intervention claims are well supported, but bounded to declared additive queryables.
  - E5: it behaves like a precise diagnostic overlay over compiled artifacts, not a canonical full truth-maintenance runtime.
  - E6: de Kleer supports the label/nogood language; the docs should stay careful about scope.
- Recommended documentation wording
  - Replace "complete map of the belief space" with "an exact-support map over current environments plus bounded additive replay over declared future/queryable assumptions."
- Recommended code change, if any
  - None required unless the project wants full dynamic ATMS features.
- Confidence: High

# Paper Evidence Summary

Well supported by local papers:
- ASPIC+ attack/defeat/preference/transposition wording is strongly supported by `papers/Modgil_2018_GeneralAccountArgumentationPreferences/` and its PNG pages.
- ATMS label/nogood/minimal-environment wording is supported by `papers/deKleer_1986_AssumptionBasedTMS/`.
- Z3-style conflict semantics are not paper-driven, but the repo's claim that `PHI_NODE` corresponds to ignorance/non-attack is consistent with the PAF framing in `papers/Coste-Marquis_2007_MergingDung'sArgumentationSystems/`.

Claims relying on analogical use of cited literature:
- git branch -> epistemic state / branching-time fork
- branch assumptions -> DATMS agents
- merge classification -> PAF relation

Claims where the papers weaken the repo's current wording:
- Konieczny & Pino Perez 2002 does not justify saying the current scalar-value operators literally satisfy IC0-IC8 in the original formal setting, because the paper's domain is propositional belief bases under integrity constraint `mu`.
- Bonanno 2007 sharpens the mismatch between BU language and the existence of two-parent merge commits in the actual repo DAG.

PNG inspection materially changed or sharpened conclusions:
- Konieczny PNGs made the missing `mu` and original formal domain impossible to ignore.
- Coste-Marquis PNGs sharpened the conclusion that the merge classifier is a structural analogy, not a direct PAF instantiation.
- de Kleer PNGs strengthened the case that the ATMS wording should stay focused on labels/nogoods/minimal environments, not "complete belief-space" rhetoric.
- Modgil/Prakken PNGs strengthened the conclusion that the ASPIC+ implementation is closer to the literature than initial skepticism suggested.
