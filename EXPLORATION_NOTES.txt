### 2. Build Pipeline Steps (sequential)
Step 0: Form schema validation
Step 1: Concept validation
Step 1b: Context validation
Step 2: Claim validation (builds concept_registry here)
Step 3: Sidecar build via build_sidecar()
Step 4: Summary via WorldModel

### 3. Core Sidecar Build (build.py:67)
- Content hashing: _content_hash() hashes concepts/claims/contexts/forms/justifications/sources/stances YAML + schema
- Caching: If hash matches existing .hash file, skip rebuild (unless force=True)
- Concept registry: Dict mapping IDs->enriched concept data with _form_definition injected
- Sidecar creation order:
  1. Load & create tables (concepts, claims, forms, contexts, relationships, parameterizations)
  2. Build concept FTS
  3. Populate claims
  4. **CONFLICT DETECTION** (calls detect_conflicts + detect_transitive_conflicts)
  5. Build claim FTS
  6. Populate stances & justifications

### 4. Conflict Detection (conflict_detector/orchestrator.py:21-79)
- detect_conflicts() builds CEL registry, injects synthetic "source" concept
- Runs: parameter, measurement, equation, algorithm conflict detectors
- Uses Z3ConditionSolver for condition solving
- Stores results in conflict_witness table

### 5. Data Structures
Concept Registry: dict[str, dict] mapping IDs/canonical/logical/aliases -> enriched concept data
Sidecar tables: concept, claim_core, claim_numeric/text/algorithm_payload, relationship, parameterization, parameterization_group, form_algebra, conflict_witness, justification, claim_stance, context hierarchy

### 6. ASPIC+ Bridge & Claim Graph
- aspic_bridge.py: T1-T7 translations from propstore to ASPIC+ (claims->literals, justifications->rules, stances->contrariness, etc)
- claim_graph.py: Builds argumentation framework + computes justified claims via grounded/preferred/stable semantics
- NOT called during pks build command
- Used post-build by WorldModel & backends

### 7. WorldModel (Post-Build)
- File: world/model.py
- Instantiated in build CLI to prove roundtrip
- Provides: .stats(), .conflicts(), .close()

### 8. Validation vs Build
- Validation FIRST (Steps 0-2), build is AFTER validation succeeds
- Validation is prerequisite: build fails fast if validation fails
- Independent systems: validate.py vs sidecar/build.py
- Build does NOT re-run validation
