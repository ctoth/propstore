## Key Definitions

- **AND/OR graph**: Represents relationships between subproblems and goals explored during problem solving. Subproblems arise for two reasons: (1) several operators might solve a particular problem (disjunctive/OR), (2) operators may decompose a problem into subproblems which must all be solved (conjunctive/AND). An and-node always has an or-node as its parent; children of an and-node are always or-nodes (subproblems). *(p.250)*

- **Control vocabulary for AND/OR graph**: Nodes have properties: `expanded` (P, IN/OUT), `open` (P, IN/OUT), `relevant` (P, IN/OUT), describing whether a problem has been expanded, is still open, and is relevant to the original problem. *(p.250)*

- **Operator protocol**: The format `(solution-of (P) (A1) ... (An))` means the solution of problem P is the expression A, where each solution-of statement must be appropriately justified in terms of the solutions of subproblems and the operator itself. *(p.254)*

- **Expanded assertions**: The first thing an operator does is look for expanded assertions, since such assertions are the first indication that a node is being worked on. *(p.254)*

- **Suggestion**: Has the form `(suggest-for (P) (OpInst))` where P is the problem that would be solved by the successful application of operator instance OpInst. *(p.255)*

- **LTMS labels**: The LTMS permits only two labels: `:IN` and `:OUT`, plus a set of enabled assumptions. A node is labeled `:TRUE` if it is derivable from C and the enabled assumptions. A node is labeled `:FALSE` if its negation is derivable from C and the enabled assumptions. A node is labeled `:UNKNOWN` otherwise. *(p.285-286)*

- **LTMS node properties**: Three types -- premise, contradiction, or assumption. A premise is regarded as if it is the only node in some inference engine-supplied clause. The LTMS has no notion of a contradiction node. An assumption is a node whose belief may later be changed by an explicit inference engine operation. *(p.287)*

- **Well-founded explanation**: For a node c's label, consists of a sequence of steps S1, ..., Sn. Each step consists of a 4-tuple: n (conclusion) (antecedents) (reason). Two kinds of steps: assumption steps (where x is a literal corresponding to an enabled assumption) and derivation steps (where x is any literal, A is a set of antecedent steps, and x follows from clause c and the consequences of the steps mentioned in A). *(p.292-293)*

- **BCP (Boolean Constraint Propagation)**: Labels every symbol `:TRUE`, `:FALSE`, or `:UNKNOWN`. BCP is provided with an initial labeling of assumptions A. All remaining symbols are initially labeled `:UNKNOWN`. The LTMS algorithm labels symbols represented by nodes, not literals. *(p.289)*

- **Clause**: A disjunction of literals with no repeated or complementary literals. Let C be the set of clauses defined over S that the inference engine has supplied. *(p.288)*

- **Nogood clause**: Adding the nogood clause to C helps avoid recurrence of the same contradiction and therefore saves subsequent backtracking. *(p.299)*

## Algorithms and Pseudocode

- **JSAINT controller algorithm** *(p.258)*:
  1. Check the original problem.
  2. If SOLVED, then halt and report success.
  3. If FAILED, then halt and report failure.
  4. If the agenda is empty, then halt and report failure.
  5. If the resource allocation has been exceeded, report failure.
  6. Select the simplest subproblem on the agenda and work on it.
  7. Return to step 1.

- **Algorithm 9.1 BCP** *(p.290)*:
  1. Repeat until S is empty.
  2. Pop clause c off of S.
  3. If c is unit open, then compute the label of the `:UNKNOWN` literal l which will satisfy c. Call SET(l).
  4. Otherwise, do nothing.

- **Algorithm 9.2 SET(l)** *(p.290)*:
  1. Set the label of the node of l to make the literal l `:TRUE`.
  2. Push every unit open clause which contains ~l on to S.
  3. Push every violated clause which contains ~l on to S.

- **Algorithm 9.3 ADD(c, l)** *(p.291)*:
  1. Add c to l.
  2. If c is violated, then push c on to V and signal the inference engine that a contradiction has occurred.
  3. If c is unit open, then push c on to S.
  4. Call BCP.
  5. If V is not empty, then signal the inference engine that a contradiction has occurred.

- **Algorithm 9.4 ENABLE-ASSUMPTION(n)** *(p.291)*:
  1. Call SET(l).
  2. Call BCP.
  3. If V is not empty, then signal the inference engine that a contradiction has occurred.

- **FTRE rules for AND/OR graph relationships** *(p.261)*: Lisp code showing rules for managing parent-child relationships in AND/OR graphs, including handling of `solved`, `failed`, `open`, `relevant` status propagation through the graph. Key rules include propagation of failure from children to parents and solution propagation.

- **defIntegration macro** *(p.256-257)*: Defines integration operators. Syntax: `(defIntegration (name) (pattern) (stuff))` where `(name)` is a symbol naming the operator, `(pattern)` is the left hand side (what the operator matches), and `(stuff)` is a list of keywords and their arguments including `:TEST`, `:SUBPROBLEMS`, and `:RESULT`.

## Data Structures

- **AND/OR graph nodes** *(p.250)*: Two types of nodes:
  - **or-nodes**: Represent problems/goals. Properties: `expanded`, `open`, `relevant` (each IN or OUT).
  - **and-nodes**: Represent operator applications. Properties: `expanded`, `open`, `relevant` (each IN or OUT).
  - Parent of an and-node is always an or-node; children of an and-node are always or-nodes.

- **LTMS labels mapping to JTMS** *(p.286-287)*:
  - JTMS `:IN` corresponds to LTMS `:TRUE`
  - JTMS `:OUT` corresponds to LTMS `:FALSE`
  - JTMS `:OUT` also corresponds to LTMS `:UNKNOWN`
  - The LTMS `:IN` corresponds to the JTMS `:IN`
  - The LTMS `:OUT` corresponds to the LTMS `:UNKNOWN`

- **LTMS label mapping table** *(p.287)*:
  | | P IN | P OUT |
  |---|---|---|
  | ~P IN | (IMPOSSIBLE) | :FALSE |
  | ~P OUT | :TRUE | :UNKNOWN |

## Equations

- **Encoding A v B v C in JTMS** *(p.282)*: Since the JTMS cannot directly represent "B is false," it can be approximated by creating another node B-bar which is IN when B is false. To prevent B from being both true and false simultaneously, a contradiction must be signaled when both are in: `B : B-bar = _|_`. Full encoding requires six nodes and six justifications. *(p.282-283)*

- **Nogood representation** *(p.284)*: The JTMS has inferred the formula `~(A1 ^ A2 ^ ... ^ An)` which is equivalent to the clause (a nogood): `~A1 v ~A2 v ... v ~An`. But the JTMS cannot directly represent or use this. *(p.284)*

- **Encoding propositional formulas as clauses** *(p.294-295)*:
  - Eliminate equivalences: replace all occurrences of `x = y` with `(x => y) ^ (y => x)`
  - Eliminate implications: replace all occurrences of `x => y` with `~x v y`
  - Eliminate all taxonomic formulas on x1, ..., xn by replacing them with the conjunction of xi -> ... -> xn and for all i != j, ~(xi ^ xj)
  - Repeatedly use the rule `~~x = x` and DeMorgan's laws
  - Repeatedly use distributivity, associativity, and commutativity to produce a conjunction of disjunctions

## Key Claims and Properties

- **JTMS bookkeeping advantage**: By implementing AND/OR graphs as collections of assertions woven together via justifications in the problem solver's dependency network, the TMS takes over a substantial portion of the bookkeeping and provides the grist for explaining what happened during problem solving and why. *(p.251)*

- **Incremental open/relevant tracking**: The method for tracking `open` and `relevant` status is incremental in exactly the right way, since each parent can be added independently, and any open parent ensures the relevance of the child. *(p.253)*

- **JSAINT avoids redundant solutions**: When JSAINT first requests a goal, it is complete, and does not process solutions added later. This choice allows representation of the parental relationships of or-nodes quite simply, by creating an assertion that explicitly lists the disjuncts. *(p.252)*

- **Retraction handling**: Two solutions for handling what happens when the inputs to a problem's open statement change: (1) Add a rule that explicitly retracts a problem's open statement when it is either solved or failed; (2) Install a `:RETRACT` between `open` and `solved` and between `open` and `failed`, and ensure that the contradiction handler will retract the appropriate open statement in such conflicts. The second solution is more general and preferable. *(p.260)*

- **BCP is sound**: BCP never labels a node `:TRUE` or `:FALSE` when it shouldn't, nor does it signal a contradiction when there isn't one. *(p.296)*

- **BCP is logically incomplete**: BCP sometimes fails to label a node `:TRUE` or `:FALSE` when it should. This is called formal incompleteness. BCP on clauses is equivalent to the circuit value problem which is P-complete (not NP-complete). *(p.296)*

- **BCP relaxation incompleteness**: BCP is also logically incomplete in that it sometimes fails to detect contradictions. *(p.296)*

- **BCP is literal-complete for positive literals**: BCP is known to be refutation complete and literal-complete for positive literals only. In practice, BCP is useful for most Horn clause sets. *(p.297)*

- **LTMS vs JTMS**: The JTMS `:IN` corresponds to the LTMS `:TRUE`, and the JTMS `:OUT` corresponds to the LTMS `:UNKNOWN`. The computational complexity of the LTMS and JTMS is the same in this case. Therefore, there is no important reason to use a JTMS over an LTMS other than it being a bit simpler. *(p.297)*

- **BCP-based LTMS more efficient**: When using the encoding of Section 9.1.1 to represent arbitrary clauses, the BCP-based LTMS is more efficient because the same literal will occur more often in the equivalent justification structures than in the original clauses. *(p.297-298)*

- **Nogood clause prevents recurrence**: Adding the nogood clause to C helps avoid the recurrence of the same contradiction and therefore saves subsequent backtracking. *(p.299)*

## Design Decisions and Rationale

- **AND/OR graphs via TMS justifications**: Rather than implementing AND/OR graphs as special-purpose data structures, JSAINT implements them as collections of assertions woven together via justifications in the problem solver's dependency network. This lets the TMS take over substantial bookkeeping and provides grist for explaining problem solving. *(p.251)*

- **JSAINT's difficulty estimator**: JSAINT uses the sum of the maximum depth of the problem expression plus the number of symbols in it as an estimate of difficulty, since larger problems typically require more work to decompose into directly solvable pieces. Given a `goal-or-set` assertion, it simply returns the answer by looking for an `in-value-of` assertion. *(p.259)*

- **Operator definition abstraction level**: In the least abstracted implementation, the authors would force users to write explicit FTRE rules. At the most abstracted level, they would implement a parser to process mathematical statements in standard notation. JSAINT chose a middle ground: selecting a set of constructs that are very easy to implement yet still shield users from most of the internals of the system. Operators are defined by the form `defIntegration`. *(p.256-257)*

- **LTMS over JTMS motivation**: The JTMS justifications are very limited in expressive power. Logically, JTMS justifications are simply definite clauses. Many applications need to express more than just definite clauses. Logic-based truth maintenance systems (LTMS) were invented to overcome this limitation. The LTMS generalizes the JTMS to allow any propositional clause as input. *(p.281)*

- **LTMS shift in perspective**: The shift from JTMS to LTMS is also a shift of point of view: from regarding the TMS as a simple device for recording the inference engine's computations, to seeing it as a device for carrying a substantial inferential burden itself. If the TMS is to be a central component of the inferential system, it pays to make it more and more powerful. *(p.285)*

- **BCP chosen over full resolution**: Although BCP is an attractive algorithm, it does not fully meet the logical specification. BCP is logically incomplete (NP-complete to fully satisfy). The authors analyze two distinct manifestations of logical incompleteness: formal and relaxation. *(p.296)*

- **Search strategy for handling logical incompleteness**: Having some node labeled `:UNKNOWN` is not necessarily a symptom of BCP's logical incompleteness -- the inference engine may not have supplied enough information to logically determine the labels of all nodes. In general, BCP cannot distinguish between its own logical incompleteness and simply having insufficient information. *(p.298)*

## Implementation Notes

- **JSAINT uses FTRE**: JSAINT uses FTRE (a forward-chaining rule engine using Rete) as its inference engine and a JTMS for dependency tracking. The combination provides pattern-directed inference with dependency-based bookkeeping. *(p.251)*

- **Agenda-based control**: JSAINT's controller is organized around an agenda, which holds the subproblems and goals that have not yet been expanded. Each agenda item includes an estimate of difficulty, to allow JSAINT to select the least difficult task to try next. *(p.258)*

- **Resource bounds**: JSAINT supports resource bounds including CPU time and memory. Implementation-dependent bounds include upper bounds on the length of a derivation or the size and complexity of subexpressions considered. *(p.258-259)*

- **LTMS implementation structure**: The LTMS needs three fundamental tasks: (1) to provide labels for nodes, (2) to detect contradictions, and (3) to provide explanations for the labels. *(p.288)*

- **BCP stack-based algorithm**: BCP is straightforward. It maintains a pending stack S of clauses to examine. It also maintains a set of violated clauses V which it uses in signaling the inference engine about contradictions. BCP processes the clauses on the stack one at a time, relabeling `:UNKNOWN` nodes to be `:TRUE` or `:FALSE` if possible. *(p.290)*

- **Clause encoding for LTMS**: To convert a propositional formula into an equivalent set of clauses, a technique from theorem proving is used. The formula is converted into conjunctive normal form and every resulting conjunct is then added to the database as a clause. *(p.294)*

- **Assumption handling in BCP**: Assumptions enabled by calling ENABLE-ASSUMPTION, which calls SET(l) and then BCP, signaling contradictions if V is not empty. *(p.291)*
