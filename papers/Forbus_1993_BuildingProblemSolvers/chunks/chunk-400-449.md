## Key Definitions

- **Heat flow process definition as LTRE rule**: The heat flow process definition of Figure 11.4 is translated into an LTRE rule that instantiates the logic of the QP description. The rule checks for the existence of heat paths, connected objects, temperature quantities, and creates the appropriate flow rate, Ds values, and influence statements. *(p.400)*

- **Influence resolution**: Resolving influences is the trickiest part of QP implementation. It requires making appropriate closed-world assumptions, avoiding causality cycles, and squeezing as many derivations as possible out of minimal information. The procedure `resolve-influences` is the entry point. *(p.401)*

- **Causal ordering**: The main job of `setup-ir` is to create constraints for the sets of influences on each quantity. It does this by first calling `qr-ize-all` to execute any pending pattern-directed rules, then ensuring that all "above" conclusions have been drawn. It then uses LTRE's closed-world-set mechanisms to create constraints for the sets Dis and IIs for each quantity. The set Dis are the direct influences on a quantity and the set IIs are the indirect influences on a quantity. *(p.402)*

- **`resolve-dis-on` and `resolve-iis-on`**: Procedures dispatched to different procedures according to the kind of influence a quantity is subject to. Direct and indirect influences operate under different constraints, handled by `resolve-dis-on` and `resolve-iis-on`, respectively. `resolve-dis-on` handles direct rules; `resolve-iis-on` handles the uninfluenced case and enforces the exclusion of mixed influence types (see Section 11.4.4). *(p.403)*

- **`resolve-dis-on` procedure**: Begins by sorting the rate parameters according to their signs, figuring out the sign contribution of an influence requires modifying the sign of the constraint by the sign of the influence statement itself, to ascertain the actual direction of effect. A positive flow rate and an `I+` statement leads to a positive effect; a positive flow rate and an `I-` statement leads to a negative effect. Stores in each set both the name of the number and the antecedents corresponding to the sign determined for it. Sorting the antecedents greatly simplifies computing justifications for the results. If a sign isn't known, no particular determination can be made, and so `analyze-di-on` immediately returns, providing `:UNKNOWN` as the reason for failure and the list of parameters whose signs were not established. Otherwise, `analyze-di-on` is used to suppose conclusions from the rate information. *(p.403-404)*

- **`analyze-di-on`**: Handles the case where a quantity is influenced in both directions. It begins by looking for opposing direct influences which may be canceled, since direct influences combine by addition. It starts by looking for identical rate parameters (calls to `cancel-vis-identity`) and then looks for opposing parameters of equal magnitude (calls to `cancel-vis-eq`). If any cancelations are made, `analyze-di-on` is called recursively to see if the simpler problem has an unambiguous solution. Otherwise, the fact that direct influences are ambiguous is duly noted, by calling `justify-ir-ambig`. *(p.404)*

- **`resolve-iis-on`**: Analogous to `resolve-dis-on` but for indirect influences. The sorting by signs of constituents is identical to the sorting by signs of rate parameters, as is the necessity for abandoning the attempt if some constituent's sign is unknown. The unambiguous and disjunctive cases in `analyze-iis-on` have the same logic as their counterparts for direct influences. The major difference is in the final case: `analyze-iis-on` concludes with a `resolve-ambig`, corresponding to a clear ambiguity. Since we do not know how qualitative proportionalities combine, we cannot invoke cancelation or any other strategy which rests on more knowledge. Our only chance is to note the ambiguity, which is done by a call to `justify-ir-ambig`. *(p.404)*

- **`justify-ir-ambig`, `justify-ir-result`, `exclude-ir-result`**: Procedures that conclude the conclusions reached by the routines above. They share a common structure of antecedents, processed by the procedure `ir-antecedents`. The bit actor starts out as the list of influence statements and is incremented with any information added by cancelation in the case of direct influences. *(p.404)*

- **Environment (ATMS)**: A set of assumptions. A node holds in an environment if, when all the assumptions of the environment are enabled in a TMS, the node is labeled `:IN`. Thus, in the previous example, node h holds in 6 environments. A nogood is an environment in which some contradiction holds. A consistent environment is one which is not nogged. A context of an environment is the set of nodes that hold in the environment. *(p.444)*

- **ATMS label**: Unlike the JTMS and LTMS, an ATMS label is a complex data structure. The label is no longer a single atom such as `:IN` or `:OUT`, because these only make sense with respect to some current context. Intuitively, the ATMS label consists of a parsimonious description of the consistent contexts in which the node holds. *(p.443)*

- **ATMS label properties** (four properties): *(p.448)*
  1. **Soundness**: n holds in each E_i
  2. **Consistency**: {} cannot be derived from any E_i (given C)
  3. **Completeness**: Every consistent environment E in which n holds is a superset of some E_i
  4. **Minimality**: No E_i is a proper subset of any other

- **ATMS node types** *(p.442)*:
  - A **node** is a *premise* if the inference engine has provided it with a justification with no antecedents. Premises are therefore not distinguished from other nodes in the architecture.
  - A **node** is a *contradiction* if the inference engine has indicated that it can never hold. In the JTMS, if a contradiction node becomes believed, the inference engine is signaled (presumably to retract some assumption to remove the contradiction). The ATMS, which has no notion of current context, does not signal the inference engine when it detects a contradiction.
  - A **node** is an *assumption* if the inference engine wishes to use it in the second argument (to `in-node?`). The ATMS will then build the appropriate data structure to make it easy to tell whether a node follows from an arbitrary set of assumptions. In the JTMS or LTMS, making all nodes assumptions incurs little overhead. However, marking a node as an assumption tells the ATMS that the node can appear in the second argument of `in-node?` and consequently labels (and sets) can grow exponentially in the number of possible assumptions. Thus it is incumbent on the inference engine to introduce as few assumptions as possible.

- **ATMS justification**: Identical syntax as JTMS justifications: a justification consists of a consequent which follows from antecedents accompanied by an informant describing the deduction (which the ATMS never looks at). *(p.443)*

- **Logical specification for ATMS**: The set of ATMS nodes defines a set of propositional symbols S. A subset of those symbols are marked as assumptions: A subset of S. As with the JTMS, every ATMS justification is directly encoded as a definite clause. For every contradiction node n, we add a unit clause `~n` to indicate that n does not hold. Let C be the set of clauses obtained. We presume C is consistent. An environment E is a subset of the assumptions, E subset of A. *(p.447)*

## Algorithms and Pseudocode

- **Influence resolution algorithm** (`resolve-influences`): The basic steps and subroutines that implement influence resolution: *(p.401)*
  1. Find the current contextual set of influences on each quantity by fetching the influences currently believed to be active (these are the only ones the bulk of `setup-IR` needs).
  2. Use the influences to impose a causal ordering on the quantities (`find-influence-ordering`, which is called from `setup-IR` and whose results are cached in `tgizmo-influence-order`).
  3. Attempt to resolve each quantity (`resolve-influences-on`). Record any ambiguous outcomes, and return this list as the result (unknowns).

- **`find-influence-ordering`**: Constructs the causal ordering by a simple propagation algorithm. An initial ordering is provided by assigning a rank of zero to directly influenced parameters and a rank of 1 to indirectly influenced parameters. The procedure `update-influence-table-ordering` loops through the set of parameters, increasing the rank of a quantity to be the highest rank of its constituents, plus one. Since a directly influenced parameter has no constituents its rank remains zero, and the rank of an indirectly influenced parameter must converge because the set of qualitative proportionalities is loop free. *(p.402)*

- **`resolve-completely`**: Provides the option of creating choice sets for the unknown Ds values by returning Ds values returned by `resolve-influences` using `make-Ds-choice-sets` and performing a dependency-directed search. As with `search-pps`, `resolve-completely` is given a procedure which is then executed for each consistent solution to the influences. *(p.405)*

- **Inequality reasoning** (`install-compare-constraints`): Procedure that is called to check whether an ordinal statement is memorized (i.e., interned in the LTRE database); if needed, it is called to install `install-compare-constraints-if-needed`. *(p.406)*

- **`check-comp-cycle`**: Procedure used to look at each cycle of comparisons to see if more information can be derived from it. Returns to `check-comp-cycle` after first considering how cycles get created. Uses `find-compare-less-than-or-eq` procedure: a comparison between two numbers N1 and N2 is invoked whenever a comparison is first interned in the LTRE database for the first time. The new comparison is effectively added as an edge to the graph of comparisons, and this procedure finds all cycles containing this new edge. It operates by a breadth-first exploration of the comparison graph, starting from all other comparisons involving N1. `find-comps-one-set` provides this service, filtering the results of a `fetch` to find the appropriate inequalities and extracting the other number. Cycles not ending in N2 are ignored, since they will have been found in previous invocations of this procedure. Each element on the queue is a list of numbers, to simplify cycle detection, but for further processing it is more convenient to represent cycles by a list of pairs of numbers. This format conversion is carried out by `make-comp-cycles`. *(p.407-408)*

- **`check-comp-cycle` case analysis**: Begins by sorting the comparisons of the cycle into six sets, each corresponding to a possible labeling of the relationship between a pair of numbers, in order to make case analysis. Three main cases: *(p.408-409)*
  1. More than one comparison in the cycle is unknown. Nothing can be inferred.
  2. Something is known about all comparisons in the cycle. Requires further analysis (transitivity or opportunities to further constrain a comparison).
  3. Exactly one comparison in the cycle is unknown. Requires further analysis, since a relationship for the unknown comparison may be derivable.

- **ATMS label update algorithm**: Operates by repetitively ensuring that each node's label is locally correct. The locally correct label for node n could be computed as follows: Let L_ik be the label of the ith node of the kth justification for some node n. *(p.448)*
  1. Compute a tentative label: E' = union of {j_i | e_i in L_ik} for all justifications k
  2. Remove from E' all nogoods and environments subsumed by others in E'.

- **Simple ATMS label propagation algorithm**: Propagates label changes throughout the dependency network. *(p.449, Figure 12.5)*

## Data Structures

- **LTRE rule for heat flow process**: Encodes the complete QP description of a heat flow process, including preconditions (existence of heat paths, connected objects, temperature quantities) and consequences (heat-flow rate quantity, Ds values, influence statements connecting temperatures). *(p.400)*

- **Influence data structures**: For each influence, the system stores both the name of the number and the antecedents corresponding to the sign determined for it. Sorting the antecedents greatly simplifies computing justifications for the results. *(p.403)*

- **LTRE database for inequality reasoning**: Ordinal statements are memorized/interned in the LTRE database. The `install-compare-constraints-if-needed` procedure checks whether an ordinal statement is memorized (interned in the LTRE database). *(p.406)*

- **ATMS node label**: A set of environments {E_1, ..., E_n} called the label, having four properties: Soundness, Consistency, Completeness, Minimality. Given this data structure, the ATMS can efficiently answer the query whether n holds in a consistent environment E by checking whether E is a superset of some E_i. To easily enforce the consistency property, the ATMS also maintains a database of subsumed nogoods. *(p.448)*

- **Basic entity types for QP domain theory** (Figure 11.13): *(p.418)*
  - `(defentity (Physical-Stuff ?sub) ...)` -- quantities heat and temperature
  - `(defentity (Container ?cont) ...)` -- quantities heat and temperature with constraint ≥ 0
  - `(defentity (Temperature-Source ?psub) ...)` -- quantity temperature with constraint ≥ 0

- **Contained-Stuff data structure**: Represents a substance in a particular phase inside a container. The `contained-stuff-existence` rule ensures that for every combination of phase and substance and container, there is a quantity denoting the amount of substance in that phase inside that container, and furthermore that amount can never be negative. *(p.419)*

## Equations

- **Causal ordering rank computation**: For a directly influenced parameter, rank = 0. For an indirectly influenced parameter, rank = max(rank of its constituents) + 1. Since the set of qualitative proportionalities is loop free, ranks converge. *(p.402)*

- **ATMS label computation**: *(p.448-449)*
  - Tentative label: $E' = \bigcup_k \{ j_i \mid e_i \in L_{ik} \}$ where $L_{ik}$ is the label of the $i$th node of the $k$th justification for node $n$
  - Then remove from $E'$ all nogoods and environments subsumed by others

- **Monotonicity of ATMS labels**: If a node follows from an environment, then it follows from any superset of that environment, and hence it is unnecessary to represent the supersets in the label. Hence the label for node h is: {(A, D), (B, D), (A, B, D), (A, C), (A, B, D, E)} which we reduce to: {(A, D), (B, D), (A, C)}. *(p.444-445)*

## Key Claims and Properties

- **Influence resolution is the trickiest part**: "Resolving influences is the trickiest part of this implementation. It requires making appropriate closed-world assumptions, avoiding causality, and squeezing as many derivations as possible out of minimal information." *(p.401)*

- **Closed-world assumptions can lead to contradictions**: Closed-world assumptions can lead to contradictions, and so `IR-Contradiction-handler` is provided to detect and retract such problems. The procedure `extract-IR-Ds-ls` is also provided to "clear the decks" after an analysis, thus reducing the need for contradiction handling. *(p.402)*

- **TGIZMO process structure integration**: "Since the laws of influence resolution are both fixed and central to the operation of the system, we decided to make them as fast as possible. This is the natural migration path when building systems: to operate on problems that live near the edge of one's computing technology. Prototype using pattern-directed rules, by all means. But if performance becomes a problem, start examining what core operations can be moved into 'hard-wired' procedures." *(p.430)*

- **Three observations on QP implementation**: *(p.430)*
  1. The second observation concerns an inevitable trade-off in building pattern-directed inference systems: efficiency versus rule-based structure.
  2. The third observation concerns the role for special-purpose programs in reasoning systems efficiency. In TGIZMO we put responsibility for several aspects of qualitative reasoning, which could easily have been implemented via pattern-directed rules, into Lisp code.
  3. The final observation concerns building habitable systems: while interfaces are natural time-sinks, providing some basic facilities is always a good idea.

- **ATMS vs JTMS/LTMS architecture shift**: "Compared to the transition from JTMSs to LTMSs, the transition to ATMSs requires a more radical shift in problem-solver design. In an ATMS-based problem solver, the implementor must break free of the 'single current context' mentality and design the inference engine to operate in all contexts at once. This is often not an easy transition." *(p.441)*

- **ATMS context switch cost**: In a JTMS, context switching is done by changing the set of underlying assumptions, requiring relabeling. The ATMS incurs the cost up front. Therefore, if there are relatively few context switches, the JTMS will be more efficient, and if there are many context switches, the ATMS will be. *(p.441)*

- **ATMS labels are complex**: "Unlike the JTMS and LTMS, an ATMS label is a complex data structure. The label is no longer a single atom such as :IN or :OUT, because these only make sense with respect to some current context." *(p.443)*

- **Empty label meaning**: A label is empty if either the dependency network contains no pathway from the assumptions to the node, or if all potential label environments are nogoods. An empty label indicates that the node holds in no consistent environment. *(p.445-446)*

- **ATMS label properties are sound, consistent, complete, and minimal**: These four properties characterize correct ATMS labels: Soundness (n holds in each E_i), Consistency ({} cannot be derived from any E_i given C), Completeness (every consistent environment E in which n holds is a superset of some E_i), Minimality (no E_i is a proper subset of any other). *(p.448)*

## Design Decisions and Rationale

- **Why influence resolution uses hard-coded procedures instead of rules**: "Since the laws of influence resolution are both fixed and central to the operation of the system, we decided to make them as fast as possible." The natural migration path: prototype with pattern-directed rules, then move core operations into hard-wired procedures when performance matters. *(p.430)*

- **Why `search-PTID` is instrumented to print at each context state**: The debugsaur flag `PTID-SSO` is part of the debugging list for TGIZMO. The procedure is invoked at the end of the file. Note also that `search-PTID` calls `extract-IR-Ds-ls`, in case influence resolution was performed as part of the execution of `think`. This should not be necessary, but it is carried out as a bit of belt-and-suspender error handling. `In-Contradiction-Handler` is always part of a TGIZMO's set of contradiction handlers, retracting these assumptions to avoid unnecessary contradiction handling. *(p.401)*

- **Why transitivity processing is centrally installed**: Transitivity processing is centrally installed (recorded in the current TGIZMO structure) to ensure that transitivity processing is periodically checked. The reason for deferring transitivity reasoning is efficiency: `install-compare-constraints` is typically called by LTRE rules which are triggered by comparisons being mentioned. Presumably there are other rules queued for execution as well. These rules may install yet more comparisons, so it will be more efficient to check the comparison cycles once, after the queue is emptied. *(p.406)*

- **Why ATMS assumptions should be minimized**: "Marking a node as an assumption tells the ATMS that the node can appear in the second argument of `in-node?` and consequently labels (and sets) can grow exponentially in the number of possible assumptions. Thus it is incumbent on the inference engine to introduce as few assumptions as possible." *(p.443)*

- **Why ATMS does not need contradiction handling**: "As the ATMS simply removes inconsistent environments from node labels, contradiction handling is not necessary for this purpose. However, contradiction handling also plays a less obvious role -- it serves to signal the inference engine that a set of choices it has made are inconsistent." *(p.446)*

## Implementation Notes

- **Influence resolution implementation uses three sub-procedures**: `resolve-dis-on` for direct influences, `resolve-iis-on` for indirect influences. Both follow a similar pattern: sort by sign, then analyze. The procedures `justify-ir-ambig`, `justify-ir-result`, and `exclude-ir-result` handle concluding the analysis. *(p.401-404)*

- **Inequality reasoning uses a cycle-detection algorithm**: The system detects cycles in the comparison graph and uses case analysis on the types of comparisons (hard: <, >, =; soft: <=, >=) to derive new constraints or detect inconsistencies. The procedure `check-comp-cycle` is the main driver. *(p.406-409)*

- **Measurement interpretation algorithm (`run-measurements`)**: Computes all qualitative states consistent with assumptions. Uses `search-pps` to calculate the set of consistent view and process structures; for each such combination, `resolve-completely` generates the set of complete states by finding what changes in parameters can be caused by the active processes. Each complete state is recorded in the `tgizmo-states` field using `snapshot`. *(p.412-413)*

- **State comparison utilities**: *(p.412)*
  - `get-state` retrieves a state from a given TGIZMO based on its number
  - `show-state-stack` provides a more easily readable description of a state as an ASCII file
  - `make-state-index` builds a set of nested states, discriminating according to the different belief fields of states
  - `summarize-state-codes` produces a readable summary of an index, providing a good overview of a set of states
  - `same-state?` returns non-nil if the states given as arguments differ in some way
  - `find-corresponding-states` produces an alert of equivalent states
  - `summarize-Ds-differences` reports on what Ds values states have in common and highlights their differences

- **TGIZMO entity system uses `defentity` and `defprocess` macros**: These macros define domain theory objects. `defentity` describes physical entities with their quantities, quantity-conditions, and relations. `defprocess` describes processes with preconditions, quantity-conditions, relations, and influences. *(p.418-422)*

- **The ATMS achieves its functionality by precomputing a complex label for each TMS node**: Instead of labeling a node `:IN` or `:OUT`, the ATMS labels every node with the consistent sets of assumptions under which the node is IN. The ATMS answers queries by checking this complex label. In making the choice of whether to use a JTMS or an ATMS it is important to keep this distinction in mind: in a JTMS one incurs the cost of context switching when the actual context switches are made; the ATMS incurs the cost up front. *(p.441)*

- **ATMS label update is incremental**: Like all TMS algorithms, the ATMS algorithm is incremental, taking a correct set of node labels and computing the incremental changes caused by adding a justification. The algorithm operates by making labels locally correct and propagating label changes until labels become globally correct. Note that assumptions are created with labels containing the single environment containing themselves, and all other nodes are created with empty labels. *(p.448)*
