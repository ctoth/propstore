## Key Definitions

- **Embedded problem solver:** A problem solver where a reasoning system is integrated as a module into a larger system. Examples include temporal reasoning components of a factory scheduling system, the abductive component of a natural language understanding system, and the domain expert subsystem of an intelligent tutoring system. *(p.700)*

- **Integrated architecture:** A system that is intended to be mind-like in some way, distinguished by three properties: (1) Integration -- must include several other parts for interacting with the world; (2) Agency -- addresses the question of what it means for a computational process to be an agent; (3) Continuous, long-term operation -- must operate over extended periods of time. *(p.702)*

- **The Day Barrier:** Create an integrated agent that performs useful services, without interruption, for at least one day. *(p.703)*

- **The Cockroach Challenge:** Create an integrated agent that functions continuously and usefully for over a year, without the direct assistance of humans with detailed knowledge of its internal structure. *(p.703)*

## Algorithms and Pseudocode

- **loader.lisp utility programs** *(p.708-709)*:
  - `load-files`: Takes as input a list of file names and loads them. Takes two optional arguments: the current file path and the default extension for files.
  - `compile-files`: Takes as input a list of file names and optionally a file path. It compiles each file in turn.
  - `compile-load-files`: Takes as input a list of file names and optionally a file path. It compiles and then loads each file in turn.

- **lst.lisp listing program** *(p.710)*:
  - `lst`: Takes as input the name of a file to print and produces as output a new file of some appropriate form.
  - Arguments include: `:MODE` (output format: LPT, ASCII, EPSON for dot-matrix printers, LATEX, BOOK), `:OUTPUT` (name of output file), `:HEADER` (string inserted on top of each page).
  - `lst-directory`: Runs lst on all files in a given path.
  - `lst-files`: Runs lst on a list of files relative to a given path.

## Key Claims and Properties

- Knowledge bases for particular domains, such as botany [26], electrochemical systems [22], and thermodynamics [8], tend to have a particular suite of tasks in mind when developing their knowledge bases, and use these tasks to guide their efforts. *(p.700)*

- The CYC group has built the largest knowledge base to date, and this has been directly confronting issues of scale in crafting and maintaining knowledge bases. There are many lessons yet to be learned. *(p.700)*

- Bounded-time response is critical for embedded problem solvers: many applications involving embedded problem solvers require interaction with people or the physical world. The ability to produce results within a pre-determined span of time becomes important, even if the results are partial. *(p.700)*

- Interruptability is a key issue: an embedded problem solver may have to postpone completion of a task, perhaps indefinitely, when new information arrives or priorities change. *(p.700-701)*

- Dependency networks are an important tool in building problem solvers that can explain the reasons for their conclusions. However, dependency networks are not the whole story -- they can provide too much information (implementation-specific details), and can provide too little information when background knowledge that conclusions depend on are "compiled out" for efficiency. *(p.701)*

- Building an integrated architecture requires both good components and good ideas on how to put them together properly. Most progress in AI has centered on developing deep understandings of particular tasks (e.g., qualitative reasoning, object recognition, planning, etc.) in a narrow context. These specialized accounts need to be broadened to serve as robust components in an architecture. *(p.702-703)*

- There need not be a unified substrate within which all capabilities are implemented (re: integrated architectures). A loose federation of modules, each with its own internal organization and representations, can serve as a useful platform. *(p.702)*

- Despite dramatic improvements in computer technology, the amount of computer power on the desk of the typical AI researcher is still minute compared to the computer power between his or her own ears. *(p.702)*

- The leading hypothesis about the nature of mind today is that it is a computational process. Cognitive science is the growing multidisciplinary interaction between artificial intelligence, psychology, neurobiology, linguistics, anthropology, and philosophy to understand the nature of the human mind. *(p.703-704)*

- Connectionism is an important approach to cognitive modeling, but it is only one approach among many. One appealing aspect of connectionism is the hope that it can provide a description of how neural systems serve as a substrate for intelligence. But the structure and behavior of neurons is quite different from what is assumed by most connectionist models. *(p.704)*

- Context-oriented cognitive modeling focuses on understanding the computational theory (in the sense of [10]) underlying a task or capability. Context-oriented studies tend to be more difficult for several reasons: they require taking knowledge representations seriously, and they often use psychological measures that are less easily quantifiable, such as protocol analysis. *(p.705)*

## Design Decisions and Rationale

- **Embedding vs. standalone systems:** The book has focused on problem-solving systems that can stand on their own, but many real-world tasks require embedding reasoning systems as modules into larger systems. *(p.700)*

- **Three key issues for embedded problem solvers:** bounded-time response, interruptability, and articulateness (ability to explain reasoning). These are key issues for work on blackboard systems, reactive systems, and any real-time algorithms. *(p.700-701)*

- **Federation vs. unification for integrated architectures:** Hardware limitations are a serious problem. A loose federation of modules is preferred over requiring a uniform substrate in which all capabilities are implemented. *(p.702)*

- **Source code organization:** All file names were chosen to fit within the most restrictive conventions (i.e., MS-DOS), so they should not require changing. On an MS-DOS machine, the sources take up about 1.8MB of space. *(p.707)*

- **Source subdirectory structure for the book's code** *(p.707-708)*:
  - `cps` -- Code for Chapter 3
  - `tre` -- Code for Chapter 4
  - `ftre` -- Code for Chapter 5
  - `jtms` -- Code for Chapters 7 and 8
  - `ltms` -- Code for Chapters 9, 10, and 13
  - `tgizmo` -- Code for Chapter 11
  - `atms` -- Code for Chapters 12 and 14
  - `tcon` -- Code for Chapter 15
  - `gde` -- Code for Chapters 16 and 17
  - `waltx` -- Code for Chapter 18
  - `utils` -- Utility programs

- **Loading system design:** Each system includes a file which defines a list of other files and the file path for that system. The Common Lisp `#+ reader` macros are used to allow the same file to be used across multiple environments. *(p.708)*

## Implementation Notes

- The code is distributed via anonymous ftp, available as a tar file (assuming a Unix directory structure). Floppy disk versions are available from MIT Press for non-Internet users. *(p.707)*

- The `*default-pathname*` global variable controls the file path used by default. `*default-source-type*` is the file extension for Common Lisp sources (used by `compile-files`). `*default-bin-type*` is the file extension for compiled Common Lisp files. *(p.709)*

- When using multiple operating systems or Common Lisp versions, the Common Lisp `#+ reader` macros are recommended to allow the same load file to be used across multiple environments. *(p.708)*

- For LaTeX users: The BPSCode environment used by the LaTeX and book listing modes is defined in `abntry.sty`, written by Leslie Lamport, commonly available on LaTeX-related ftp sites. *(p.710)*
