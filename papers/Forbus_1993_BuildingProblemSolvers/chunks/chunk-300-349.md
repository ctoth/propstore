## Key Definitions

- **LTMS node labels**: `:TRUE`, `:FALSE`, or `:UNKNOWN`. A node labeled `:TRUE` indicates its datum is believed true; `:FALSE` means believed false; `:UNKNOWN` means no label assigned. *(p.300)*

- **BCP (Boolean Constraint Propagation)**: The search procedure for the LTMS. Labels all nodes by propagating constraints through clauses. Although BCP is linear, the search is exponential because backtracking can add an exponential number of additional clauses (nogoods). *(p.300)*

- **Nogood clause**: When a search labels a node `:TRUE`, and it is later retracted and labeled `:FALSE`, without invoking contradiction handling or backtracking, the clause recording that retraction is the nogood clause. *(p.300)*

- **`create-ltms`**: Creates a data structure containing the entire state of the LTMS. Takes keyword arguments including `node-string`, `debugging`, `checking-contradictions`, `contradiction-handler`, `enqueue-procedure`, `complete`, `delay-sat`. *(p.301)*

- **`contradiction-handler`**: A procedure to call when the LTMS detects a contradiction. The LTMS will unwind the contradiction handler stack until a handler returns a non-nil result indicating it has handled the contradiction. *(p.301)*

- **`enqueue-procedure`**: Called when a node becomes labeled. Allows the inference engine to change initial settings provided to `create-ltms`. *(p.302)*

- **`(tms-create-node ltms datum)`**: The inference engine creates an LTMS node with this procedure, specifying the datum for the node and whether it is an assumption or not. *(p.302)*

- **`assumed-node?`**: Checks whether the node is labeled `:UNKNOWN`. *(p.303)*
- **`known-node?`**: Checks whether the node is not labeled `:UNKNOWN`. *(p.303)*
- **`true-node?`**: Checks whether the node is labeled `:TRUE`. *(p.303)*
- **`false-node?`**: Checks whether the node is labeled `:FALSE`. *(p.303)*

- **`add-formula`**: The main procedure to add formulas to the LTMS. Formulas must be expressed using conventions of Section 9.4, stated in conventional Common Lisp prefix notation. *(p.303)*

- **`compile-formula`**: Like `add-formula` except it attempts to expand its formula argument at compile time; expands into conjunctive normal form and creates calls to `add-clause` at run time. *(p.304)*

- **`support-for-node`**: Returns a description of the formula which forced a node's label (`:ENABLED-ASSUMPTION`, or `:nil`). Two values: first is list of antecedent nodes, second is informant. *(p.305)*

- **`assumptions-of-node`**: Returns the set of enabled assumptions underlying the well-founded explanation for a node. *(p.306)*

- **`assumptions-of-clause`**: Makes only sense if every node in the clause is labeled and hence the clause is either satisfied or violated; otherwise it reports an error. Returns a set of enabled assumptions constructed by finding the assumptions underlying each node of the clause whose label contradicts its sign in the clause. *(p.306)*

- **Construal**: The construal of a set is defined as the hypothesis that the members of a set known to the problem solver are indeed the only members of that set (i.e., they comprise its extension). *(p.343)*

- **Closed-world assumption (CWA)**: Assumptions called closed-world assumptions are used when the problem solver assumes that the resources it knows about are the only ones available. *(p.342)*

- **CWA statement format**: `(CWA (Set) (KnownMembers))` where KnownMembers is a list of the known members of the set when it was closed. *(p.348)*

## Algorithms and Pseudocode

### BCP Algorithm (p.309-310)
The BCP algorithm described is cleaner than the earlier one in Section 9.3. It never needs to examine the original form of clauses; instead it creates auxiliary datastructures so that BCP can be implemented by just incrementing and decrementing counters. The node datastructure includes two lists: the clauses in which the node appears positively, and the clauses in which the node appears negatively. The clause datastructure includes a counter of the number of nodes which can potentially help violate it. Maintaining these counts is key to BCP.

Algorithm for enabling an assumption: *(p.309)*
1. If the node receives label `:TRUE`, find all clauses in which it appears negatively and decrement their counts. Schedule any clause whose count drops below two for processing in step 2.
2. If the node receives label `:FALSE`, find all clauses in which it appears positively and decrement their counts. Schedule any clause whose count drops below two for processing in step 2.
3. Check each of the clauses whose count has been decremented. If count becomes zero, clause has been violated and a contradiction should be indicated. If the clause has one potential violator, then the clause forces a node label.

Algorithm for retracting an assumption (similar to JTMS retraction): *(p.310)*
First phase: For nodes that become `:UNKNOWN`, examine clauses that mention them. If any clause's count is one, set node's label (unless some other clause has already been relabeled) and propagate.
Second phase: Attempts to find alternative support for nodes that became `:UNKNOWN` in phase one. For each node just marked `:UNKNOWN`, examine all clauses that mention it. If any clause's count is one, set the node's label and propagate.

### Indirect Proof Mechanism (p.324-325)
`try-indirect-proof` procedure: *(p.324-325)*
```lisp
(defun try-indirect-proof (fact &optional (*LTRE* *LTRE*))
  (unless (known? fact)
    (with-contradiction-handler (ltre-ltms *ltre*)
      #'(lambda (contradictions ltms &aux assumptions)
          (setq assumptions
                (assumptions-of-clause
                  (car contradictions)))
          (let ((the-node
                  (find (datum-tms-node (referent fact T))
                        assumptions)))
            (when the-node
              (let ((status (tms-node-label the-node)))
                (retract-assumption the-node)
                (add-nogood the-node status
                            assumptions))))))
      (assuming '((:NOT ,fact)) *LTRE*
        (run-rules))
      (known? fact))))
```
Works by assuming the negation. If a contradiction is detected, the initial assumption (negation) is retracted and a nogood is recorded. *(p.324)*

### Contradiction Handling in LTMS (p.307)
`violated-clause?` checks whether clause is still violated. Important because the contradiction handler processes multiple contradictions at a time, resolving earlier contradictions in the list may automatically resolve others. *(p.307)*

Steps for processing a contradiction: *(p.307)*
1. Check whether the contradiction is still a problem with `violated-clause?`
2. Find all the enabled assumptions underlying the contradiction
3. Pick one assumption, the culprit, to retract via `retract-assumption`
4. Record the contradiction by adding a clause consisting of the negations of all the assumptions

## Data Structures

### LTMS Datastructure (`ltms`) (p.312)
Fields:
- `title` — Ignored by the LTMS, useful for debugging
- `node-counter` — Provides a unique name for nodes
- `clause-counter` — Provides a unique name for clauses
- `nodes` — A hash table of all nodes created, keyed by their data. Two nodes are considered the same if their data are equal
- `clauses` — List of all clauses either directly supplied by the inference engine or inferred from formulas
- `debugging` — A debugging flag to trace LTMS operations
- `checking-contradictions` — Flag that defaults to `t`, allows advanced programs to turn off contradiction checking temporarily
- `node-string` — An inference-engine-supplied procedure returning a descriptive string for a node
- `contradiction-handler` — A stack of inference-engine-supplied contradiction handlers
- `pending-contradictions` — A list of violated clauses upon which contradiction handling will be invoked at the conclusion of the current LTMS operation *(p.312-313)*

Additional LTMS datastructure fields: *(p.313)*
- `enqueue-procedure` — User-supplied procedure called when a node becomes labeled
- `complete` — Flag indicating whether the LTMS ensures logical completeness (for basic LTMS it is always nil)
- `violated-clauses` — The BCP algorithm detects violated clauses while propagating labels through the database
- `queue`, `conses`, `delay-sat`, `cons-size` — See Chapter 13 *(p.313)*

### TMS Node Datastructure (p.313-314)
Fields:
- `index` — Integer serving as unique name for this node
- `datum` — The (positive) datum supplied by the inference engine
- `label` — Represents current belief status: `:TRUE`, `:FALSE`, `:UNKNOWN`
- `support` — This field is nil if the node is unknown. If the node is an enabled assumption, this field contains the symbol `:ENABLED-ASSUMPTION`; otherwise this field contains the clause which BCP used to force the node's label
- `true-clauses` — Generalization of the JTMS justifications and consequences. List of clauses in which this node occurs positively
- `false-clauses` — List of clauses in which the node occurs negatively
- `mark` — Holds a marker for the sweep algorithm used in finding assumptions underlying a node
- `assumption?` — If non-nil, indicates this node should be treated as an assumption whose belief can be explicitly enabled and retracted
- `true-rules` / `false-rules` — Rules that should be run when the node becomes labeled `:TRUE` / `:FALSE`
- `ltms` — The LTMS instance to which this node belongs
- `true` / `false` — Contains a unique Common Lisp cons cell whose car is the node itself and whose cdr is `:TRUE` / `:FALSE`. This is how the LTMS represents the positive/negative literal *(p.313-314)*

### Clause Datastructure (p.314-315)
Fields:
- `index` — Integer for unique name
- `informant` — Description of the clause: if added by `add-clause` procedure, then informant is whatever the inference engine supplied; if from `add-formula`, then informant is `(:IMPLIED-BY)` and the two original arguments to `add-formula`
- `literals` — A description of the literals appearing in the clause. Consists of a list of node-label pairs, e.g., `(a . :TRUE) (y . :FALSE)` represented as `((a . (:TRUE)) (y . (:FALSE)))`
- `pvs` — Counter critical to making BCP efficient. Counts the number of nodes which, if relabeled, might violate the clause (pvs counts from potential violators). Counts the number of literals which are not yet known to be satisfied
- `length` — Contains the number of literals in the clause (could be computed by calling `length` on its literals list)
- `sats` — Count of number of literals currently satisfying the clause; if > 0 then clause is satisfied
- `status` — See Chapter 13 *(p.314-315)*

### LTRE Datastructure (p.336)
LTRE consists of the following files:
- `linter.lisp` — Datastructures and interface
- `ldata.lisp` — Database system
- `lrules.lisp` — Rule system
- `unify.lisp` — Unifier, identical to FTRE version
- `funify.lisp` — Open-coding unification, identical to FTRE version

## Key Claims and Properties

- **BCP is linear but search is exponential**: Although BCP is linear, the search is exponential because backtracking can add an exponential number of additional clauses for BCP to consider. For many applications, the number of nogood clauses added to the clause set is so great that it is counterproductive to record nogood clauses at all. Chapter 11 explores various examples which have an exponential number of nogoods. *(p.300)*

- **LTMS does not have contradiction models**: The LTMS does not have contradiction models like a JTMS. The LTMS node is used to represent both the positive and negative instances of a datum. *(p.302)*

- **LTMS can only have one instance**: A node can be in only one LTMS, and every node keeps track of the LTMS it belongs to. Hence, most LTMS procedures need not be supplied with the explicit LTMS argument. *(p.302)*

- **Two preconceptions about compile-formula**: (1) It assumes that the structure of the entire formula is available at compile time. (2) It assumes that different Common Lisp variables always refer to different nodes (if they don't, either Common Lisp errors or incorrect node labels will result). *(p.305)*

- **LTMS shifts burden of propositional reasoning to TMS**: The LTMS moves the dividing line of responsibility between the TMS and the inference engine. The burden of propositional reasoning is shifted from the inference engine to the TMS in an LTMS-based problem solver. *(p.325)*

- **Direct translation strategy is preferable**: The direct translation strategy comes out slightly better in terms of clauses required to encode our example proposition, and substantially better in terms of the number of assertions required (building and indexing assertions and their corresponding TMS nodes takes more space and time than building clauses). LTRE uses the direct translation strategy. *(p.331)*

- **Multilevel vs. direct translation strategy comparison**: Under direct translation strategy, 7 TMS nodes are required; under multilevel expansion, 12 assertions and 25 clauses; under direct translation, 7 assertions and 24 clauses. *(p.330)*

- **Explanations via direct translation are clearer**: The explanations given by direct translation schemes are found to be easier to use than those given by multilevel expansion schemes. Explanations given by the latter tend to be boring, since they step through each intermediate connective. *(p.331)*

- **Relative closure restriction for contradiction handlers**: A single contradiction handler which can be rebound is adequate only under three restrictions: (1) all assumption-manipulating operations must be identified with appropriate handlers, (2) assumption-manipulating operations must proceed depth first, (3) relative closure (every consequence that holds for the current set of assumptions and which might lead to a contradiction must be computed before making more assumptions). *(p.333)*

- **Relative closure is practically impossible**: Strictly speaking it is impossible to satisfy since the set of consequences can be infinite, but in developing reasoning algorithms the authors have eschewed completeness in favor of efficiency — consequences derivable via Boolean constraint propagation are considered sufficient. *(p.334)*

- **Construals must be updated**: Construals must be updated when new information appears about what a set contains. Usually this means abandoning the processing based on the old construal. *(p.344)*

- **CWA justifications must include negative information**: The justification for a construal must include all `has-member` statements known to be either true or false at the time the set is closed. Failure to include information about negative `has-member` statements can result in entire subproblems being skipped in searches. *(p.347)*

## Design Decisions and Rationale

- **LTMS uses both positive and negative nodes**: The LTMS node represents both the positive and negative instances of a datum. Problem solvers adopt the convention of using `:NOT` to refer to the negation of a node. If the datum of some node is `(MORTAL SOCRATES)`, then its negation is represented as `(:NOT (MORTAL SOCRATES))`. But this detail is irrelevant to the LTMS — the inference engine must indicate to which LTMS node is to be added. *(p.302)*

- **Contradiction handler stack design**: The LTMS provides a stack of contradiction handlers. During problem solving there can be a stack of contradiction handlers. Upon encountering a contradiction, the LTMS will unwind the contradiction handler stack until a handler returns a non-nil result indicating it has handled the contradiction. *(p.301)*

- **Why use direct translation over multilevel expansion**: (1) Fewer assertions required; (2) Clearer explanations; (3) The efficiency advantage goes to direct translation. What about clarity of explanation? In the multilevel expansion strategy, some of the intermediate nodes in explanations are propositions which are recognizably something asserted by the user; in direct translation the same effect is achieved by allowing `assert!` to include an informant as well as a constraint. *(p.331)*

- **LTRE stack-based contradiction handling design**: Multilevel backup requires visible handlers. The purpose of assuming relative closure is the same as the requirement in FTRE (Section 5.1.3) that all rules that do not require assumptions be executed before any rules that do. *(p.334-335)*

- **Indirect proof via contradiction handling**: The essence of indirect proof is to assume the negation of the fact F to be shown and derive a contradiction. Once a contradiction has been derived, F can be justified based on the assumptions underlying the contradiction, minus the assumption ~F. The procedure `try-indirect-proof` is a simple implementation using the LTMS's contradiction handling mechanism. *(p.324)*

- **Two strategies for avoiding CWA staleness**: (1) Limiting the temporal extent of closed-world assumptions by closing a set, then retracting the previous derivation of a CWA before the new one; (2) Since only clauses that mention previous CWA assumptions include them as negative literals, they can only be derived as false. *(p.349)*

## Implementation Notes

- **Clause representation**: A clause like `x -> y` is represented as `((a . (:TRUE)) (y . (:FALSE)))`. The pvs counter counts the number of literals which are not yet known to be satisfied. *(p.314)*

- **Node-string, debugging-ltms, ltms-error, default-node-string procedures**: These are nearly identical to their JTMS counterparts. The procedure `satisfied-clause?` returns true if the clause is satisfied (used within search-based inference engines). *(p.315)*

- **`enable-assumption` and `retract-assumption`**: Procedures to enable and retract an assumption. If a node is created with the assumption property, the assumption is initially disabled. The LTMS procedure to enable assumptions must indicate which label (`:TRUE` or `:FALSE`) to assign the node — this will cause an error if the assumption is already known and has a label different than the one supplied. *(p.303)*

- **`add-clause` vs `add-formula`**: `add-clause` adds the clause with positive literals (true-nodes and negative literals (false-nodes). The preferred procedure is `add-formula` with `add-clause` intended primarily for internal use. *(p.304)*

- **`compile-formula` preconceptions**: (1) It assumes the structure of the entire formula is available at compile time; (2) it assumes that different Common Lisp variables always refer to different nodes. *(p.305)*

- **`bcp-add-clause`**: Indexes the clause, initializes the clause counts. At its conclusion it calls `check-clauses`, which invokes BCP. The `bcp-add-clause` is sometimes requested to create the clause without indexing it (indicated by an optional fourth argument which is always `t` in the LTMS). *(p.317)*

- **`set-truth` and `propagate-unknownness`**: `set-truth` is a top-level way to call `set-truth`. After calling `set-truth` it calls `check-clauses` to propagate the changed label (if any) and then performs contradiction handling if needed. *(p.318)*

- **Retraction**: `propagate-unknownness` performs the first phase of retraction. For every node that becomes labeled `:UNKNOWN`, every clause it participates in is checked to see whether that clause forced the label of some other node. *(p.318)*

- **`find-alternative-support`**: Attempts to find alternative support for nodes labeled `:UNKNOWN` by `propagate-unknownness`. *(p.319)*

- **Contradiction-handling interface**: LTMS contradiction handling is affected by two fields: `checking-contradictions` and `contradiction-handler`. If `checking-contradictions` is nil, then violated clauses are placed on the LTMS slot `ltms-pending-contradictions` in case the inference engine might later want to examine them. *(p.319)*

- **`with-contradiction-handler`**: Allows the inference engine to temporarily and cleanly install a contradiction handler during some fragment of problem solving. *(p.319)*

- **`with-assumptions`**: Allows the inference engine to temporarily and cleanly enable some assumptions. *(p.320)*

- **LTRE database changes**: LTRE translates logical connectives directly into clauses. The variable `*connective-list*` indicates what connectives are known to the LTMS (i.e., `:NOT`, `:AND`, `:OR`, `:IMPLIES`, `:IFF`, and `:TAXONOMY`). The procedure `simple-proposition?` returns non-nil exactly when a proposition should be treated as a literal by the TMS. *(p.337)*

- **`assert!` in LTRE**: The definition of `assert!` is similar to `assert!` with one crucial difference: since an assumption can be retracted, we must take care to ensure that any clauses we add to the LTMS can be decommissioned if the assumption is disabled. *(p.338)*

- **`fetch-global` debugging procedure**: Provides filtering, ensuring that `fetch-global` only returns facts with a particular label. *(p.339)*
