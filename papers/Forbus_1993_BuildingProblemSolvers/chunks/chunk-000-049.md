## Key Definitions

- **Problem space model** *(p.36-p.37)*: A classical model of problem solving. A problem space consists of a set of states (representing configurations of the domain), a set of operators (which define how to move between states), an initial state, and a goal. The problem space is the set of all legal configurations and moves. Solving a problem in this model is accomplished by finding a sequence of operators which, when applied to the initial state, allows one to reach a state satisfying the goal criterion.

- **State** *(p.37)*: A distinguished state representing the starting configuration. The initial state is a distinguished state, and the goal is a specification of the subset of states which could serve as a solution.

- **Operator** *(p.37)*: Operators typically have parameters. For chess, the legal moves available in a board position depend on what types of pieces are on the board and where they are. An operator can be applied in several different ways to a situation.

- **CPS (Classical Problem Solver)** *(p.38)*: Conceptually consists of two parts: an interface for user-supplied problem spaces, and a search engine. The interface requires four distinct state manipulations: goal detection, state identity, state display, and operator application.

- **Path** *(p.46)*: The path struct defines queue elements. Each path includes the current state (called `current`), the sequence of states and operators traversed to arrive at this state (called `so-far`), implemented as a list of alternating states and operator instances. Also includes `pr` back-pointer and `distance-remaining` for heuristic search.

- **Five desiderata for AI programs** *(p.27)*:
  1. **Efficiency**: Procedural languages excel at saving "how."
  2. **Coherence**: A well-written program handles routine situations without much differing around; its behaviors are highly tuned to its normal ecological niche.
  3. **Flexibility**: Human programmers can anticipate an amazing variety of problems but cannot anticipate all possible conditions.
  4. **Additivity**: Large conventional programs are extremely hard to change. A small modification in one routine may have unexpected effects that crop up all over.
  5. **Explicitness**: The program works, but it is opaque in its operation; it is hard to find out why one action was taken rather than another.

- **Design space axes for problem solvers** *(p.30)*:
  - **Knowledge model**: How is the system's knowledge represented?
  - **Reference mechanism**: How do procedures get the data they need to execute?
  - **Procedure model**: How are procedures decomposed and organized?
  - **Execution strategy**: How are procedures carried out?
  - **Dependency model**: What information is recorded about the relationships between a system's beliefs?

- **Four phases of AI programming** *(p.31)*:
  1. **Conceptualization**: A problem is carefully chosen and analyzed.
  2. **Initial exploration**: Rapid exploratory programming provides a program that embodies your ideas so you can test them.
  3. **Experimentation**: Running the program on dozens or even hundreds of examples.
  4. **Production**: Transforming the program built in previous phases into a product.

## Algorithms and Pseudocode

- **Basic CPS search algorithm** *(p.41)*: Search proceeds via the following steps:
  1. Pop a state from the queue.
  2. If the state satisfies the goal criterion, halt and signal success by returning the successful path.
  3. Otherwise, calculate the operators that can be applied to the current state and the states that will result from each of them. Update the queue accordingly, and begin again.
  4. If the queue is empty, then return, signaling failure.

- **Queue organization determines search strategy** *(p.41)*: An unordered FIFO queue corresponds to breadth-first search, and an unordered LIFO queue corresponds to depth-first search. Given a heuristic estimate of distance remaining to a goal state, more powerful search strategies are possible: sorting the queue by minimum estimated distance corresponds to best-first search. Branch-and-bound is a version of best-first search which places a fixed upper bound on the size of the queue.

- **Breadth-first search (naive version)** *(p.47)*: Figure 3.1 shows an elegant but inefficient breadth-first search using recursive list construction. Uses cons pairs `((InitState) . (Path))` instead of structs. The authors note the first problem is that the recursion is `backviz`; it is an example of tail recursion that a good compiler should turn into iteration but that has practical problems.

- **Breadth-first search (better version)** *(p.48)*: Figure 3.2 shows a better version that encapsulates most of the work in one procedure, uses `*debug-cps*` variable to control whether internal details are printed. Remedies overhead by using `*debug-cps*` to determine whether or not internal details are printed.

- **extend-path** *(p.49)*: Two-loop structure. The outer loop goes through the list of operators, and the inner loop examines each operator instance and the result of applying it. The information needed by these loops (`pr-operators` and `pr-operator-applier`) is obtained from the problem struct pointed to by `path-pr`.

## Data Structures

- **Problem space struct** *(p.45)*: Uses `defstruct` with `CONS-NAME` providing abbreviation for field names. Contains all fields mentioned in the design:
  - `pr-goal-recognizer` -- implements the goal criterion
  - `pr-states-identical?` -- detects when two states are identical
  - `pr-display-state` -- produces a human-readable description of a state
  - `pr-operators` -- provides a list of operators for the problem space
  - `pr-operator-applier` -- given a state and operator, finds all applicable instantiations and resulting states

- **Path struct** *(p.46)*: Defines queue elements with fields:
  - `path-current` -- the current state
  - `path-so-far` -- sequence of states and operators traversed (alternating list)
  - `path-pr` -- back-pointer to problem that generated this search effort
  - `path-filter` -- domain-specific guidance to prune search paths
  - `distance-remaining` -- numerical heuristic estimate of "distance" from state to goal
  - `action-element-printer` -- produces a printable string for solution path
  - `state-printer` -- used by `print-answer`

- **CPS program files** *(p.43)*:
  - `cps.lisp` -- System definitions
  - `search.lisp` -- Structure definitions and breadth-first search
  - `variants.lisp` -- Other search strategies

## Key Claims and Properties

- **AI programs differ from conventional programs in two key ways** *(p.26)*: (1) explicit representation of knowledge and (2) increased modularity. The ability to use explicit representations of knowledge is perhaps the best hallmark of AI programs.

- **Increased modularity** *(p.26)*: The increased modularity in AI programs comes from decomposing procedures into small, fine-grained pieces. This extensive decomposition leads some to claim that AI programs necessarily consist of rules, but this need not be the case.

- **Pattern matching is fundamental** *(p.26)*: Explicit knowledge representations often require more complex reference mechanisms than simple variable/binding, involving some form of pattern matching. The increased modularity and explicit communication between parts of a program increase its ability to make its own decisions, dynamically, rather than forcing the programmer to anticipate all possibilities in advance.

- **Rule-based programs are invariably slower** *(p.29)*: Consequently, any rule-based program will invariably be slower than the corresponding procedural program, due to overhead of pattern matching to establish execution environments rather than more compact and efficient stack.

- **Flexibility vs. efficiency trade-off** *(p.28)*: Efficiency and coherence almost always trade off against flexibility and additivity. A program that stores the absolute minimum information for a particular job will always perform faster than one that must consult explicit knowledge and record the rationales for its decisions.

- **Explicitness enables other properties** *(p.28)*: The ability to respond to new facts implies the ability to change course, and thus to change when things happen. So coherence trades off against flexibility and additivity.

- **State identity is important** *(p.39)*: State identity is important because we make no progress by re-exploring already examined states. Duplicating states is important even if the results of CPS are only intended to be used by another program.

- **Book focuses on truth maintenance systems** *(p.34)*: The book focuses on exploiting truth maintenance systems for building reasoning systems. It introduces a classification scheme: justification-based, logic-based, and assumption-based truth maintenance systems. These basic ideas are developed in Chapters 6-9, with the next eight chapters devoted to exploring their implications.

- **Tail recursion warning** *(p.47)*: It is easy while debugging to transform inadvertently a program with merely syntactic recursion into one with true recursion, with dreadful results the next time it is executed on large datastructures.

- **Non-local exits preferred** *(p.49)*: When there is more than one exit condition, the only alternative to using a non-local exit is to use a disjunctive end test in the `do`. If there are several termination conditions the logic of the end test can become quite baroque.

## Design Decisions and Rationale

- **Why Common Lisp over C/C++** *(p.20-p.21)*: Goals are pedagogical; Lisp chosen because it is the best generally accepted language for easily communicating AI ideas. Lisp's treatment of procedures as first-class entities, provision of automatic storage management, and simplified syntax allows programs to easily write and analyze other programs. Three reasons commercial tools (expert system shells) are not appropriate: (1) deep understanding of underlying tools gives a better chance of using them successfully; (2) sometimes source code access is necessary; (3) the state of the art in building reasoning engines is far from static.

- **Global variables vs. structs for CPS** *(p.42-p.43)*: Given that CLOS is eschewed, the most obvious approach is global variables, but this has legion disadvantages: hard to keep track of what must be defined, subtle bugs from uninitialized variables, prevents multiple copies of system from being active. The best compromise: define a structure that combines the necessary information "under one roof."

- **Why not CLOS** *(p.42)*: Two reasons: (1) the CLOS specification was not completely standardized when the book was written; (2) they wanted to ensure maximum portability across implementations.

- **Problem space as reusable description** *(p.40)*: A pure problem space description could be shared across many copies of CPS, each solving a different problem in that space. On the other extreme, a complete problem description encapsulates all the information defining a specific problem.

- **`kaux`/`setq` style over `let`** *(p.49)*: The authors use `kaux` with `setq` instead of `let` for local parameters to an outer procedure. Reason: indentation conventions that normally make Common Lisp readable can sometimes result in code clumping on the right-hand side, leaving large amounts of blank space. `kaux` and `setq` typically leads to less indentation than `let`.

## Implementation Notes

- **Programs written in standard Common Lisp** *(p.22)*: Known to run in Symbolics Common Lisp, Lucid Common Lisp on IBM RT/IBM RS/6000, Sun SPARCStation. Also tested with Symbolics CLOE, Gold Hill's GCLisp, and Apple's Macintosh Common Lisp.

- **CPS interface operations** *(p.38-p.41)*: Four distinct state manipulations required:
  1. Goal detection: Ascertain whether a given state satisfies the goal criteria
  2. State identity: Detect when two descriptions of states refer to the same state
  3. State display: Produce a human-readable description of a given state
  4. Operator application: Given a state and an operator, find all instantiations and resulting states. `pr-operator-applier` returns a list of pairs whose car is an instance of the operator and whose cdr is the resulting state.

- **`*debug-cps*` variable** *(p.44)*: When non-nil, causes CPS to print extra information about the state of the search. Defined by `*debug-cps*`.

- **`CONS-NAME` convention** *(p.45)*: Struct field names use an abbreviation prefix (e.g., `pr-` for problem space fields) to enforce naming convention. Generically it is easiest to use the name of the struct itself, but sometimes prepending the struct's name to field names would result in long, unwieldy names, and `CONS-NAME` provides a subform.

- **`pr-operator-applier` return format** *(p.49)*: Returns a list of pairs whose car is an instance of the operator and whose cdr is the state that results from applying that instance to the argument state.

- **Two-loop extend-path structure** *(p.49)*: The outer loop goes through the list of operators; the inner loop examines each operator instance and the result of applying it. Information needed by these loops is obtained from the problem struct pointed to by `path-pr`.
