## Key Definitions

**PROPAGATE** *(p.450)*: The core incremental label update algorithm for the ATMS. When a new justification J is suggested, the ATMS calls PROPAGATE(x, a1, ..., an, x = n1, a, f, l) where x is the node being justified, the ai are antecedents, and f and l specify the justification. It computes the incremental label update by taking the union of the label of f and a set of environments just added to the label of x.

**UPDATE** *(p.450)*: UPDATE adds a set of new potential label environments L to node n. It is an auxiliary procedure used by PROPAGATE.

**WEAVE** *(p.451)*: WEAVE(a1, I, X) iterates over antecedent nodes, incrementally constructing the incremental label by computing the union of an environment of I and an environment of N's label. It ensures that the result F is minimal and contains no known inconsistency, removing from F all duplicates and nogoods, as well as any environment subsumed by any other.

**NOGOOD** *(p.451)*: NOGOOD(F) is called whenever an environment E is newly discovered to be nogood by the propagation algorithm. It removes all nogood environments from all labels.

**Solution (in ATMS context)** *(p.454)*: A solution to a default reasoning task must be defined entirely differently from standard solutions. A solution is defined to be a consistent set of assumptions to which no other assumption can be added without introducing an inconsistency. Such solutions are sometimes referred to as an extension or maximally consistent environment.

**Definition 13.1** *(p.472)*: For every symbol c, s and ~s are complementary literals. A clause is a disjunction of literals with no literal repeated and no complementary literals.

**Definition 13.2** *(p.473)*: An implicate of a formula or a set of formulas C is a clause entailed by C. If we replace every formula with its entire set of implicates, then BCP on the resulting set of implicates produces the same result as BCP on the original formulas.

**Definition 13.3** *(p.473)*: Clause A subsumes clause B if all the literals of A appear in B.

**Theorem 13.1** *(p.473)*: Suppose C is a set of clauses and C' is a subset of C, where C' are the clauses of C not subsumed by any other. BCP on C produces the same labels as BCP on C'.

**Definition 13.4** *(p.474)*: A prime implicate of a formula C is an implicate of C which is not subsumed by any other implicate of C. Therefore it is sufficient to only use the prime implicates in our encoding, which greatly reduces the number of implicates needed to achieve formula BCP.

**Theorem 13.2** *(p.474)*: Given a set of formulas F and the union I of the prime implicates of each individual formula of F, then BCP on I produces the same labeling as BCP on F.

**Theorem 13.3** *(p.484)*: Suppose that the set of clauses I is the set of prime implicates of some set of formulas. Then BCP on I is logically complete.

**Clause Management System (CMS)** *(p.488-489)*: The CMS accepts arbitrary clauses (or formulas converted into clauses) and computes ATMS labels for them. A literal l is said to hold in environment E if it can be derived from E's subset of C. The CMS maintains for each literal l a set of environments {E1, ..., En} (called the label) having four properties: (1) Soundness: l holds in each Ei; (2) Consistency: l cannot be derived from any E given C; (3) Completeness: every consistent environment E in which l holds is a superset of some Ei; (4) Minimality: no Ei is a proper subset of any other.

## Algorithms and Pseudocode

**Algorithm 12.1 (PROPAGATE)** *(p.450)*:
1. Compute the incremental label update: L = WEAVE(a1, a2, {x1, ..., xn}). If L is empty, return.
2. Update label and recur: UPDATE(L, n).

**Algorithm 12.2 (UPDATE(L, n))** *(p.450)*:
1. Detect nogoods: if n = ~, then call NOGOOD(E) on each environment E in L and return.
2. Update n's label, ensuring minimality:
   a. Delete every environment from L which is a superset of some label environment of n.
   b. Delete every environment from the label of n which is a superset of some element of L.
   c. Add every remaining environment of L to the label of n.
3. For every justification J in which n is mentioned as an antecedent:
   a. Propagate the incremental change to n's label to its consequences: invoke PROPAGATE(f, n, L).
   b. Remove subsumed and inconsistent environments from L. Remove from L all environments no longer in n's label.
   c. Early termination: if L = {}, return.

**Algorithm 12.3 (WEAVE)** *(p.451)*:
1. Iterate over antecedent nodes. Repeat the following steps for each N in a, n, b, and then return I.
2. Incrementally construct the incremental label. Let F be the set of all environments formed by computing the union of an environment of I and an environment of N's label.
3. Ensure that F is minimal and contains no known inconsistency. Remove from F all duplicates and nogoods, as well as any environment subsumed by any other. Set I to F.

**Algorithm 12.4 (NOGOOD(E))** *(p.451)*:
1. Mark E as nogood.
2. Remove E and any superset from every node label.

**Algorithm 13.1 (BRUTE-FORCE prime implicates)** *(p.477)*:
1. Let P (the result) be {}.
2. Let Q be the set of clauses obtained by converting the formula to CNF.
3. Take the first clause q off of Q. If none, we are done.
4. If q is subsumed by any clause of P, then go back to step 1.
5. Remove all clauses of P that are subsumed by q.
6. Try to compute the consensus of q and every clause in P. Whenever the consensus exists, add it to Q.
7. Add q to P.
8. Go to step 3.

**Algorithm 13.2 (IPA(N, S))** *(p.492)*: Incremental prime implicate algorithm based on Tison's method.
1. Delete any D in N \ S that is subsumed by another D' in N \ S.
2. Remove a smallest C clause from S. If none, return.
3. For each literal l of C, construct IL which contains all classes of N that resolve successfully with C.
4. Let Z be the set containing C.
5. Perform the following steps for each literal l of C:
   a. For each clause in Z which is still in S, compute the consensus of it and every clause in IL which is still in N.
   b. For every new consensus, discard it if it has been subsumed by N \ S. Otherwise, remove any clauses in N \ S subsumed by it. Add the clause to N and S.

**Algorithm 13.3 (SUBSUMED?(L, N))** *(p.495)*: Checks whether the set of literals L is subsumed by trie N. Uses recursive traversal of a discrimination tree.
1. If N is a terminal clause, return success.
2. Remove literals from the front of L do the following until success, or until the id of the first edge of N is no longer greater than that of L:
   a. If the first literal of N matches L, recursively invoke SUBSUMED on the remaining literals and the edges below the first element of N.
   b. If the recursive call returns success, return success.
3-6. Continue traversal following tree structure.

**Algorithm 13.4 (ADD-TO-TRIE(L, N))** *(p.496)*: Adds a clause to the discrimination trie database.
1. Remove edges at the front of N, until the id of the first edge of N is greater than or equal to the first literal of L.
2. If the label of the first edge of N is the first literal of L, then recursively call ADD-TO-TRIE with the remainder of L, the edges underneath the first edge of N, and return.
3. Construct the edges to represent the literals of L and side-effect the trie such that it appears just before the current position N.

**Algorithm 13.5 (REMOVE-SUBSUMED(L, N))** *(p.497)*: Removes all classes subsumed by L from the trie.
1. If there are no literals, delete the entire trie represented by N, and return.
2. If we are at a leaf of N, return.
3. For each edge e of N, do the following:
   a. If the label of e is the first literal of L, then recursively call REMOVE-SUBSUMED with the rest of the literals and the edges below e.
   b. If the label of e is lower than that of the first literal of L, then recursively call REMOVE-SUBSUMED on the same literals but the edges below e.

## Data Structures

**ATMS datastructure** *(p.460)*:
- `title`: Ignored by the ATMS but useful for debugging.
- `node-counter`: Provides a unique name for nodes. Also used to order the assumptions of an environment.
- `just-counter`: Provides a unique counter for justifications.
- `env-counter`: Provides a unique name for environments.
- `nodes`: List of all nodes created. Used only for debugging.
- `justs`: List of all justifications supplied by the inference engine. Used only for debugging.
- `contradictions`: A list of all nodes which have been marked contradictory. Used only for debugging.
- `assumptions`: A list of all nodes which have been marked assumptions.
- `debugging`: A debugging flag to trace the internals of ATMS operations.
- `nogood-table`: An environment table containing all the unsubsumed nogoods.
- `contra-node`: An internal node created by the ATMS to encode contradictions.

**TMS node datastructure** *(p.461)*:
- `index`: Integer serving as unique name for this node.
- `datum`: Supplied by the inference engine; for simple demonstration systems, one should put something printable in here; for IRL-like systems, a pointer to the assertion object goes here.
- `label`: A list of environments representing the set of minimal sets of assumptions that support belief in this node.
- `justs`: The set of justifications which could provide support for this node.
- `consequences`: The justifications which use this node as an antecedent.
- `nogood?`: If non-nil, this field indicates that belief in this node represents a contradiction. Note that any contradictory node will always have an empty label.
- `assumption?`: If non-nil, this field indicates that this node should be treated as an assumption. The initial label of an assumption node is the singleton environment containing itself.
- `rules`: Rules which should be run when the node's label receives an environment. The ATMS enqueue-procedure is called on each element of this field if it is non-nil. The queue is then cleared.
- `atms`: The ATMS instance to which this node belongs.

**ATMS justification** *(p.461)*: Identical to the JTMS justification:
- `index`: Integer for unique name.
- `informant`: An inference engine-supplied description of the justification.
- `consequence`: The node which this justification can support.
- `antecedents`: The nodes that must be believed in order for this justification to provide support for its consequence.

**Environment (env) datastructure** *(p.462)*:
- `index`: Integer for unique name.
- `count`: Number of assumptions in this environment.
- `assumptions`: The assumptions of the environment in sorted order.
- `nodes`: The nodes which have the environment in their labels.
- `nogood?`: This flag is non-nil when the environment is nogood. Note that if the environment is contradictory, it necessarily has no nodes.
- `rules`: Rules which are run when this environment becomes consistent.

Note: The fields index, count, and assumptions cannot change once the environment has been created. *(p.462)*

**Discrimination trie for clause database** *(p.494-495)*: A tree structure where all edges are literals and whose leaves are clauses. The trie stores the canonical forms of clauses in a discrimination tree. By storing canonical forms in a discrimination tree, a single clause can be checked against all existing clauses in a single operation. Clauses are stored in ascending order of literal ids. Two sets of literals refer to the same clause if their respective clause-literals are `eql`. Complementary literals have the same id, but they can never appear in the same clause.

## Equations

**Consensus rule** *(p.476)*: Given two clauses:
- $x \lor \beta$
- $\neg x \lor \gamma$

where x is a symbol and $\beta$ and $\gamma$ are (possibly empty) disjunctions of literals, the consensus of these two clauses with respect to x is the clause $\beta \lor \gamma$, with duplicate literals removed. If the two clauses have more than one pair of complementary literals, then the consensus would contain complementary literals and is discarded (since it is a logical tautology). *(p.476-477)*

**Qualitative equations encoded as taxonomic formulas** *(p.482)*: Each qualitative variable x is encoded as a taxonomic formula: `tax(x = +, x = 0, x = -)`. Each qualitative equation is expanded into a set of clauses listing all the combinations of values disallowed by qualitative addition. For example, x + y ~ 0 expands to:
- $\neg(x = +) \lor \neg(y = +) \lor \neg(z = +)$,
- $\neg(x = +) \lor \neg(y = 0) \lor (z = 0)$,
- ... (full set of constraint clauses) *(p.482)*

**Prime implicates for qualitative variables** *(p.482)*:
$$\neg([dP_1] = +) \lor \neg([dP_2] = +)$$
meaning: from a qualitative equation $[dP_1] - [dP_2] - [dQ_1 \cdot q] = 0$ and the known values $[dP_1] = +$ and pressure fixed at C (i.e. $[dP_c] = 0$), BCP can infer that $[dQ_1] = +$. *(p.483)*

**Lazy consensus computation** *(p.487)*: In the lazy full LTMS, rather than computing the consensus of a satisfied clause, instead every such consensus calculation is deleted, and the satisfied clause is marked as dirty such that if it ever becomes non-unit (per Section 9.5) consensus calculations will proceed. However, intercepted within the prime implicate algorithm are calls to BCP in order to avoid as many consensus calculations as possible. *(p.487-488)*

**Pressure regulator qualitative equations** *(p.485)*:
$$[dP_1] - [dP_2] - [dQ_1 \cdot q] = 0$$
$$[dP_3] - [dP_2] - [dQ_m] = 0$$
$$[dQ_m] - [dA] = 0$$

Converting into formulas and providing no initial labels, the full LTMS yields 2,814 prime implicates. *(p.485-486)*

**CMS label construction** *(p.489)*: The CMS label can be computed directly from the prime implicates of C which mention l. The label for a non-assumption literal l is constructed as follows: for every prime implicate of the form $\neg a_1 \lor \cdots \lor \neg a_n \lor l$, the label environment $\{a_1, \ldots, a_n\}$ is added. *(p.489)*

## Key Claims and Properties

**PROPAGATE correctness** *(p.450)*: The algorithm is guaranteed to terminate with correct labels. Although this algorithm is easy to understand, it is inefficient, as it repeatedly recomputes node labels (a step is <= 1). A more efficient version only propagates incremental changes to node labels.

**ATMS thread safety** *(p.459)*: The code in atms.lisp is a simple assumption-based truth maintenance system. As with the previous JTMS and LTMS implementations, ATMS operations cannot be safely aborted, nor can one ATMS operation be safely invoked before another is finished, nor can multiple processes safely access the same ATMS simultaneously.

**Label update procedures** *(p.463)*: The procedures propagate, weave, and update perform exactly as outlined in Section 12.2. update-label is an auxiliary procedure to update. The procedure update includes additional functionality to maintain ATMS datastructures: it ensures that the nodes slot of all environments is always correct.

**Environment table properties** *(p.464)*: The ATMS maintains two tables in identical formats. atms-env-table contains all environments created during ATMS processing. atms-nogood-table contains all unsubsumed nogoods discovered so far. The table groups environments by their size. The table consists of a set of buckets, each bucket consisting of an integer indicating the size of all environments in this bucket, followed by those environments. The table is kept sorted, so smaller environments come first. The procedure insert-env-table adds an environment to a table. The nogood table is used to check whether newly created environments are subsumed by known nogoods.

**Formula BCP enhances logical completeness** *(p.480)*: The basic advantage of formula BCP is that it enhances the logical completeness of the LTMS without paying an excessive penalty in computational performance.

**NAND gate example** *(p.480-481)*: Encoding the NAND gate truth table as four implications and using BCP on the prime implicates allows inferring unknown outputs. Using BCP on prime implicates, it is possible to make the desired inference: if z is FALSE, BCP labels x TRUE regardless of y's label.

**XOR gate from NANDs** *(p.481)*: When modeling an XOR gate built up from NAND gates, x remains UNKNOWN using BCP, which is not at all what we want. If instead we replace the formula with its prime implicates, we correctly infer from z = FALSE that x = TRUE and y = TRUE.

**Two-pipe qualitative model** *(p.482-483)*: In the context of qualitative physics, compiling a model into its prime implicates allows simple propagation (BCP) to derive inputs from outputs and vice versa, which BCP on the original confluences cannot do.

**BCP completeness properties** *(p.484)*: BCP is reflexive-complete and partially literal-complete for a broader range of clause types. In particular, BCP is reflexion-complete for any clause set which, by substituting literals with their negations, can be put in Horn clause form.

**Prime implicate explosion for full LTMS** *(p.484)*: Unfortunately, the number of prime implicates of a set of formulas is often exponential in the size of the formulas. Therefore, unless the formula representing the encoding of the entire database is small, this technique is somewhat impractical.

**Pressure regulator prime implicates** *(p.485-486)*: Converting into formulas and providing no initial labels, the full LTMS yields 2,814 prime implicates. Of the 2,814 prime implicates, 496 are such definite clauses. If we regard all except P, P1 and Q as internal variables about which we will make no additional queries, there remain 21 prime implicates. Equivalently, of the 2,814 prime implicates, only 50 do not contain internal variables, and 21 of these are definite clauses. Thus we see that the input-output behavior of the pressure regulator can be compiled into relatively few very simple clauses.

**Lazy full LTMS efficiency** *(p.487-488)*: A lazy full LTMS algorithm never computes the consensus of a satisfied clause. Instead, every such consensus calculation is deleted, and the satisfied clause is marked as dirty such that if it ever becomes non-unit (per Section 9.5) consensus calculations will proceed. As a consequence of this efficiency consideration, the LTMS lazily actually computes the set of prime implicates. Instead it computes a set of unsubsumed implicates which are sufficient for BCP alone to achieve full completeness for the current assumption labels.

**CMS four properties** *(p.489)*: The CMS maintains for each literal l a set of environments (the label) having four properties: (1) Soundness: l holds in each Ei. (2) Consistency: cannot be derived from any E given C. (3) Completeness: every consistent environment E in which l holds is a superset of some Ei. (4) Minimality: no Ei is a proper subset of any other.

**Consensus commutativity optimization** *(p.491)*: Tison introduced the key intuition which suppresses the majority of the consensus calculations. The consensus operation is often commutative and associative. If we have three clauses A, B, and C, typically computing consensus(A, consensus(B, C)) produces the same result as other orderings. Tison's method tells us that the result will be irrelevant because all the useful consensus calculations with respect to B have already been made.

**Discrimination trie properties** *(p.494-495)*: Because clauses are canonically ordered, discrimination trees have the important property that the id of any edge is less than the id of any edge appearing below it at any depth. This data structure is called a trie, which has been explored extensively for representing dictionaries of words.

## Design Decisions and Rationale

**Incremental label update** *(p.450)*: The intuition behind the algorithm is that it assumes that node labels are correct before the introduction of the new justification and therefore only propagates the incremental changes caused by a new justification.

**Constructing solutions via GOAL node** *(p.452-453)*: Many problem-solving tasks are reducible to the following simple model: there are a set of choice sets (characterized in terms of assumptions), and any solution must pick one choice from each set. The approach creates a GOAL node and reads off the solutions from its label. The label for the GOAL node will contain all the solutions to the problem-solving task. If choice sets are known in advance, this is elegant but leads to efficiency problems on larger tasks since the label can be exponential in choice set size.

**Constructing solutions directly** *(p.453-454)*: Has two major drawbacks: (1) it relies on a general label updating algorithm, while the justification structure is very stylized; (2) during problem solving many intermediate goals might be constructed and discarded, leaving large labels which are never looked at again but occupy a significant portion of available memory. Therefore most ATMS implementations include a specialized solution construction procedure called `interpretations`, which given a set of choice sets and returns a set of environments representing the solutions.

**Defaults in ATMS** *(p.454-455)*: To simultaneously support both the defaults and the choice set paradigm for defining solutions, the ATMS `interpretations` procedure is supplied two arguments: the set of choice sets, and all the assumptions that are to be treated as defaults. This procedure returns solutions which have the property that they pick one choice from each choice set and that no default assumption can be added to them without introducing an inconsistency.

**No contradiction handling in ATMS** *(p.456)*: There is no concept of contradiction handling in the ATMS. An existing ATMS can be changed by `change-atms`, which takes the same keyword arguments as `create-atms`.

**Why extend BCP to formulas** *(p.471)*: The LTMS implementation of Chapter 9 achieves a subtle balance between two tensions: on the one hand it has sufficient expressive power to represent arbitrary formulas. On the other hand it is based on an efficient but incomplete Boolean constraint propagation algorithm. The ATMS of Chapter 12 accepts only Horn clauses. We would like to be able to generalize the ATMS to accept arbitrary clauses and formulas.

**Prime implicates reduce redundancy** *(p.474)*: It is sufficient to only use the prime implicates in our encoding. This greatly reduces the number of implicates needed to achieve formula BCP.

**Lazy LTMS design** *(p.487)*: The basic difficulty with the previous full LTMS is that it may be extremely difficult to construct the prime implicates, and there might well be an exponential number of them. Instead, if we reduce the LTMS database to prime implicates, all BCP propagations are one step long and could just as well have been done by table look-up. The lazy variant avoids computing consensus of satisfied clauses.

**CMS design rationale** *(p.488)*: The ATMS of Chapter 12 is restricted to accepting only Horn clauses as input justifications. The prime implicate formulation provides a way of extending the ATMS to accept arbitrary clauses (and formulas) as input. Such a system is sometimes called a clause management system (CMS).

**Discrimination trie for subsumption** *(p.493-494)*: A key observation is that we are maintaining a database of unsubsumed clauses. We need to implement 3 transactions with this database: (1) Check whether clause x is subsumed by some clause of the database. (2) Add clause x to the database. (3) Remove all clauses from the database which are subsumed by x. Instead of the most obvious implementation using subsetp, which would be computationally disastrous, the discrimination trie approach is used.

## Implementation Notes

**ATMS interface procedures** *(p.456)*:
- `create-atms`: Returns a datastructure containing the entire state of the ATMS.
- `change-atms`: Modify an existing ATMS with same keyword arguments.
- `true-node?`, `in-node?`, `out-node?`: Query node status in environments.
- `node-consistent-with?`: Check if there exists some consistent environment which is a superset of the node's environment.
- `create-atms`, `create-node`: Standard constructors.
- `assume-node`: Mark a node as an assumption.
- `justify-node`: Create a justification (identical to JTMS).
- `interpretations`: Fundamental procedure for interpretation construction. Returns a set of consistent environments in which at least one assumption from each choice-set holds.
- `explain-node`: Returns an explanation for why a node holds in an environment. The explanation returned is a list of justifications or CADR `(assumption)`.

**Key difference from JTMS/LTMS** *(p.441, 456)*: The ATMS has no concept of contradiction handling. The JTMS node has a `belief` field; the ATMS node has a `label` field (list of environments). A node is identical to the JTMS counterpart; a justification is identical to the JTMS justification.

**Environment creation and management** *(p.464)*: The ATMS maintains env tables. Every unique instance of the env datastructure is assigned to every environment. The most basic way to create a new env structure is `create-env`, which presumes that the set of assumptions supplied is properly sorted (via `assumption<-order`) and that no datastructure has yet been created for this environment. `union-env` returns the environment resulting from the union of its two arguments. `cons-env` returns the environment resulting from adding an assumption to an environment.

**Processing nogoods** *(p.465)*: `new-nogood` is called whenever the ATMS discovers that an environment, thus far presumed to be consistent, is nogood. It first removes the newly nogood environment from all the node labels it appears in by calling `remove-env-from-labels`. It then adds the new nogood to the nogood table, then it removes any subsumed nogoods from the table. The nogood table is kept in minimal form because every new environment must be checked to see whether it is nogood by comparing it to every nogood in the nogood table.

**Explanation generation** *(p.465-466)*: The procedure `explain-node` constructs an explanation for why a node follows from the set of assumptions represented by env. It is important to note that the naive way of constructing explanations by just recursively identifying any justification that supports a label environment is not guaranteed to construct well-founded explanations if the dependency graph contains cycles. The only way to truly find a non-circular support is to ignore the node labels and trace back through the justifications.

**Interpretation construction** *(p.466)*: `interpretations` is the basic procedure to create interpretations. The main work is done by two auxiliary procedures: `get-depth-solutions` and `extend-via-defaults`, which construct solutions by simple depth-first backtrack search. `extend-via-defaults` takes every interpretation that satisfies the choice-sets and adds as many defaults to it as possible.

**Full LTMS interface** *(p.498-499)*:
- `(create-ltms title &key ...)`: Creates an LTMS with keyword arguments for `node-string`, `default-node-string`, `debugging`, `checking-contradictions`, `contradiction-handler`, `enqueue-procedure`, `complete`, `delay-sat`.
- `complete` keyword: Three options -- `nil` (conventional BCP only), `:DELAY` (lazy full LTMS), `t` (complete full LTMS).
- `(add-formula ltms formula &optional informant)`: Adds the prime implicates of the formula to the clause database.
- `(change-ltms ltms &key ...)`: Can change the type of LTMS used.
- `(complete-ltms ltms)`: Performs consensus computations which have been delayed because :COMPLETE flag is set to :DELAY. No effect if :COMPLETE is t or nil.
- The `support-for-node` procedure is extended: the second and third elements of the list are two clauses which resolve to produce this one. This information is stored in the `clause-informant` slot of the clause currently supporting the node.

**Retraction in full LTMS** *(p.499)*: Notice that the LTMS procedure `set-truth` queues any nodes that receive new TRUE or FALSE labels for delayed consensus computations. New consensus computations are only called for when a retraction is done. Therefore, any call to `retract-assumption` eventually leads to a call to `ipsa`, which either performs the delayed consensus computation or delays it again.
