## Key Definitions

- **Path record:** Each path explored is encapsulated in a path struct, and the set of these structs is cached in the variable *open-path*. The original state variable was the initial state. *(p.50)*

- **`path-grst-assurer`**: A procedure that assumes its argument is a path struct corresponding to a legitimate solution. It also assumes `pr-exist-element-printer` is a procedure that takes an operator instance and a state and produces a string that makes sense to the user. *(p.50)*

- **Best-first search (beam-solve)**: An alternative to depth-first and breadth-first search that requires a *filler supply* procedure which, given a state, provides an estimate of the distance remaining to the goal. The secret of `beam-solve` is to update the queue so that the "best" path will be the next one tried, carried out when queues are stored as the `run-loop`. A temporary variable queue is initialized so that the state we just expanded is no longer available for consideration. *(p.51)*

- **Beam search**: Final variation of classical search strategies. `beam-solve` takes an optional argument `n`, a maximum size of the queue, which corresponds to the "width of the beam" used to search through the space. Updating the queue works almost exactly like `best-solve`. The only difference is that we clip the queue off whenever it gets longer than n, using `nstd` to access the new end of the list and `setf` to do the clipping. *(p.51)*

- **Operator (in subway domain)**: Since there is only one operator in the subway domain (the act of taking a subway line from one station to another), a unified `take-line` operator is defined, and the definition of `ops` instances put into the procedure `subway-operator-finder`. An instance of an operator in the subway domain is defined to be `(take-line (Old-station) (line) (NewStation))`. *(p.54)*

- **Attraction methods (algebra)**: Methods that involve moving terms to move occurrences of x closer together. The intuitive role of attraction methods, in bringing occurrences of x closer together, can be generally defined as reducing the total distance between occurrences of x in the tree, as measured by the number of steps required to move through the tree from one occurrence to another. *(p.60)*

- **Collection methods (algebra)**: Methods that reduce the number of occurrences of x. They reduce the number of variables present. Here we now have three kinds of variables: *loop variables* (such as x), *pattern variables* (such as `?`), and *algebraic variables*. *(p.61)*

- **Isolation methods (algebra)**: Methods that remove stuff that is not the unknown from the left-hand side of the equation by moving it to the right-hand side. They "strip off" layer after layer of expressions from the left-hand side until the single remaining x stands alone. *(p.59)*

- **Least dominating term**: A term that contains at least two subterms that contain the unknown. Both attraction and collection laws are applied only to least dominating terms. *(p.61)*

- **Simplifier**: A set of rewrite rules. An expression matches `<pattern>`, then it is replaced by `<result>`. A simplifier consists of a set of such rules. The knowledge of simplification can be encoded by a set of such rules, making the job of maintaining and extending this knowledge easier. *(p.72)*

- **`:EVAL` special form**: A form to be used as such cases, as an instruction to the substitution facility that it should evaluate an argument. Thus the right-hand side of the simplification rule could be written with `:EVAL`. *(p.67)*

- **`:SPLICE` special form**: When a subexpression starts with `:SPLICE`, the rest of that subexpression should be spliced into the containing list rather than simply being a member of it. The use of `:SPLICE` is described in Section 3.6.4. *(p.67)*

- **Segment variables**: A kind of variable that can only match a single item in a list (element variables) vs. segment variables that can match multiple items. Segment variables were motivated by the need to manipulate algebraic expressions, where the added complexity of non-deterministic matching was worthwhile. *(p.65)*

- **Pattern variables**: Variables represented by symbols with "?" as their first character. Thus `?foo`, `?BAR`, and `?Gremlin` are variables, while `Foo`, `bar1`, and `0Gremlin` are not. *(p.72 - Chapter 4)*

- **Unification (in pattern-directed inference)**: A form of pattern matching first developed in logic. Two patterns *unify* if there exists a set of values for their variables so that, when substitutions are made, the patterns become identical. This is somewhat different from the form of pattern matching used in the CPS algebra system. *(p.71-72)*

- **Assertion**: Data, whether it be raw information, facts about the world, or the consequences of internal deliberations, is encoded in the form of assertions. Assertions take the form of lists, containing symbols, lists, and numbers, such as `(Robot Robbie)`, `(Classof (Physical-Being Robbie) (Mortal Robbie))`. *(p.87)*

- **Assert! and fetch**: The primary interface between the assertion database and the rest of the system. `assert!` inserts a form into the database as a new assertion (if the form has already been asserted, `assert!` does nothing). Given a pattern, `fetch` retrieves assertions in the database that match it. *(p.88)*

- **Rule**: Procedural knowledge expressed in rules. Each rule has two parts, a *trigger* and a *body*. The trigger is a pattern that specifies the kinds of assertions to which the rule is intended to respond. Whenever an assertion matching the trigger enters the database, the rule will "wake up" and execute its body on that piece of data. *(p.90-91)*

- **Database class (class indexing)**: An indexing technique where the database is organized using class indexing. The idea of class indexing is to partition both assertions and rules into *dbclasses* (i.e., *database classes*), corresponding to sets whose elements are likely to match. The class of an assertion (or trigger pattern) is defined as its leftmost constant symbol. *(p.96-97)*

## Algorithms and Pseudocode

- **`beam-solve`**: Best-first search with queue width limiting. Takes an optional argument `n` for maximum queue size. Similar to `best-solve` but clips the queue when it exceeds `n` elements. *(p.51)*

- **Boston subway navigation operators**: `print-path-answer`, `pr-exist-element-printer`, `setup-subway-problem`, `OPERATORS` (containing the symbol `take-line`) to define the subway search operators. Definitions are illustrated in file `boston.lisp`. *(p.53-55)*

- **`dsolve`**: Depth-first subway solver. Example: `(dsolve *bundy* a)` produces solution path finding the route from Logan Airport to Kendall Square. *(p.56)*

- **`bsolve`**: Breadth-first subway solver. Example: `(bsolve *bundy* a)` examines 17 states and produces optimal answer. *(p.56)*

- **CPS algebra system file structure** *(p.68)*:
  - `match.lisp` - The pattern matcher
  - `algebra.lisp` - Operator implementations and environmental setup
  - `simplify.lisp` - A simple algebraic simplifier

- **Pattern matcher (`match.lisp`)** *(p.68-70)*: The basic routine is `match`, which takes a pattern (pat) and an expression (dat). It also accepts an optional argument `dict`, a dictionary of bindings. Any `match` must respect the variables already bound in dict. Match returns a dictionary corresponding to the complete set of bindings required to complete the match. The special value `:FAIL` for a dictionary indicates that the match is impossible. Basic structure is a recursive tree walk, stepping through the pattern and expression together looking for mismatches and potential variable bindings. *(p.68)*

- **`match-element-var`**: Straightforward. It uses `lookup-var` to access the value for the variable from the dictionary; if there is an existing value, the restriction associated with the variable of `attr` is applied to the potential new value, and if successful, a new dictionary is returned which includes the old dictionary plus the binding of the element variable to this piece of the pattern. Otherwise the match fails. *(p.69)*

- **`match-segment-var`**: Has the same overall structure as `match-element-var`. See if the variable has already been bound, check its value if so, and otherwise bind it. But since we haven't stored the value exactly, the procedure `try-segment-var` is used, which creates a copy of the segment using `>list`. *(p.69)*

- **`substitute-in`**: Uses results of a match (a dictionary of bindings). Handles `:EVAL`, segment variables, and `:SPLICE` forms. Three differences from `match`: (1) segment variables whose values are spliced into new expressions using `append`, (2) forms `(:eval <form>)` apply dictionary substitutions and evaluate, (3) `:SPLICE` subexpressions are spliced in rather than nested. *(p.71)*

- **`simplify-it`**: The main entry point into the simplifier. It calls `simplify-it` to do the real work. `try-simplify-rules` is used to see if a rule is applicable; if so, it finds a match, then `substitute-in` is used to produce the replacement expression. *(p.73)*

- **Algebra operator procedures** *(p.74-75)*: The rules used by simplify are stored in the global variable `*algebra-rules*`. The first thirteen rules take care of special-case arguments to operators. The next four rules handle equivalences involving multiplication, squaring, and exponentiation. The next nine rules reduce expressions when numerical values are available. The last two rules provide some canonicalization.

- **`algebra-distance`**: Provides a heuristic distance estimate for algebraic expressions; the predicate `sorted?` is satisfying. *(p.73-74)*

- **Rule body implementation strategy** *(p.98-99)*: Rules are lexically scoped. Bindings found via pattern matching must be made available during body execution. Best strategy: arrange so values bound to pattern variables are bound as the Lisp values of the symbols that serve as pattern variables for the duration of the execution of the body of the rule. This means we can treat the body simply as a piece of Lisp code. The alignment strategy is preferred over substitution strategy for best combination of simplicity and efficiency.

## Data Structures

- **Pattern variable data abstraction** *(p.66)*:
  - `pattern-variable?` - Returns non-nil if its argument is a pattern variable, nil otherwise
  - `element-var?` - Returns non-nil if its argument is an element variable, nil otherwise
  - `segment-var?` - Returns non-nil if its argument is a segment variable, nil otherwise
  - `var-name` - Returns name of a pattern variable
  - `var-restriction` - Returns procedure of one argument corresponding to a restriction on the binding of the variable
  - `var-value` - Given a dictionary and a variable, returns its value; signals an error if not bound
  - `lookup-var` - Takes a dictionary and a variable; if the variable is bound, returns a pair whose second element is the value of the variable; otherwise returns nil
  - `bind-element-var` - Takes a dictionary, an element variable, and a new value; produces a new dictionary consisting of the old dictionary augmented by the binding of the given variable to the given value
  - `bind-segment-var` - Like `bind-element-var` for segment variables

- **Assertion database** *(p.87-88)*: Assertions are stored in a database. In some systems the internal representation is more complicated than a piece of list structure, but here the term "assertion" refers to its form (the actual list structure expression) and the term "form" refers to the actual list structure of the assertion.

- **Database class (dbclass) indexing** *(p.96-98)*: The database is organized using class indexing. Each symbol has a unique database class. `DBclass(implies foo bar)` -> implies. `DBclass(foo)` -> foo. `DBclass((gremlin mumble) bar)` -> gremlin. Assertions and rules are stored by their leftmost constant symbol. This trades a slightly "Foxes-ish" flavor for efficiency in retrieval.

- **Rule execution environments** *(p.98-99)*: When rules are triggered, bindings form part of an environment for the rule. The environment is a table of bindings linking pattern variables to their values. The alignment strategy binds Lisp values of pattern variable symbols for the duration of rule body execution.

## Key Claims and Properties

- **Best-first search optimality**: It is straightforward to prove that `beam-solve` with unlimited beam width always leaves queue properly sorted. *(p.51)*

- **Alternative data structures for queues**: Good engineering practice consists in part of knowing when to optimize. The queue is not accessible to external routines, and hence it is safe to perform destructive operations on it. In general, destructive operations on data structures can be crucial in making programs efficient. *(p.52)*

- **Algebra search space is manageable**: Despite worst-case branching factor of 14, the analysis is far too pessimistic. The form of the equation strongly limits the set of laws applicable at each stage, so the branching factor is invariably smaller than 13. Many pairs of transformations negate each other, so the search space is not actually a tree, but a (much smaller) graph. *(p.58)*

- **Bundy's classification of algebra methods**: Bundy [2, 3] suggested that what expert mathematicians have — that beginners lack — is a set of implicit methods which control how they use algebraic laws. Bundy divides these implicit methods into three distinct categories: attraction, collection, and isolation. *(p.59)*

- **Simplification is NP-hard**: It is important to realize that algebraic simplification is actually an extremely hard problem. In fact, showing that any two arbitrary algebraic expressions are identical is NP-hard. Consequently, the simplifier only handles some easy cases of removing constants, combining numerical subexpressions, and canonicalizing expressions to make finding applicable operators easier. *(p.67)*

- **Search is unavoidable but insufficient**: Search by itself is no longer viewed as the essence of intelligence. Clearly search is a part of it, but only a part. A search routine is just like a routine for sorting or taking square roots or printing numbers: simply a component that is used as part of a larger system. *(p.79)*

- **Control knowledge via domain-specific methods reduces search**: If Bundy's thesis is right, we will not see large differences between different search strategies because the additional control knowledge in the method formulation heavily constrains each step. *(p.76)*

- **Assertions have no particular meaning to the PDIS system**: Assertions have no particular meaning to themselves to the PDIS system. Any meaning they have for the system is entirely determined by how they are used by the system's rules. *(p.88)*

- **Rules have indefinite extent**: Each rule spawned remains in the database forever. This holds for rules created by nested specifications as well as for their progenitors. In this model, rules are never deleted from the system or made inactive; they always trigger on any matching assertion that appears. *(p.92)*

- **Order-independence of pattern-directed inference**: Rules and assertions can be put into the system in any order you choose, and eventually the same set of consequences should be produced. A drawback of this property is that it becomes easier to generate combinatorial explosions and can be harder to maintain coherency. *(p.92)*

- **The importance of quiescence in TRE design**: Given a starting set of assertions, the operation of the TRE should stop after a finite number of rule executions. This can fail to happen if the system of rules "feeds on itself," producing an infinite number of assertions. We should not presume that rules will be executed or any particular order. *(p.95)*

## Design Decisions and Rationale

- **Using `defLine` and `defStation` macros for subway representation**: These macros allow a user to define a map of a subway while knowing very little about the internals of the representation. This is the way it should be. Modularity requires hiding irrelevant details from both programs and people. A guiding principle of all interface designs is that, wherever possible, any facility available to a human user should be available to other programs, and vice versa. *(p.53)*

- **Pattern matching design for CPS algebra vs. PDIS**: Pattern matching appropriate for pattern-directed inference systems is somewhat different from the form of pattern matching used in CPS algebra. Patterns can contain variables, but no value restrictions can be associated with variables. This stipulation forces us to put more of the knowledge into assertions and rules rather than special-purpose procedures. Furthermore, there are no segment variables; segment variables were motivated by the need to manipulate algebraic expressions, where the added complexity of non-deterministic matching was worthwhile. For most pattern-directed inference systems the extra overhead of considering segment variables is not worthwhile. *(p.88)*

- **Class indexing over linear search**: To organize the database, a single but powerful technique called *class indexing* (also known as *car indexing*) is used. It partitions both assertions and rules into dbclasses corresponding to sets whose elements are likely to match. *(p.96)*

- **Alignment strategy over substitution strategy for rule bindings**: The alignment strategy binds values to pattern variables as Lisp values. The substitution strategy instead analyzes the rule's body, replacing all references to pattern variables with forms that access the appropriate pattern variables. The alignment strategy is substantially more complex but allows analysis of code in the body in detail. The substitution strategy is only used when substantial traversing of the rule body is required for other reasons. *(p.99)*

- **Canonicalization for simplifier efficiency**: Having such an ordering is useful because it allows expressions to be turned into a canonical form, which simplifies further matching operations. A canonical form helps make expressions that mean the same look the same. Rewrite rules are used to sort the arguments to commutative operators, so that match is more likely. These reorderings can greatly reduce the complexity of the rewrite rules needed, and hence reduce the computational cost of the simplifier. *(p.73)*

- **Destructive operations justified for internal queues**: Since the queue is not accessible to external routines, and hence it is safe to perform destructive operations on it. *(p.52)*

- **Production rule systems vs. procedural deduction systems**: A third class of pattern-directed inference systems is made up of procedural deduction systems, such as MicroPlanner [12], Conniver [8], and Schemer [14]. Procedural deduction systems sometimes include rules, but generally organize their procedures around traditional programming constructs which include pattern matching and non-deterministic operators to provide more flexible control structures. Procedural deduction systems have received less attention than the other forms of pattern-directed inference systems, and have not seen widespread applications. *(p.93)*

## Implementation Notes

- **CPS search framework recap** *(p.50-51)*: The search procedures are built on `path` structs with `grst-assurer` for path formatting. Each search variant (`bsolve`, `dsolve`, `beam-solve`) is invoked by assigning `*open-path*` and calling the search engine with the appropriate problem setup.

- **Subway representation** *(p.53-54)*: Stations and lines stored in global variables `*station*` and `*lines*`. A subway station struct includes the name of the subway station, the lines that station is on, and optionally the coordinates of the station. The subway line has its name, its line and the list of stations on that line. Macros `defLine` and `defStation` provide the input interface.

- **Algebra rewrite rules storage** *(p.74)*: Rules are stored in the global variable `*algebra-rules*`. First thirteen handle special-case operators, next four handle equivalences, next nine reduce when numerical values available, last two handle canonicalization.

- **Pattern matcher data abstraction** *(p.66)*: Extensive data abstraction defined for pattern variables. The procedures `pattern-variable?`, `element-var?`, `segment-var?`, `var-name`, `var-restriction`, `var-value`, `lookup-var`, `bind-element-var`, `bind-segment-var` form the complete interface.

- **Segment variable matching implementation** *(p.69-70)*: The value of a segment variable is specified by two pointers into the original list, one to the start of the segment and the other to the end. These pointers are `beg` and `end` pointers. This makes it easy for `var-value` to figure out what kind of variable it is dealing with.

- **PDIS database: class indexing implementation** *(p.96-98)*: Unless an assertion and a trigger are in the same database class, they cannot match (for our purposes). This means that when adding a new assertion we need only test it against rules indexed under its own database class, and when adding a new rule we need only test it against assertions in its own database class. We suppose the leftmost symbol in a pattern is a variable; in that case, we take the database class of that variable's value. We consider it an error if the variable is unbound.

- **TRE architecture** *(p.94-95)*: TRE (The Rule Engine) is the implementation of the pattern-directed inference system model. Rules and assertions are added incrementally. The goal of the execution strategy is to ensure that each rule is run on every assertion that matches its pattern. Whenever an assertion is added (each rule that might trigger on that assertion must be retrieved and tested against it), a successful match results in a set of bindings which form part of an environment for executing the body of the rule.
