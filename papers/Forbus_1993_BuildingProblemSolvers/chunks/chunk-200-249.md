## Key Definitions

- **JTMS data structure fields** *(p.202)*: An instance of the JTMS contains: `title` (ignored by JTMS, for debugging), `node-counter` (unique name for nodes), `just-counter` (unique name for justifications), `nodes` (list of all nodes created), `justs` (list of all justifications), `debugging` (flag to trace internals), `contradictions` (list of nodes declared contradictory), `assumptions` (list of nodes declared assumptions), `checking-contradictions` (flag, defaults to `t`), `node-string` (inference engine-supplied procedure for descriptive string of a node).

- **TMS node fields** *(p.203)*: `index` (integer serving as unique name), `datum` (supplied by inference engine), `label` (current belief status: `:IN` = believed, `:OUT` = not believed), `support` (field is nil if node is labeled `:OUT`; if node is derived, contains the justification currently providing support; if node is an enabled assumption, contains `:ENABLED-ASSUMPTION`), `justifications` (set of justifications that could provide support), `consequences` (justifications that use this node as antecedent), `contradictory?` (if non-nil, this node represents a contradiction), `assumption?` (if non-nil, this node should be treated as an assumption whose belief can be explicitly enabled/retracted; if value is `:DEFAULT`, the auxiliary procedure `enable-defaults` maintains the semantics of default for such nodes).

- **Justification (just) fields** *(p.204)*: `index` (integer for unique name), `informant` (inference engine-supplied description), `consequence` (the node this justification can support), `antecedents` (nodes which must be believed for this justification to provide support).

- **JTRE structure** *(p.225)*: The `jtre` struct differs from `ftre` struct in several respects: all fields for caching local rules and data are gone (no longer makes sense with a JTMS); the two queues have been re-merged into one (since assumption making is not an activity typically done via pattern-directed rules in a JTMS-based system). New fields: `jtms` (the JTMS associated with this JTRE), `datum-counter`, `rule-counter`.

- **JTRE database assertion (datum struct)** *(p.226)*: Fields: `id` (integer identifier, unique within JTRE), `jtre` (pointer to the JTRE it belongs to), `lisp-form` (the list structure comprising the form of the assertion), `tms-node` (pointer to corresponding node in TMS), `dbclass` (pointer to database class it belongs to), `assumption?` (whether the assertion is an enabled assumption), `plist` (property list for extra information).

- **JSAINT architecture** *(p.244)*: Three main parts: (1) Central controller - gathers suggestions from Integration Library, chooses subproblems to work on, ensures resource limitations are not exceeded; (2) AND/OR graph - maintains status of progress on problems and subproblems, detects when problems have been solved or cannot be solved; (3) Integration Library - provides direct solutions to simple problems (standard forms), makes suggestions for decomposing complex problems (transformations).

- **JSAINT control vocabulary** *(p.246-249)*: Predicates for representing problems and goals:
  - `(integral <expr> <var>)` represents an integral expression
  - `(integrable <integral>)` and `(failed <integral>)` indicate solvability
  - `(calculation-of <P> <A>)` holds exactly when A is the result of solving problem P
  - `(open <P>)` believed when problem P has been expanded but is not yet solved or known to be unsolvable
  - `(open <G>)` believed when goal G has been expanded but is not yet achieved or known to be unachievable
  - `(relevant <PG>)` holds when goal or problem PG is still potentially relevant to solving the original problem
  - `(expanded <PG>)` is true exactly when JSAINT has begun working on problem or goal PG

## Algorithms and Pseudocode

- **Adding a justification algorithm** *(p.200)*: When the inference engine adds a justification: (1) The JTMS adds the new justification to the dependency network; (2) The JTMS checks whether the consequence of the justification is already labeled `:IN`. If so, the JTMS returns control to the inference engine. Otherwise, it checks whether the justification is satisfied. If so, it labels the consequence `:IN`, and makes the new justification the supporting justification for the consequence. Continues recursively for any nodes which become `:IN`.

- **Retracting an assumption (two-phase algorithm)** *(p.200)*: The correct algorithm for retracting an assumption without creating meaningless circularities proceeds in two phases. Phase 1: all nodes whose well-founded explanation contains the retracted assumption are labeled `:OUT`. This is done by looking at all the supporting justifications, deleting these nodes out, and continuing recursively. Phase 2: only when the first phase is complete, every node that was labeled out in the first phase is analyzed to see if alternative support can be found for it.

- **Canonical sequence for justify-node** *(p.205)*: When belief in a new fact is established: (1) Make this particular node be believed (`make-node-in`). (2) Compute what other nodes must be believed as a consequence (`propagate-inness`). (3) See if any contradictions result (`check-for-contradictions`).

- **Assumption retraction steps (make-node-out)** *(p.207)*: (1) Make the particular node be disbelieved (i.e., `:OUT`, via `make-node-out`). (2) Forget all nodes that depend on this particular node (`propagate-outness`). (3) Attempt to find alternative support for nodes which have been retracted (`find-alternative-support`). The `propagate-outness` procedure works by recursively extracting all justifications which relied on the given node. As each node is forgotten it is placed on the out-queue. `find-alternative-support` then attempts to find alternative support for everything on this queue.

- **Dependency-directed search (DDS) for N-Queens** *(p.233)*:
  ```lisp
  (defun DDS (choice-sets)
    (if (null choice-sets) (record-solution)
      (dolist (choice (first choice-sets))
        (unless (nogood? choice)
          (assume! choice)
          (DDS (rest choice-sets))
          (record-nogood! choice)
          (retract! choice)))))
  ```
  Key additions over chronological search: `nogood?` checks if the choice is already known to be inconsistent with the dependency network. If so, there is no point in assuming it.

- **Consumer architecture for rule execution** *(p.219-221)*: JTRE uses a consumer architecture. Rule instantiations are passed into the TMS as consumers and stored with TMS nodes corresponding to the rule antecedents. The algorithm checks whether the belief condition is satisfied, and the consumer will re-trigger when the triggering assertion's label changes to match the condition.

- **JTRE rule trigger syntax** *(p.220)*: Changed from FTRE. The trigger pattern now has form `(:condition :pattern :options)` where `:condition` is one of `:INTERN`, `:IN`, or `:OUT`, and `:options` are the `:VAR` and `:TEST` keywords.

## Data Structures

- **JTMS** *(p.202)*: Central record containing all state: nodes list, justifications list, contradiction list, assumptions list, debugging flag, checking-contradictions flag.

- **TMS Node** *(p.203)*: Nodes carry label (`:IN`/`:OUT`), support pointer (to the current justification or `:ENABLED-ASSUMPTION`), lists of justifications and consequences, contradiction and assumption flags.

- **Just (Justification)** *(p.204)*: Records a dependency: informant (from inference engine), consequence node, list of antecedent nodes.

- **JTRE datum** *(p.226)*: Assertion record linking inference-engine assertions to TMS nodes. Fields: id, jtre, lisp-form, tms-node, dbclass, assumption?, plist.

- **AND/OR graph** *(p.243-244)*: JSAINT uses an AND/OR graph (implemented as assertions in the JTRE database) to represent problem decomposition. OR branches represent alternate methods, AND branches represent subproblems that must all be solved. Not a separate data structure but JTRE assertions maintained by rules.

## Equations

- **Standard integration form example** *(p.242)*: $\int x \, dv = \frac{v^2}{2}$ (simple standard form)

- **Integration identities used by SAINT** *(p.242)*:
  - $\int c \, g(v) \, dv = c \int g(v) \, dv$ where $c$ is constant in $v$
  - $\frac{1}{c} \int f(cx + d) \, dx$ (linear substitution)

- **Let $g(v)$ be the integrand** *(p.242)*: For each non-constant nonlinear subexpression $v(x)$ such that neither its main connective is minus nor is it a product with a common factor, and such that the number of non-constant factors of the fraction $\frac{f'(v)}{g(v)}$ after cancellation is less than the number of factors of $g(v)$, try substituting $u = v(x)$.

## Key Claims and Properties

- **JTMS operations cannot be safely aborted** *(p.201)*: The JTMS makes certain assumptions about its processing environment: JTMS operations cannot be safely aborted, nor can one JTMS operation be safely invoked before another is finished; nor can multiple processes safely access the same JTMS simultaneously.

- **Well-founded support is essential for correct retraction** *(p.200)*: A naive approach to retraction that immediately looks for alternative support can cause circularities. The correct algorithm operates in two phases: first labeling all dependent nodes `:OUT`, then searching for alternatives.

- **Duplicate justifications decrease efficiency but cause no errors** *(p.205-206)*: Calling `justify-node` twice with the same arguments will result in the creation of distinct data structures representing the same justification. No errors result, but duplicate justifications decrease efficiency. It is rare to find redundancy checks in TMS implementations.

- **Justification-satisfied? condition** *(p.206)*: A justification is satisfied when all of its antecedent nodes are `:IN`.

- **`:TEST` in JTRE differs from FTRE** *(p.220)*: In FTRE a common use of `:TEST` was to skip executing rules whose conclusions were currently known, by using `fetch` as part of a `:TEST` option. This worked because the same fact could be asserted repeatedly in different contexts. This is not the case with a truth maintenance system.

- **Rule triggers must not examine TMS state** *(p.221)*: Tests involving only structure will always return the same result because the structure of each assertion is fixed for all time. The `:TEST` must not examine the state of the TMS; it must not use any global variables, and it must not examine the state of the database in any other way.

- **Dependency-directed search outperforms chronological for complex problems** *(p.221-223)*: Tables 8.1 and 8.2 show performance comparisons for N-Queens. For 8-Queens: FTRE uses 15,720 assumptions in 95.13s; JTRE uses 2,514 assumptions in 153.68s. Dependency-directed search makes many fewer assumptions but each is more expensive due to TMS overhead. The crossover favors dependency-directed search for more complex problems where the overhead of testing assumptions is larger.

- **Both search strategies are exponential** *(p.221)*: Dependency-directed search only postpones the day of reckoning by increasing the range of problems that can be solved with a fixed computational resource.

- **Eight constraints for TMS-inference engine integration** *(p.214-215)*: (1) Inform TMS when new node needed and connect to assertion; (2) Retrieve TMS node for any assertion; (3) Interface for marking premises/assumptions and enabling/retracting; (4) Facilities for representing justifications; (5) Facilities for inspecting node labels; (6) Method for handling contradictions; (7) Facilities for constraints on beliefs in rule trigger conditions; (8) Rules executed only when both belief constraints and syntactic matching constraints are satisfied.

- **SAINT solved MIT calculus exam problems** *(p.241)*: SAINT was one of the first AI programs. It performed indefinite integration and solved most problems from an MIT calculus exam. JSAINT reconstructs a subset of SAINT's functionality using JTRE.

- **JSAINT design principles** *(p.240-241)*: (1) Explicit representation of control knowledge; (2) Control via assumptions to avoid combinatorial explosions; (3) Decomposition into FTRE (pattern-directed inference) and Controller; (4) Exploration generation via TMS for understanding rationale.

## Design Decisions and Rationale

- **Why separate JTMS from inference engine** *(p.201)*: The JTMS makes assumptions about its processing environment (no safe abort, no interleaving, no multi-process access). These are concerns beyond the JTMS scope. A truly bulletproof JTMS would use guarded processes and proper process locking primitives.

- **Why use unique name counters** *(p.202)*: `node-counter` and `just-counter` provide "unique names" for nodes and justifications. Common trick for making defstructs that print compactly yet remain distinguishable.

- **Why datum struct for assertion mapping** *(p.216)*: In FTRE, an assertion was simply the corresponding Lisp expression. In JTRE, more properties need to be associated with an assertion (its TMS node, whether it is an enabled assumption). A datum struct was introduced for bidirectional mapping between assertions and TMS nodes. This design provides the ability to move from any perspective: assertion datum, Lisp expression, TMS node to any other.

- **Consumer strategy chosen over probing strategy** *(p.219)*: Two approaches: (1) Probing: inference engine probes TMS for each rule instantiation, executing only when beliefs satisfied -- disadvantage of burdening inference engine with monitoring; (2) Consumer: pass rule instantiations into TMS and let it schedule execution. Consumer is simpler because consumers are stored with TMS nodes and fired on label changes.

- **Filter on assertion structure before belief** *(p.222)*: Better to filter based on assertion structure because an assertion that does not match the trigger structurally will never do so, and that rule-assertion pair can be forever ignored.

- **Check belief after matching but before body execution** *(p.222-223)*: Modified the original execution sequence: first check if assertion's TMS node label matches the condition. If not satisfied, store the original body as consumer on the TMS node. When the node's label changes appropriately the rule is queued for execution.

- **`retract!` requires same source of support** *(p.227-228)*: The implementation of `retract!` carefully checks whether the assertion is an enabled assumption and whether it has the same source of support as was given as the reason for enabling. This constraint prevents tasks from accidentally retracting assumptions they were not responsible for.

- **`referent` with `:virtual` flag** *(p.229)*: The `:virtual` flag in `referent` controls whether an assertion has been mentioned at all, not just whether it is believed. When debugging, one may want to ask if a fact is not in the database without triggering any `:INTERN` rules. Intuitively, an interrogative should not add structure to the database.

- **JSAINT uses FTRE/JTRE decomposition** *(p.240-241)*: The inference engine is decomposed into two parts: FTRE (pattern-directed inference system) responsible for maintaining the assertional database and drawing obvious conclusions, and Controller responsible for guiding the problem-solving process. Dividing the process adds overhead but improves overall efficiency through better control.

- **AND/OR graph vs AND/OR tree** *(p.248)*: JSAINT uses an AND/OR graph rather than tree. SAINT used an AND/OR tree. The choice diverges slightly because JSAINT implements the graph as assertions. One could implement a tree via assertions by reifying each problem and goal as unique objects.

## Implementation Notes

- **JTMS program structure** *(p.201)*: Seven parts: (1) Definitions, datastructures, initialization; (2) Basic interface for programs using JTMS; (3) Adding justifications and computing results; (4) Retraction and finding alternative support. Additional: (5) Contradiction-handling interface for signaling contradictions to inference engine; (6) Well-founded support procedures; (7) Inference engine stubs and stand-alone procedures.

- **JTMS code standalone capability** *(p.201)*: The code in `jtms.lisp` is a simple justification-based truth maintenance system. The end of the file contains some auxiliary procedures to allow the JTMS to be used by itself.

- **`create-jtms` procedure** *(p.204)*: Creates a new JTMS instance with keyword arguments for initializing fields. `change-jtms` allows changing externally settable fields.

- **Utility procedures** *(p.204)*: `node-string`, `debugging-jtms`, `tms-error` are simple utilities for printing and unusual conditions. `default-node-string` constructs a string describing a node from the datum field.

- **`assume-node` and `make-contradiction`** *(p.205)*: `assume-node` converts a non-assumption node to an assumption node and enables it. `make-contradiction` converts a non-contradiction node to a contradiction node.

- **Contradiction macros** *(p.208)*: `without-contradiction-checking` turns off checking within its body. `with-contradiction-handler` temporarily changes the handler. Both use `unwind-protect` to ensure state restoration. Two settable variables control behavior: first, non-settable variable names are created to distinguish them from variables that might occur from `unwind-protect`. Second, `unwind-protect` ensures handler/flag are restored.

- **`default-assumptions` procedure** *(p.208-209)*: Achieves semantics of defaults. Called by inference engine usually right after contradiction handling. Uses the macro `just` to bring in all defaults unless they introduce a contradiction.

- **`supporting-justification-for-node` and `assumptions-of-node`** *(p.208-209)*: `supporting-justification-for-node` returns the supporting justifications for a node if there is one. `assumptions-of-node` finds the enabled assumptions of the current well-founded explanation for the node, searching backward through antecedents of justifications until enabled assumptions are found. Uses a marker to avoid revisiting graph parts.

- **`enabled-assumptions` procedure** *(p.209)*: Returns a list of all the enabled assumptions.

- **Stand-alone operation procedures** *(p.209)*: `why-node` shows status and source of support. `why-nodes` uses `why-node` for entire JTMS database. `ask-user-handler` calls `handle-one-contradiction` to remove the first contradiction. `handle-one-contradiction` finds enabled assumptions underlying well-founded explanation, prints them, allows user to type Common Lisp command. `explore-network` is higher-level interface.

- **JTRE interface** *(p.225-226)*: `(create-jtre <title> &key <debugging>)` creates new JTRE. Interface procedures parallel FTRE: `run`, `run-forms`, `show`. New variables: `assume!` and `assume!*` for interactive dialogs.

- **JTRE database procedures** *(p.227-229)*: `assert!` makes permanent TMS changes. `quiet-assert!` suppresses contradiction handling. `assume!` converts to enabled assumption. `retract!` retracts enabled assumption. `contradiction` marks TMS node as contradiction. All accept optional `*jtre*` argument.

- **JTRE interrogatives** *(p.228-229)*: `in?`, `out?`, `why?`, `assumptions-of`, `wfs`, `show-justifications`, `fetch`, `show-data`, `referent`, `get-tms-node`, `view-node`, `show-datum`, `get-datum`, `get-just`, `map-dbclass`, `get-tms-node`, `view-node`.

- **JTRE rule modifications** *(p.230)*: `parse-rule-trigger` modified to extract belief condition. `build-rule` modified substantially to handle TMS integration.

- **`shakedown-jtre` for testing** *(p.231)*: Runs standard examples and checks results match expectations. Very useful when making changes to internals.

- **N-Queens dependency-directed implementation** *(p.235-236)*: JTRE implements `try-in-context` via contradiction handler binding. `solve-queens` modified to create warnings when captures detected, uses TMS contradiction information to avoid repeating mistakes.

- **JSAINT organized around JTMS** *(p.240)*: JSAINT records both control and data dependencies in the JTMS. This allows it to explain its answers and operations. Integration Library is implemented as assertions in the dependency network, using JTRE rules.
