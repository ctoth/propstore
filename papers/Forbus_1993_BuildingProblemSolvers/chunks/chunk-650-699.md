## Key Definitions

**Definition 17.1** *(p.653)*: A system is a triple (SD, COMPS, OBS) where:
- SD, the system description, is a set of first-order sentences.
- COMPS, the system components, is a finite set of constants.
- OBS, a set of observations, is a set of first-order sentences.

**Definition 17.2** *(p.653)*: Given two sets of components Cp and Cn, define $D(Cp, Cn)$ to be the conjunction: $\bigwedge_{c \in Cp} AB(c) \wedge \bigwedge_{c \in Cn} \neg AB(c)$

**Definition 17.3** *(p.654)*: Let $\Delta \subseteq$ COMPS. A diagnosis for (SD, COMPS, OBS) is $D(\Delta, \text{COMPS} - \Delta)$ such that $SD \cup OBS \cup \{D(\Delta, \text{COMPS} - \Delta)\}$ is satisfiable.

**Definition 17.4** *(p.654)*: A diagnosis $D(\Delta, \text{COMPS} - \Delta)$ is a minimal diagnosis iff no proper subset $\Delta'$ of $\Delta$ is $D(\Delta', \text{COMPS} - \Delta')$ is a diagnosis.

**Definition 17.5** *(p.656)*: An AB-clause is a disjunction of AB literals containing no complementary pair of AB-literals. A positive AB-clause is an AB-clause all of whose literals are positive.

**Definition 17.6** *(p.656)*: A conflict of (SD, COMPS, OBS) is an AB-clause entailed by $SD \cup OBS$. A positive conflict is a conflict all of whose literals are positive. A minimal conflict of (SD, COMPS, OBS) is a conflict no proper sub-clause of which is a conflict of (SD, COMPS, OBS).

**Definition 17.7** *(p.657)*: A conjunction C of literals covers a conjunction D of literals iff every literal of C occurs in D.

**Definition 17.8** *(p.657)*: Suppose $\Sigma$ is a set of propositional formulas. A conjunction of literals $\pi$ containing no pair of complementary literals is an implicant of $\Sigma$ iff $\pi$ entails each formula in $\Sigma$. $\pi$ is a prime implicant of $\Sigma$ iff the only implicant of $\Sigma$ covering $\pi$ is $\pi$ itself.

**Constraint satisfaction problem** *(p.664)*: A problem P consists of the set of variables $v_1, v_2, \ldots, v_n$ and relations $R_1, R_2, \ldots, R_m$. Any collection of assignments of values to $v_1, v_2, \ldots, v_n$ which satisfies all of $R_1, R_2, \ldots, R_m$ is a solution to P.

**Node consistency** *(p.665-666)*: A node is consistent when the following formula holds: $\forall x \in D, R(x)$. That is, every value in the domain satisfies the unary constraint.

**Arc consistency** *(p.666)*: For a k-ary relation, it is arc consistent exactly when for each combination of values for $k-1$ of the variables, there is a value for the kth which satisfies R. For binary relations: $\forall x \in D_1 \exists y \in D_2, R(x,y)$.

**Allen's temporal relationships** *(p.684-685)*: Seven basic temporal relationships and their inverses:
- before(A,B) / after: A's temporal extent is earlier than B's
- during(A,B) / contains: A's temporal extent lies completely within B's
- equal(A,B): A's temporal extent is exactly the same as B's
- overlap(A,B) / overlapped-by: begins sometime in the middle of A, and ends sometime after A does
- meets(A,B) / met-by: A ends just before B begins, with no time "in between"
- starts(A,B) / started-by: A and B start at exactly the same time
- finishes(A,B) / finished-by: A and B end at exactly the same time

## Algorithms and Pseudocode

**TGDE measurement scoring** *(p.650-652)*: TGDE scores each measurement point by counting how many diagnoses predict each possible outcome value (0 or 1). The best measurement is the one that most evenly splits the current diagnoses. Measuring `<Cell (>> RIPPLE ADD)>` has cost 0.7548878. TGDE identifies the best places to measure as those that provide equal information (split the diagnosis set evenly).

**WALZER constraint operations** *(p.669)*:
- `exclude`: Takes two arguments, a cell and a value. Its effect is to remove that value from the list of possible values for that cell.
- `pick`: Takes two arguments, a cell and a value. Its effect is to force the given cell to have the given value.
- `update`: Takes a constraint as its argument. Its effect is to run the constraint's update procedure.

**Constraint update rules** *(p.669)*:
1. Each update procedure terminates in finite time.
2. The only information in the network an update procedure is allowed to draw upon are the values of the cells which are the parts of its associated constraint.
3. All effects on the constraint network occur through requests for `exclude` and `pick` operations.
4. Update procedures are deterministic; given the same values for a constraint's cells, the update procedure always generates the same set of `exclude` and `pick` requests.

**fire-constraints procedure** *(p.674)*: If the network is not overconstrained, fire-constraints proceeds to empty the queue. It performs all cell updates before any constraint updates to ensure that the full consequences of each constraint are recorded before the next constraint is used. The global time stamp is incremented with each constraint update.

**search-thru-plausible-cells** *(p.675-676)*: A simple depth-first search routine. It starts by seeing if the network is overconstrained. If it is, the contradiction procedure given to search-network is exercised. If not, it checks if the network is fully determined. The procedure handles three cases: (1) overconstrained -- invoke contradiction handler, (2) fully determined -- invoke solution handler, (3) undetermined -- pick a value for the first undetermined cell, recurse.

**Transitivity table for Allen's temporal logic** *(p.686)*: The transitivity table can be used to restrict the possible relationships between two intervals A, C as follows. For any intervals $I_1, I_2$, let $R(I_1, I_2)$ stand for the set of possible relationships between them. Given relationships $R(A,B)$ and $R(B,C)$, the entire set of possibilities can be found by taking the union of the transitivity results for each pair in the cross product of $R(A,B)$ and $R(B,C)$.

**Temporal database implementation** *(p.689-690)*: The system consists of three files:
- `timedb.lisp`: Temporal database and definitions
- `allen.lisp`: Definition of Allen's transitivity table
- `ttest.lisp`: Simple test cases

Key procedures:
- `find-transitive-relations`: Begins by determining the cell's name to get the intervals of the newly created relationship, then iterates over each relationship involving the first interval to find candidate third intervals to complete a triangle.
- `update-trel-transitivity`: Provides the update procedure for constraints. It computes `possible-trel-values` and then finds what subset of $R(A,C)$ is consequently forbidden through the `set-difference` call.

## Data Structures

**WALTZER network data structure** *(p.670-671)*: Three main data structures: `network`, `cell`, and `constraint`.

**Network structure fields** *(p.671)*:
- `cells`: Alist linking cells to their names
- `constraints`: Alist linking constraints to their names
- `name-test`: Procedure to use in looking up names
- `equality-test`: Procedure for comparing cell values
- `cell-queue`: Queue of operations on cells
- `constraint-queue`: Queue of constraints to update
- `constraint-space`: Queue of constraints to update
- `timestamp`: Global clock, useful in explanations
- `event-list`: List of constraint updates, also useful in explanations
- `debug?`: Controls printing of information about internal operations
- `status`: One of `:NEW`, `:QUIESCENT`, `:IN-PROGRESS`, `:OVERCONSTRAINED`
- `contradiction-reason`: If non-nil, a cell currently overconstrained
- `contradiction-hook`: Procedure to call when overconstrained
- `plist`: Cache for storing miscellaneous information about the network

**Cell structure fields** *(p.671)*:
- `name`: Name used for indexing
- `network`: The constraint network it belongs to
- `value`: Current set of consistent values
- `constraints`: Set of constraints it participates in
- `possible-values`: Domain of the cell
- `out-reasons`: Alist of reasons for each value excluded from the domain
- `plist`: Cache for miscellaneous information about it

The `out-reasons` alist provides a dependency record: if the entry for a value is `:IN`, it is believed to be consistent. Otherwise, the entry is a pair consisting of the constraint responsible for excluding the value and a time stamp indicating when in the propagation sequence the exclusion took place. *(p.671)*

**Constraint structure fields** *(p.672)*:
- `name`: Name used for indexing
- `network`: Constraint network it belongs to
- `parts`: Cells the constraint relates
- `update-procedure`: Procedure to call to update it
- `queued?`: If non-nil, already queued
- `plist`: Cache for miscellaneous properties

**Junction catalog** *(p.678-679)*: The core data structure is the `jlabel` struct. The global variable `*jlabel-table*` holds the junction catalog, and `*line-label*` is a constant representing the possible interpretations of lines. The `type` field refers to the geometric configuration. The `lines` field describes the assignment of interpretations a catalog entry makes to its constituent lines.

**Temporal database** *(p.689)*: Temporal databases are implemented via the `timedb` struct. The `intervals` field stores the intervals for the database, with `interval-id` providing a unique index for each interval. The `relation` field holds the possible temporal relationships (an explicit field of temporal database rather than a global). The `transitivity-table` field holds transitivity tables in the form of nested alists. The `network` field holds the WALTZER constraint network associated with the database.

**Interval structure** *(p.689)*: The `interval` struct provides the internal representation of intervals. The `name` field holds the name for the interval used by external systems. Intervals are given names whose identity can be established by `eq`.

## Equations

**Diagnosis conjunction** *(p.653)*:
$$D(Cp, Cn) = \bigwedge_{c \in Cp} AB(c) \wedge \bigwedge_{c \in Cn} \neg AB(c)$$

**Number of possible diagnoses** *(p.654)*: Unfortunately, there may be $2^{|COMPS|}$ diagnoses.

**Node consistency** *(p.666)*:
$$\forall x \in D, R(x)$$

**Arc consistency for binary relations** *(p.666)*:
$$\forall x \in D_1 \exists y \in D_2, R(x,y)$$

**k-consistency** *(p.667)*: A network being k-consistent means that for any collection of $k-1$ variables which satisfy all the constraints on them, any kth variable also can be bound such that the constraints on all k variables are satisfied.

**Cost of enforcing k-consistency** *(p.667)*: The cost of enforcing k-consistency rises exponentially with increasing k. The best computational trade-off appears to be the combination of arc consistency followed by backtracking search.

**Allen's compact notation for disjunctive relationships** *(p.685)*:
$$A \{<, > \} B$$
indicates that {before(A,B)} or {after(A,B)} holds. Using abbreviations: b = before, bi = after, d = during, di = contains, o = overlap, oi = overlapped-by, m = meets, mi = met-by, s = starts, si = started-by, f = finishes, fi = finished-by, e = equal.

## Key Claims and Properties

**Theorem 17.1** *(p.654)*: If $D(\Delta, \text{COMPS} - \Delta)$ is a diagnosis, then there is a minimal diagnosis $D(\Delta', \text{COMPS} - \Delta')$ such that $\Delta' \subseteq \Delta$.

**Hypothesis 17.1 (Minimal Diagnosis Hypothesis)** *(p.654)*: If $D(\Delta, \text{COMPS} - \Delta)$ is a minimal diagnosis and $\delta \subset \Delta \subset$ COMPS, then $D(\Delta, \text{COMPS} - \Delta)$ is a diagnosis. This hypothesis does NOT hold in general. *(p.654)*

**Theorem 17.2** *(p.656)*: If $D(\text{COMPS, OBS})$ satisfies the IAB condition, then the minimal diagnosis hypothesis holds for (SD, COMPS, OBS).

**IAB condition** *(p.656)*: The ignorance of abnormal behavior condition. If we use `assume-constraint` to model components, then OK assumptions only appear as antecedents to justifications and therefore AB only occurs negatively in antecedents. It is possible to prove that TGDE's models obey the IAB condition. *(p.656)*

**Theorem 17.3** *(p.657)*: (Characterization of minimal diagnoses) $D(\Delta, \text{COMPS} - \Delta)$ is a minimal diagnosis iff $\bigwedge_{c \in \Delta} AB(c)$ is a prime implicant of the set of positive minimal conflicts of (SD, COMPS, OBS).

**TGDE incompleteness** *(p.658)*: One very obvious limitation of TGDE is that it cannot represent all possible models easily. As TGDE obeys IAB it is not possible to model faults as discussed for Figure 17.10 (e.g., short circuits). Generating TGDE to be able to represent such models is difficult and also leads to a violation of the minimal diagnosis hypothesis.

**TGDE limitation on values** *(p.658)*: Another, less obvious, limitation is that the only information that TGDE can represent about a variable is that it is at a particular value. It cannot represent negations or disjunctions of assignments.

**Nogoods and real conflicts** *(p.658)*: All the nogoods TGDE finds correspond to real conflicts. The preceding example shows that minimal nogoods do not necessarily correspond to minimal conflicts.

**CSP as graph** *(p.664)*: A CSP graph can be defined as a set of nodes connected by a set of arcs. In thinking of a problem as a graph, the variables of the problem become the nodes of the graph, and the relationships between the variables become the arcs of the graph.

**Arc consistency enforcement** *(p.666-667)*: Given a constraint network, we can enforce node consistency by restricting the domain of each variable so as to satisfy the unary relations. Arc consistency can be enforced in linear time, making it quite efficient. A network being arc consistent does not even guarantee that a consistent solution exists, unless it produces a solution (i.e., a network is arc consistent then it has no consistent solution).

**k-consistency trade-off** *(p.667)*: Can a consistency-based formulation be generalized to gain completeness? Yes, but at the usual cost. The notion of k-consistency means that for any collection of $k-1$ variables which satisfy all the constraints on them, any kth variable also can be bound such that all k variables are satisfied. If a network contains n variables, then enforcing n-consistency yields a network that can be solved without backtracking. Unfortunately, the cost rises exponentially with increasing k.

**WALTZER update determinism** *(p.669)*: Update procedures are deterministic; given the same values for a constraint's cells, the update procedure always generates the same set of exclude and pick requests.

**Scene analysis: one plunk suffices** *(p.683)*: For the cube scene, symbolic relaxation alone does not provide a complete solution but one additional choice suffices to completely constrain the network. Similarly for the wedge scene. *(p.683)*

## Design Decisions and Rationale

**TGDE uses assume-constraint for OK assumptions** *(p.655-656)*: The AB(c) literal which holds when a component $c \in$ COMPS is abnormal. TGDE adopts the convention of specifying a diagnosis by a set of failing components, leaving implicit that the remaining components are OK. The formalization generalizes this to make it possible for a diagnosis to leave the faultiness of a component unspecified.

**Why TGDE conflicts map to ATMS nogoods** *(p.656)*: TGDE's conflicts correspond one-to-one. For TGDE a conflict is represented as a set of components one of which is abnormal. TGDE obeys the IAB condition so all minimal conflicts are positive. TGDE, through use of the ATMS nogood database, only retains minimal conflicts. The motivation for only representing minimal conflicts is the same as for only representing minimal diagnoses -- there typically are an exponential number of possible conflicts. Therefore, reducing the conflicts to a small set of those possible which is still sufficient to generate the minimal diagnoses is very important.

**Symbolic relaxation vs. other approaches** *(p.663)*: Symbolic relaxation, also known as Waltz filtering, is a commonly used and powerful AI technique. The structure of discrete CSPs can be exploited to supplement very efficient problem solvers, using a variation of the constraint knowledge model introduced in Chapter 15.

**Why constraint networks over search** *(p.667)*: A standard backtracking search to find all consistent combinations would search (in the worst case) $3^6 = 729$ combinations. Using the results of the analysis so far, a backtracking search would only have to examine at most 25 more combinations. And sometimes the constraint analysis suffices to yield a unique solution, and no backtracking search is required at all.

**Procedure model for constraint networks** *(p.668-669)*: Two operations on cells: `exclude` (remove a value from possible values) and `pick` (force a value). One operation on constraints: `update` (run the constraint's update procedure). A constraint is updated whenever the value of one of its cells changes. Constraints terminate in finite time, only use their own cells' values, act through exclude/pick, and are deterministic. *(p.669)*

**Cell queue priority over constraint queue** *(p.669)*: Any `exclude` or `pick` operations suggested by a constraint are queued separately. The queue of cell operations is serviced with higher priority than the queue of constraint updates, since the more tightly restricted cells are, the more information each constraint update is likely to yield.

**Backtracking in WALTZER** *(p.672)*: Often, solving a network requires backtracking. WALTZER supports backtracking by calling on each cell's property `:STACK`. Each entry on `:STACK` consists of a copy of the value and out-reasons for the cell.

**Temporal database design** *(p.687-688)*: A temporal database is a reasoning utility that maintains relationships between intervals. It can be cast as a CSP and thus provide a foundation for temporal databases. Each "triangle" of relationships in the network enforces the consequences of transitivity. WALTZER's enforcement of arc consistency provides a good trade-off of efficiency versus completeness.

**Why not a general-purpose TMS for constraint relaxation** *(p.672)*: As with antecedent constraint languages, a general-purpose TMS would not be appropriate for symbolic relaxation. A constraint network itself plays that role, so the caching provided by the dependency network would be redundant. There is another reason as well: constructing reasonable and efficient justifications for this style of reasoning by exclusion is quite tricky (see Exercise 3).

**Underdetermined vs. overconstrained networks** *(p.670)*: A CSP is solved when the network is determined. Each set of values which determines the network is a solution to the problem. Sometimes an entire task can be cast as a CSP, but such circumstances are relatively rare. Typically it makes more sense to decompose a complex task into subproblems, some of which can naturally be formulated as CSPs. Symbolic relaxation provides a filter that can significantly speed up additional searches.

## Implementation Notes

**WALTZER cell operations** *(p.672-673)*:
- `exclude` rules out a value as a possibility for a cell, citing the constraint (informed) as the source of this decision. Simultaneously, `pick` forces the remaining value to be the only possibility for the cell. If providing a symbol as the informant (e.g., the default `:USER`), this allows the same procedures to be used as an interface to external systems.
- When excluding, it first checks to see if the value is still believed as a possibility, by fetching its entry in the cell's out-reasons and testing whether `:IN` is found as its state. If so, it starts the actual exclusion process by first updating the cell's `out-reasons` entry, setting it to be the pair of the informant and network time stamp.
- If the last possible value was excluded, the cell is determined. *(p.673)*

**Network status states** *(p.671)*: `:NEW` (just created), `:QUIESCENT` (all queues empty, stable), `:IN-PROGRESS` (currently propagating), `:OVERCONSTRAINED` (contradiction detected).

**Constraint update procedure structure** *(p.672)*: No restrictions on the format of the `parts` field of the constraint, so that domain-specific constructors can organize it as desired. The `update-procedure` field holds the domain-specific procedure for accomplishing that constraint's update.

**WALTZER search** *(p.675-676)*: `search-network` ends with a simple depth-first search routine. The local procedure `search-thru-plausible-cells` does the actual work. It is called with a list of undetermined cells (found via the call to `ep-plunk`) and a string for printing the debugging flag.

**Scene labeling implementation** *(p.680-681)*: The `Scene` macro creates a WALTZER constraint network. The `Line` macro creates a cell for a line in the scene. The `Junction` macro creates a cell for a junction. Constraints are built depending on the specifics of the domain, via `add-constraint-cell`.

**Temporal database files** *(p.689)*:
- `timedb.lisp`: Database definitions and procedures
- `allen.lisp`: Allen's transitivity table definition
- `ttest.lisp`: Test cases

**Temporal constraint propagation** *(p.690)*: `find-transitive-relations` determines the cell's name to get the intervals of the newly created relationship, iterates over each relationship involving the first interval to find candidate third intervals, filters candidates to ensure they relate to the second interval, and dispatches based on an internal order.
