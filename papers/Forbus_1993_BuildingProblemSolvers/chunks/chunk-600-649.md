## Key Definitions

- **Cell in ATCON**: A cell is a location which holds a value. Writing `= 1` to indicate that cell x has the value 1. Using a TMS, each possible assignment of a value to a cell is represented as a TMS node. In ATCON, `x = 1` refers to a particular node representing the fact that cell x has the value 1. While each TCON cell could contain only one value at a time, when using a TMS, typically there will be multiple nodes associated with any particular cell. *(p.606)*

- **Assumption in ATCON**: The syntax of ATCON constraints is almost identical to that of TCON. As ATCON includes a TMS, each conclusion of a constraint will now also have a justification. Each rule of a formula is of the form `(Seti) (Usei) (RuleBody)`. `(Seti)` is a cell for which the rule can provide a value. `(Usei)` is a list of cells whose values must be known before the rule is executed. `(RuleBody)` is a fragment of Common Lisp code which is executed and whose result is the value for `(Seti)` unless this value is `:DISMISS` or `:DISMISS188`. *(p.606-607)*

- **ATCON Assumptions**: ATCON has a new part type: `assume-part`. Every time a new instance of a constraint is created, a new assumption is created for each one of its `assume-part`s (other parts need not be any of course). For example, in the diagnostic task, one makes the default assumption that components are working correctly unless there is evidence to the contrary. *(p.607)*

- **`assume-constraint`**: Creates an assumption for each instance of `adder` and `multiplier` which is implicitly included in all the `(Use)` of the formulae. The `assume-constraint` creates an ATMS assumption. *(p.607)*

- **Cell value `domain`**: A list of the allowable values a cell can have. The ATMS `tms-node` defstruct has fields: `datum`, `cell` (pointer back to the cell containing the node for which this value is), `processed` (a list of constraints which have looked at this value and processed it; used to prevent any rule from being executed more than once). *(p.625)*

- **Constraint definition fields**: `name` (local, unqualified name), `atcon` (the ATCON instance it is part of), `owner` (the instance of the constraint that first created this component), `parts` (an alist describing parts which comprise the constraint). *(p.626)*

- **Constraint prototype fields**: `name` (global), `parts` (the list of parts directly specified by the constraint definition), `creation-form` (either an interpreted or compiled Common Lisp procedure invoked when an instance is created), `cells` (an alist of all the parts that can hold values -- cells and assumptions). *(p.626)*

- **Diagnosis**: A diagnosis is a minimal set of assumptions which intersects every conflict. Conflicts can be viewed from a complementary point of view: consider the maximal sets of assumptions which do not intersect any conflict. These maximal sets are the maximal sets of components which can all be OK but do not support any prediction that conflicts with an observation. Such maximal sets clearly are in one-to-one correspondence with the diagnoses. *(p.645)*

- **Minimal diagnosis**: The minimal diagnoses are those which contain no other diagnoses as subsets. Thus the minimal diagnoses occur immediately above all those eliminated by the conflicts. *(p.644-645)*

- **Minimal cardinality diagnosis**: Diagnoses with the fewest faulted components. *(p.646)*

## Algorithms and Pseudocode

- **`create-atcon`**: Creates a datastructure which returns the entire state of the ATCON. Takes keyword arguments: `nearly-equal?` (procedure for determining if two values are approximately equal), `debugging` (flag for tracing), `delay` (defaults to 1; if nil, ATCON does not try to prevent execution of rules that would produce useless results). *(p.621)*

- **`set-parameter`**: Permanently sets cell 1 to expression by providing it an ATMS justification with an antecedent, i.e., it creates a premise. Thus, cell 1 will be expression in all possible environments. *(p.609)*

- **`assume-parameter`**: Makes an ATMS assumption for the purpose. The optional `string` argument provides a pretty way to print out the ATMS assumption when it appears in environments. *(p.609)*

- **ATCON rule execution procedure**: The `run-rules` procedure controls the execution of ATCON rules. ATCON uses the ATMS-provided interface for scheduling execution. The `tms-node-rules` of every mode contains a queue of ATCON rule objects which are to be executed such that their value records a non-empty label. ATCON installs `constraint-node-to` on the ATMS to specify it as the procedure to be called when a node is added. Since `cons-node-to` is called within ATMS operations, it cannot perform any ATMS operations recursively. Hence, ATCON maintains a queue of rules pending execution and runs these rules after the ATMS returns. *(p.630-631)*

- **Two execution modes**: Normal mode (when `atcon-delay` is 1) prevents all unnecessary rule executions. When rule execution is cheap, the overhead of preventing all unnecessary rule executions is not worth the savings. In normal operation, `fire-constraint` performs an additional preliminary check to determine whether the rule object is executable. *(p.631)*

- **`fire-constraint` procedure**: Fires conventional rule objects constructed by `add-rule`. In this case, `rule-queue` is a pair consisting of the set of rules which use the cell and the constraint instance itself. `fire-constraint4` is designed to prevent duplicate rule executions (as opposed to the two procedures just discussed, which prevent the first execution of a rule when the result is known to be without value). *(p.632)*

- **`has-external-support` and `has-complete-external-support`**: Procedures that help prevent the execution of unnecessary rules. `has-external-support` checks ATMS primitive `in-antecedent?` to identify those justifications which support the node, and if any of those justifications derive from outside of the current constraint, then the rule object is considered executable. *(p.631-632)*

- **Sequential diagnosis algorithm (TGDE)**: TGDE continues until one minimal cardinality diagnosis remains after taking into consideration the new observation. At each step, chooses the best next measurement to discriminate among remaining diagnoses. *(p.645-647)*

- **Measurement scoring**: A measurement is scored by: $\sum_i c_i \log c_i$ where $c_i$ is the number of diagnoses which predict the $i$th outcome. As $k$ and $w$ are constants, the best measurement is the one that maximizes $\sum_i c_i \log c_i$. *(p.649)*

## Data Structures

- **ATCON datastructure fields**: `title` (ignored by ATCON but useful for debugging), `cells` (list of all cells in the ATCON network), `queue` (list of all rule-mode pairs waiting to be run by ATCON), `constraints` (list of all constraints in this ATCON), `user-parts` (a hash table of all the parts the user has created directly), `atms` (the ATMS instance this ATCON is associated with), `nearly-equal?` (procedure to call to test whether two values are equal), `disjunction` (list of disjunctions supplied by the user), `debugging` (flag), `delay` (flag controlling rule execution), `executions` (counter). *(p.624)*

- **Cell defstruct fields**: `atcon` (the ATCON instance), `name`, `owner` (the instance of the constraint that first created this cell), `nodes` (the list of nodes associated with this cell), `users` (describes all the rules that use this cell), `domain` (list of allowable values a cell can have). *(p.624-625)*

- **TMS node datum defstruct (`tms-node-datum`)**: Has fields: `datum` (the datum), `cell` (pointer back to cell), `processed` (list of constraints which have looked at this value). *(p.625)*

- **Value defstruct (within ATMS)**: `atcon`, `name`, `cell`, `processed`, `string` (used only to print cell values). *(p.625)*

- **Constraint defstruct fields**: `atcon`, `name`, `owner`, `parts`, `prototype`. *(p.626)*

- **Constraint prototype**: `name` (global), `parts`, `creation-form`, `cells` (alist of all parts that can hold values). *(p.626)*

- **Diagnosis lattice**: The set of all diagnoses can be arranged as a lattice. At the bottom is the empty set (nothing is faulted). At the top is the diagnosis where all components are faulted. Every edge represents a link between a diagnosis and its immediate superset or subset. In order for a diagnosis to explain every symptom, the diagnosis must contain at least one component from every conflict. *(p.643)*

## Equations

- **Probability of diagnosis D**: $p(D) = \epsilon^{|D|} \cdot (1 - \epsilon)^{n-|D|}$ where $n$ is the number of components in the device and $|D|$ is the number of faulted components in $D$. Assuming $\epsilon$ is very small: $p \propto \epsilon^{|D|}$ and only the smallest cardinality diagnoses are of interest. *(p.646)*

- **Measurement scoring function**: The best measurement maximizes $\sum_i c_i \log c_i$ where $c_i$ is the number of diagnoses which predict the $i$th outcome. *(p.649)*

## Key Claims and Properties

- **TCON vs ATCON performance tradeoff**: The basic advantage of ATCON is that all of TCON's duplicate rule executions are avoided. On the other side of the balance is the additional bookkeeping cost. Which wins depends on the task. If execution is cheap, and there are few context switches (i.e., few duplicate rule executions), TCON performs better. If rule executions are expensive or there are a large number of context switches, ATCON will be better. However, seeing this tradeoff purely in terms of efficiency misses much of the picture. As context switching is truly free in ATCON because it becomes an unnecessary operation, problem-solving tasks that would otherwise require a vast number of context switches become more tractable. *(p.620)*

- **TCON's wasted effort**: TCON connected to no TMS gets into trouble by not caching results. Consider: if A1 changes, TCON must redo additions and set SUM, and if A2 subsequently changes, the same additions must be recomputed. If TCON had been hooked up to a TMS which cached all results, the same addition need not be recomputed. *(p.618)*

- **ATCON with ATMS vs single-context TMS**: If using a single-context TMS, every duplicate rule execution is replaced by a JTMS label propagation. In ATCON, even this false propagation is avoided. If A1=1 holds in environment {A}, and A2=1 holds in environment {B}, then SUM=2 holds under environment {A,B}. To find the value of a cell in a context, one just looks at that node associated with the cell which has a label environment that is a subset of the context's environment. *(p.619)*

- **ATCON does not need context switching**: In ATCON there is no necessity to change values of cells. To find the value of a cell in a context, one just looks at the node associated with the cell which has a label environment that is a subset of the context's environment. There will always be at most one such node. *(p.619)*

- **ATCON is not a stand-alone problem-solving system**: At a minimum, a user must make top-level Common Lisp calls to start an ATCON task, and for most tasks the user will write some additional Common Lisp procedures which function in tandem with ATCON. *(p.620)*

- **TGDE uses many-worlds strategy**: TGDE works on all hypotheses simultaneously, using the ATMS to explicitly share results among them. The designer does not need to pay attention to contexts, context-switching, backtracking, or contradiction handling. *(p.635)*

- **TGDE uses ATMS interpretation constructor**: TGDE does not require a distinct algorithm to construct diagnoses -- this work is done by the ATMS interpretation construction algorithm, when directed to interpret each OK assumption as a default. *(p.645)*

- **Every observation can potentially shift the probabilities of diagnoses**: Under the simplifying assumptions that all components fail independently with probability epsilon, only the smallest cardinality diagnoses are of interest. The diagnostic process should continue until a single smallest cardinality diagnosis remains. *(p.646)*

- **Constraint programming style principle**: The basic conceptual style principle is that the Common Lisp code in `(RuleBody)` should be such that the results of an ATCON program are insensitive to the particular order in which the rules are executed. Syntactic conventions which help ensure this: `(RuleBody)`s should not set the values of cells; `(RuleBody)`s should not access cells other than those in `(Use)`s; `(RuleBody)`s should not perform any side effects on global datastructures or extern state in any way. *(p.609)*

## Design Decisions and Rationale

- **Why ATCON over TCON**: Implementing ATCON requires extending TCON in two directions simultaneously. First, TCON must be adapted to work with a separate TMS module. Second, TCON, which takes a single context perspective, must be extended to operate in multiple contexts. Therefore, to successfully implement this desired functionality requires reexamining many of the issues discussed in earlier chapters -- this time in the context of a constraint language. *(p.605)*

- **ATCON cell-type system**: ATCON allows constraints to specify the values a cell can have. For example, the adder can only add quantities between 0 and 1 (i.e., of two bits), this can be specified by enumerating the possible values to a cell-part type. If a value is produced for a1, a2 or sum which is not between 0 and 1, a contradiction is detected and an ATMS nogood will be constructed. *(p.611)*

- **Disjunction in ATCON**: The ATMS allows an ATCON application to express very simple disjunctions. The ATCON function `disjunction` is provided a set of nodes, and the interpretation constructor ensures that every solution contains at least one node from each disjunction. *(p.613)*

- **`assume-constraint` design**: Every instance of an `assume-constraint` has an ATMS assumption created such that whenever any variable is set this assumption will be included as an antecedent. This allows expressing assumptions about component behavior (e.g., that a component is functioning correctly). *(p.607-608)*

- **Rule execution scheduling**: Creating an instance of a constraint will cause the creation of its subparts. Whenever a new cell is created, datastructures must be created to ensure that when this cell receives a value the rules which depend on that value are scheduled for execution. The code does not treat every individual rule of a constraint disjointly. Instead, if any of the rules of a constraint depend on a given cell, then ATCON creates a rule object that includes all the rules which use this cell and the constraint itself. This rule object is then pushed on the `tms-node-rules` of the node. *(p.630)*

- **Why TGDE uses ATCON instead of TCON**: TGDE was implemented using ATCON rather than TCON (as in Section 15.4.1) because using ATCON avoids the need for contexts, context-switching, backtracking, or contradiction handling. Each component is described by an ATCON constraint. Each component device also has an `assume-constraint` creating an assumption that it is working correctly. *(p.638)*

- **Sequential diagnosis measurement selection**: TGDE goes to considerable effort to plan the best measurement to make next. Depending on the method used to select measurement points, there can be dramatic differences in the average number of measurements required to isolate a fault. A very long chain of one thousand buffers where one of the buffers is faulted: measuring in the middle versus sequentially gives exponential improvement. *(p.647)*

## Implementation Notes

- **ATCON uses `*atcon*` as special variable**: The variable `*atcon*` is declared as a special variable. The macro `with-network` is the recommended way to bind `*atcon*`. All the constraint definitions (i.e., prototypes) are stored in a separate hash table `*prototypes*`. The same prototype definitions are then used for every ATCON instance. *(p.627)*

- **`create-atcon` creates instance with `debug-atcon` set from `debugging` parameter**: The default procedure to override: `nearly-equal?` -- the equality checker of the current ATCON determines whether two values should be considered identical. The default procedure is `overly-simplistic` and presumes that two values are identical if they are `equal` or within numeric tolerance. *(p.627)*

- **Constraint creation procedure `create`**: It immediately calls `create1`, which is the main procedure for creating constraint instances. `create1` creates constraints. There are only two primitive constraint types: `cell` and `assume-cell`. All other constraints are treated identically. *(p.628)*

- **`set-value` main procedure for setting cell values**: It first invokes `the procedure lookup-node` to find a node of the cell having that value. Then it calls the ATMS procedure `justify-node` to set the cell to that value under the appropriate environment. The procedure `lookup-node` works from the final indicator and works from back to front, repeatedly looking for a subpart with the given name. *(p.629-630)*

- **ATCON delay flag**: Controls whether `fire-constraint` performs preliminary checks before executing rules. When delay is nil, ATCON executes rules immediately without checking. When delay is t (the default), it checks for external support before executing. *(p.631)*

- **`fire-constraint4`**: Designed to prevent duplicate rule executions. It handles two cases: (1) rule objects produced by `add-rule`, and (2) rule objects created with the ATCON rule execution strategy (those nodes where `domain` is demonstrated as containing the value of `<empty>-datum`). *(p.632)*

- **TGDE implementation**: Each component is described by an ATCON constraint. For adders: `(assume-constraint adder ...)` with formulae for sum computation. For multipliers: `(assume-constraint multiplier ...)` with product formulae. The polybox is described by a `(constraint poly ...)` connecting all components. *(p.638-639)*

- **TGDE uses ATMS label database**: After creating the polybox and setting input assumptions, TGDE examines the ATMS labels. The labels indicate all the patterns of inputs that force the high-order output bit. Examining labels reveals which environments (sets of assumptions) support which values. *(p.616-617)*

- **TGDE conflict detection**: When F=10 is observed and the assumption that all components work correctly produces F=12, TGDE identifies conflicts. The two conflicts in the polybox example are {A1,M1,M2} and {A1,A2,M1,M3}. *(p.641-642)*

- **Minimal diagnoses via ATMS interpretation constructor**: TGDE uses the ATMS interpretation constructor to find minimal diagnoses. The ATMS already has an algorithm that constructs maximal consistent sets (interpretations). These maximal sets are in one-to-one correspondence with minimal diagnoses. *(p.645)*

> "The basic advantage of ATCON is that all of TCON's duplicate rule executions are avoided. On the other side of the balance is the additional bookkeeping cost. Which wins depends on the task." *(p.620)*

> "As context switching is truly free in ATCON because it becomes an unnecessary operation, problem-solving tasks that would otherwise require a vast number of context switches become more tractable." *(p.620)*

> "ATCON is not a stand-alone problem-solving system. At a minimum, a user must make top-level Common Lisp calls to start an ATCON task, and for most tasks the user will write some additional Common Lisp procedures which function in tandem with ATCON." *(p.620)*

> "TGDE is an example of the many-worlds strategy (Chapter 14) for building problem solvers. Although one can characterize what TGDE does in terms of the contexts representing hypotheses which explain the symptoms, the internal operations of TGDE pay almost no attention to them. Instead, TGDE works on all hypotheses simultaneously, using the ATMS to explicitly share results among them." *(p.635)*

> "Diagnosis is difficult because we almost never have direct evidence of the malfunctioning of any component. Components usually do not look faulted, and it typically is too expensive to remove all the components and check them individually. Instead, we only have indirect evidence from their behavior." *(p.635)*

> "In this kind of problem-solver design, once the designer has chosen the assumptions to be manipulated, no attention is to be paid to contexts, context-switching, backtracking, or contradiction handling. This leaves the designer free to focus on the issues that are specific to the task at hand." *(p.635)*

> "ATCON is designed as an extensible language. The most general definition of a constraint is: (constraint (Name) (Parts) (Body)). The (Body) contents simply of a list of forms. These forms are evaluated by a very simple interpreter." *(p.620)*
