## Key Definitions

- **Constraint**: A relationship between data declarations expressing that computations should occur automatically, without special effort from the user. Consider the equation z = x + y. The constraint metaphor says bookkeeping occurs internally. *(p.552)*

- **Cell**: The principal feature of a constraint; a cell has a value. A cell also records several other pieces of information, including its relationships to the rest of the constraint network and how its value was supplied. All connections between constraints are made via cells. *(p.552)*

- **Terminal**: From the perspective of a constraint, the cells it is connected to are its terminals. Each terminal has a name, corresponding to the particular role it plays in the constraint. *(p.553)*

- **Constraint prototype**: A definition of the parts and rules for a type of constraint. The format is `(constraint (name) (parts) (body))` where `(name)` is the class of constraint, `(parts)` is a list of parts each instance must have, and `(body)` specifies rules and relationships. *(p.559)*

- **Part**: Each entry in `(parts)` takes the form `((PartName) (PartType))` where `(PartName)` is the name by which the part is referred to within `(body)`, and `(PartType)` is the kind of thing the part is -- either a cell or another kind of constraint. *(p.559)*

- **Formula**: A rule within a constraint consisting of `((sets) (uses) (RuleBody))` where `(sets)` is the cell for which the rule provides a value, `(uses)` are cells whose values must be known before the rule is executed, and `(RuleBody)` is the procedure which is executed. *(p.560)*

- **Informant**: For a cell, either a rule, symbol of ground support, or nil if the cell is unknown. *(p.570)*

- **Contradiction handler**: Default is `default-coincidence-handler`. User-supplied contradiction processor is `contradiction-handler`. *(p.569)*

- **Equality link (==)**: A special constraint that propagates value information between two cells. When a value is derived for one side, the other is set to the same thing. Equality links can be broken by calling `agm-apew`, retracting all values derived using the equality link. *(p.564)*

- **Indirect cell**: A new class of cell where the value, if known, is always another constraint or cell. Indirect cells can participate in formulae statements just like any other. They allow building a constraint network that is completely distinct from the art of reasoning within a constraint network. *(p.591-592)*

- **Wiring rules**: A new kind of rule which specifies connections between the value of an indirect cell and the constraint of which it is a part. For instance, to use action constraints for simulation, wiring rules must specify that position and velocity of the ball should be equated with the initial position and velocity in the action description. *(p.592)*

- **Plunk**: A symbol given to a variable to allow analysis to proceed. When a cell gets multiple symbolic values, it must be the case that each expression can be equated. *(p.596)*

- **Constraint suspension**: A diagnostic technique where the correct behavior of a system can be modeled using constraints, and diagnosis is performed by suspending constraints to find which component, if suspended, allows predicted values to match observed values. *(p.583)*

## Algorithms and Pseudocode

### Constraint Propagation Model *(p.553)*
Information can flow in any direction that makes sense. If we know x = 2 and y = 3, then the constraint should figure that z = 5. And if y = 3 and z = 6, then the constraint should figure that x = 3.

### Adder Constraint Rules *(p.560)*
The adder constraint has cells a1, a2, and sum. Its rules fire antecedently -- whenever values for the prerequisites are known:
- If a1 and a2 are known, compute sum by adding
- If sum and a1 are known, compute a2 by subtracting a1 from sum
- If sum and a2 are known, compute a1 by subtracting a2 from sum

### Multiplier Constraint Definition *(p.562)*
The multiplier constraint has special-case rules for handling zeros: if one input is zero, the product must be zero regardless of whether the other input is known. Also handles the case where the product is zero -- uses `LDERP!` to signal contradictions if both inputs are non-zero but the product is zero. Five rules total handle all cases.

### TCON Interface Procedures *(pp.566-568)*
- `(create-tcon title &key selftype)` -- Builds a new constraint network
- `(load-prototype file-name &optional (tcon *tcon*))` -- Loads constraint definitions
- `(change-tcon tcon &key (debugging nil))` -- Switch to a different network
- `(create name type &optional (tcon *tcon*))` -- Build a cell of given type
- `(>> &rest args)` -- Refer to a constraint part by name path
- `(with-network tcon &rest forms)` -- Evaluate forms in context of tcon
- `(use-network tcon)` -- Make tcon the current constraint network
- `known?` -- Returns non-nil if the cell has a known value
- `what-is` -- Prints the value of a cell
- `constraint-values` -- Prints values of all cells in a constraint (recursive)
- `show-network` -- Prints the value of every cell in the network
- `why` -- Describes the informant for a value (like a stripped-down JTMS)
- `premises` -- Shows list of ground values (similar to `assumptions-of-node`)
- `needs` -- Describes what other cells might be given values to determine value of a cell
- `set-parameter` -- Sets a cell to a given value
- `forget-parameter` -- Retracts an assumed value
- `change-parameter` -- Changes the assumed value for a cell
- `constant` -- Provides a fixed value for a cell
- `tcon-answer` -- Tells the contradiction handler which assumption to retract
- `cell-ground` -- Lists the cells providing ground support to the given cell

### Process-constraint-rules Procedure *(p.572)*
The procedure `process-constraint-rules` analyzes the contents of a formulae statement: it builds up two alists, one describing rules and the other describing the cells referred to by the rules. These alists supply the dependency information that links rules and cells. The `sets` and `uses` of each rule are cached on the rules alist, and inverted to find the Users and Suppliers for each cell.

### Constraint Propagation via set! *(p.574)*
`set!` is the internal procedure that does the work involved in giving a value to a cell, including consistency checking and propagating results. Steps:
1. Check if the cell is already known -- if the new value is not consistent with the previous value, the `coincidence?` handler returns and contradiction handling is invoked
2. Otherwise, the cell value and informant fields are updated to reflect the new value and its source
3. The source argument will always be either a pair indicating a specific rule or a symbol indicating ground support
4. Finally, `spread-results` propagates effects of the new value by queuing for possible execution the rules which use the newly set cell

### Candidate Generation for Diagnosis *(pp.586-587)*
1. Identify discrepancies by comparing observed outputs with predicted outputs derived via constraint propagation from observed inputs
2. Compute the initial candidate set by intersecting the constraints involved as the explanation underlying each discrepancy
3. Evaluate each candidate by suspending its corresponding constraint, and use constraint propagation to derive potential new observations from the observed output values

### Diagnosis Example *(pp.587-589)*
For the polybox circuit example with 5 components (MULT-1, MULT-2, MULT-3, ADD-1, ADD-2):
- `generate-candidates` runs on the combination of measurements and initial values
- Discrepancies are found by comparing predicted vs observed outputs
- Candidates are sets of constraints whose suspension makes predictions consistent
- Example: 2 discrepancies found with candidates {MULT-2}, {MULT-1, ADD-1} -- suspending MULT-2 eliminates both discrepancies
- When multiple faults are present, the intersection of candidates from all outputs yields the correct diagnosis, but constraint suspension assumes single faults

## Data Structures

### Cell Structure *(p.570)*
- name: A symbol
- tcon: The network it belongs to
- owner: The constraint it is part of. If global, this is :USER
- value: The current value
- informant: Either a rule, symbol of ground support, or nil if the cell is unknown
- roles: The parts the cell plays in various constraints. Used to find its users and suppliers
- place: Usual place to store extra information

### TCON Network Structure *(p.569)*
- title: String for printing
- prototypes: List of prototypes associated with this network
- cells: List of cells in the network
- constraints: List of constraints in the network
- bag-queue: Queue of cells to re-examine for alternate support
- bdq-queue: Queue of rules ready for execution
- coincidence-handler: Identity procedure for cell values
- contradiction-handler: User-supplied contradiction processor
- debugging: Controls printing of information about internal operations

### Prototype Structure *(p.570)*
- name: A symbol
- tcon: Network it is part of
- parts: List of parts
- creation-forms: Evaluated for each constraint of this type when created

### Constraint Structure *(p.570)*
- name: A symbol
- tcon: Network it is part of
- owner: The constraint it belongs to. If global, this is :USER
- parts: An alist of names and associated parts for this instance
- prototype: Where to get information about rules and cells

### Rules for Cells *(p.561)*
For cells:
- sets: The cell the rule provides a value for
- uses: The set of cells which use the cell as an input
- Suppliers: The set of rules which can provide a value for the cell

Guidelines for rules within constraint body:
1. A rule body should only rely on the values of its uses cells
2. All inferential work of a rule should occur via the result it returns
3. A rule must refuse to return a value when only a subset of information from its uses cells suffices to obtain that value
4. A rule must detect inconsistent data and signal the interpreter when it is found

## Equations

### One-dimensional uniform motion *(p.564)*
xend = xstart + velocity * deltat

### 1D motion under uniform acceleration *(p.579)*
S_f = S_i + V_i * deltaT + (1/2) * a * deltaT^2
V_f = V_i + a * deltaT

where deltaT is the interval over which the motion occurs.

### 2D motion -- polar/rectangular conversion *(p.580)*
X = R * cos(phi); Y = R * sin(phi)
R = sqrt(X^2 + Y^2); phi = atan2(Y, X)

### Satellite interception condition *(p.578)*
For interception to succeed: X_f^s - X_f^i = Y_f^s - Y_f^i = 0

### Energy conservation constraint *(p.593)*
No matter how the details of the system are transformed between its state as described by before and after, the energy must be the same. Expressed as equality between (>> energy before) and (>> energy after).

## Key Claims and Properties

- **Constraint networks differ from logic circuits**: In constraints, information can flow in any direction that makes sense, whereas in logic circuits information only flows from inputs to outputs. *(p.553)*

- **Advantage over pattern-directed systems**: The TCON constraint data model needs only local references. No pattern matching is required, thus avoiding overhead in both the database and the reference mechanism for procedures. Efficient data structures can be built which wire the connections between a cell and the procedures that use it. Rules associated with a constraint are indexed by instances of that constraint. *(p.557)*

- **PDS version would be less efficient**: A PDS implementation would ignore several strengths of the constraint network model. The PDS overhead of building new rule instantiations for each combination of triggering data is avoided. The principal relationship (that a cell having a value implies certain relationships in the datastructure representing the constraint network) would require explicit assertion, meaning cells are not primitive data items. *(p.557)*

- **Constraint languages are efficient for large systems**: These efficiencies can be crucial for building large systems. A typical VLSI design tool might consist of hundreds of thousands of logic elements, implemented via millions of transistors. The local nature of constraint domains (parts that are physically adjacent are the only parts that can interact) makes this a natural use for the antecedent constraint model. *(p.558)*

- **Constraint networks are not metacircular**: The art of building a constraint network is completely distinct from the art of reasoning within a constraint network. This limitation is also a source of strength -- limiting inference to be within the network makes it cheap, and thus simple antecedent control strategies suffice. *(p.591)*

- **Sharing structure savings**: Instead of creating two separate cells, we build a single cell that represents both. Savings in terms of cells may not seem very large, but if shared parts are themselves complex constraints, savings can be considerable. Sharing can reduce storage requirements of a constraint network by up to 50 percent. *(p.590-591)*

- **Single-fault and non-intermittence assumptions**: The two common assumptions in constraint suspension diagnosis are: (1) at most one component of the system has failed; (2) if a component is broken, it remains broken in the same manner while reasoning proceeds. *(p.584)*

- **Constraint suspension limitations**: Constraint suspension has its limitations. There is no single component failure that could explain certain problems, and indeed the constraint suspension algorithm cannot find an explanation in such cases. A more powerful technique (GDE, described in Chapter 17) is needed. *(p.589)*

- **Symbolic propagation (plunking)**: The technique of introducing extra rules to break loops is a good one when there is some standard locality in which to put such information. If the domain of the cells is a discrete set of values, dependency-directed search can be used to find possible solutions. If the cells are real numbers, non-local mathematical techniques can be applied. *(p.595-596)*

- **Historical cells for dynamic systems**: Historical cells have become the preferred technique for temporal reasoning. The technique was first used in de Kleer's PhD work where maintaining consistent sets of ATMS-like assumptions. *(p.598)*

- **Symbolic algebra via plunking**: The second technique is to use the symbolic pieces of the constraint network to drive the formulation of equations which can then be solved via algebraic manipulation. This is essentially what was done by hand in defining the 1D-motion constraint. *(p.596)*

## Design Decisions and Rationale

- **Why constraint languages over rules**: Even though pattern-directed rules could achieve the same external behaviors, constraints are more efficient: the has-value relationship is internal, not requiring assertion. Constraint definitions need not refer to a network explicitly. The rule system is more tightly coupled than in PDS. *(p.556-557)*

- **Why formulae are interpreted antecedently**: Whenever the values for the prerequisites (uses) are known, the rule is queued and eventually executed; otherwise no particular execution order is asserted between rules. A rule may be used by multiple formulae, but to ensure well-founded support, no (sets) of a rule can include the cell which it sets. *(p.560)*

- **Multiplier special cases for zero**: Enforcing multiplication requires five rules because of special cases where certain values provide more or less information than normal. When one input is zero, the product must be zero regardless. When the product is zero and both inputs are non-zero, there is a contradiction (LDERP!). *(p.562)*

- **Why the fourth guideline (detect contradictions)**: The last two rules in the multiplier constraint illustrate the fourth guideline. When appropriate, they use the value of the product and one input to compute a value for the other input. If the known input is zero, these rules detect contradictions (LDERP!). *(p.563)*

- **Why not to share structure always**: Sharing structure does not always make sense. In defining a constraint to model a specific system, the ability to undo connections is crucial. For instance, a workstation-to-LAN connection represents a physical connection that may need to be broken if the machine is moved. *(p.590)*

- **Historical cells vs network replication for dynamic systems**: Network replication (copying the entire constraint network N for each time point) has the virtue of simplicity but seems wasteful since the same relationships govern at both times. Historical cells with a discrete temporal ontology provide a more principled approach. *(p.597-598)*

- **Constraint language as abstraction not metainterpreter**: The constraint data model provides a natural use for antecedent inference because the domain data (cells, values) and inference structure (rules, constraints) are tightly integrated. This is unlike rule-based systems where pattern matching adds overhead. *(p.556-558)*

## Implementation Notes

- **TCON is based on CONLAN [9]**, which extended the language described in [10]. Carrying out the extensions suggested in Exercises 9, 10, and 11 will yield a modern implementation of CONLAN [5]. *(p.598)*

- **The file tcon.lisp** contains a complete implementation of TCON. *(p.566)*

- **The Constraint macro** provides syntactic sugar for defining prototypes. `*tcon*` is dynamically bound so that constraint definitions need not refer to a network explicitly. *(p.571)*

- **The >> operator**: `>>` resolves chains of references. An implementation of `>>` must allow for the fact that a chain of references can be arbitrarily long. The simplest way is to define `>>` as a macro that alters its argument to simplify subsequent processing. Iteration is carried out by `nested-lookup`, which uses `lookup-global` to get the base of the chain and `lookup-part` to find appropriate parts from a constraint list of parts. *(p.572)*

- **process-constraint-body**: Analyzes the body of a constraint definition. Formulae statements are passed to `process-constraint-rules`. Within the body of a constraint definition, everything other than Formulae statements are treated as forms which should be executed whenever an instance of that type of constraint is built. *(p.571-572)*

- **Values provided via constant** have the same status as premises in truth maintenance systems. They cannot be retracted without violating the interface conventions. As with the JTMS, the macro `with-contradiction-handler` rebinds a network contradiction handler during a computation. *(p.569)*

- **Each procedure performs the appropriate local change and then propagates the effect of this change through the rest of the network.** When called on a cell whose value was assumed, `change-parameter` retracts its value and does not attempt to find alternate support before assuming a new value for the cell. *(p.569)*

- **The dependency system** is basically a stripped-down JTMS. `why` provides the informant for a value. `premises` shows the list of ground values (similar to `assumptions-of-node`). `needs` allows examining information that is difficult to get in a PDS because the dependency system is more tightly coupled to the rule system. *(p.568)*

- **Symbolic propagation strategy**: Instead of placing a numerical guess in the network, choose some variable to receive a symbolic value (a plunk), which is then propagated through the system. The act of giving a variable a symbolic value to allow the analysis to proceed is called plunking the variable, and the symbol given is called a plunk. *(p.596)*

- **Constraint suspension for diagnosis** (Section 15.4.3): The file examp.lisp contains an implementation of this algorithm. The procedure `generate-candidates` organizes the computation. The first step finds discrepancies via `setup-inputs` and `find-discrepancies`. The second step uses `find-contributing-constraints`, adapted from `rule-ground` in tcon.lisp. The third step is `evaluate-candidates`. *(p.587)*

- **Removal rules** (Section 15.5.1.4): Interfacing constraint networks to other inferential facilities often requires using side effects. For instance, in building a VLSI design system, once the coordinates of a rectangle are known, a rule associated with the constraint can call a graphics system to display it. But if a coordinate is retracted, we want the rectangle to be erased. Such cleaning up can be handled by rules triggered when a cell value is removed. *(p.593)*
