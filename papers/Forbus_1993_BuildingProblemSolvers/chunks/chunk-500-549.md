## Key Definitions

- **CLTMS data structure fields** *(p.501)*: The LTMS `datastructure` contains a `clauses` field (all clauses in the database), `queue` (pending clauses to resolve), `conses` (cons cells for temporary use in consensus), `delay-sat` (flag: if true, satisfied clause should never be resolved), `cons-size` (number of conses available). Clause `datastructure` has field `clause-status` with four possible non-nil values: `:SUBSUMED` (subsumed by another clause), `:QUEUED` (part of database, queued for pending consensus), `:DIRTY` (BCP was blocked because clause is satisfied, must be re-queued), `:NOT-INDEXED` (just created, not yet inserted into connection graph).

- **Many-worlds strategy** *(p.512-513)*: Work in all consistent contexts at once; seek all possible solutions. The ATMS provides the medium for tracking multiple contexts simultaneously.

- **Focused strategy** *(p.512-513)*: Work in a single context (or small number of contexts) at a time to find a good solution. Switch contexts opportunistically. The focus environment is typically called the *focus environment* or *focus*.

- **Causally primitive predicates** *(p.522)*: Statements that are directly affected by some action. In ATMS-based world representation, a state is an environment of causally primitive statements.

- **Defining form** (of an operator) *(p.530)*: Lists the name of the operator and variables to be defined for each instance. E.g., `(Pickup ?x)`.

- **Preconditions** (of an operator) *(p.530)*: A list of statements that must hold in a situation for the operator to be applicable. E.g., for Pickup: `(on ?x Table)`, `(clear ?x)`, `(hand-empty)`.

- **Delete-list** (of an operator) *(p.530)*: Statements believed in the situation before the action that will no longer be true afterward.

- **Add-list** (of an operator) *(p.530)*: Statements not believed before the action but true afterward.

- **STRIPS representation** *(p.529-530)*: Actions are viewed as operators that transform one state into another. Each operator has four parts: defining form, preconditions, delete-list, add-list.

- **Envisioner** *(p.536)*: A procedure that generates all possible states for a domain. Given a finite set of blocks, it generates all legal situations, finds applicable operator instances for each state, computes resulting states, and stores the state graph as a table of transitions.

## Algorithms and Pseudocode

- **CLTMS simple example** *(p.500)*: Demonstrates `compile-formula` adding clauses, `pretty-print-clauses` showing the clause database, `complete-ltms` performing delayed consensus, and `sky-nodes` showing node labels. Two original clauses are removed and replaced by a new singleton clause after consensus.

- **add-formula procedure** *(p.502-503)*: Top-level procedure for adding formulas. Converts formula to conjunctive normal form, converts to prime implicates, and adds to clause database. Two key improvements: (1) converts formula to prime implicates before adding (avoids exponential blowup), (2) creates a temporary LTMS instance, installs all CNF clauses, copies them back to the top-level LTMS (exploits efficiency of consensus algorithm on small sets).

- **compile-formula** *(p.503)*: Transforms formula at compile time (improves execution time at expense of compile time).

- **simplify-consensus** *(p.504)*: Interleaves three primitive operations: computes consensus of two clauses, checks if result would contain complementary literals (if so, it's a tautology and is discarded), checks whether result is subsumed by current database. Uses temporary cons cells extensively. The body is written as if calling the usual cons procedure but actually uses allocated cons cells from a free set.

- **simplify-subsume-consensus** *(p.504)*: Called from the main loop of BPA. First calls `simplify-consensus` to check, and if consensus entry exists, obtains the literal sequence, checks whether result is subsumed.

- **process-clause** *(p.502)*: The LTMS procedure that handles new clauses. Calls `find-alternative-support` (checks if BCP determination exists for label), then invokes a procedure to compute additional implications.

- **propagate-unknownness** *(p.502)*: Called when a node's label becomes `:UNKNOWN`; the CLTMS procedure `propagate-more-unknownness` checks whether previously satisfied clauses need to be resolved with other clauses.

- **tae-create-node** *(p.502)*: Maintains a sequence of temporary conses used by the consensus algorithm. Recreates the sequence of conses every `fifty-node-creation` cycles.

- **remove-subsumed** *(p.507)*: Removes clauses from the connection graph and updates relevant BCP records.

- **process-clause and install-clause** *(p.507)*: Top-level processing of new clauses. `install-clause` checks if subsumed by current database; if not, calls `process-clause` which parses the clause to BCP, removes subsumed clauses, and adds to database.

- **apply-operator procedure** *(p.533-534)*: Implements the semantics of action for STRIPS operators. Takes a state (input environment) and operator instance. Produces an output environment corresponding to the resulting state. Steps: (1) instantiate add and delete lists with specific bindings, (2) transform the set of assumptions of the initial situation (assumptions for delete list are removed, assumptions for add list are ordered/inserted), (3) the environment is found (or built) by the ATMS primitive `find-or-make-env`.

- **Envisioner algorithm** *(p.536)*: (1) Generate all legal situations for the given blocks. (2) For each state S: (a) Find applicable operator instances for S; (b) For each operator instance O on S: find state T from applying O to S, install a pointer from S to T as a table of transitions.

- **find-plan procedure** *(p.537)*: Uses the envisioner to find a sequence of actions transforming a start state to a goal. Uses `fetch-states` to find candidate initial states, explores the ATMS interpretations.

- **aexia-find-plan** *(p.537)*: Uses the environment to find a sequence of actions from start to goal, with resemblance to CPS (Chapter 3). The ATMS provides a simple interface comparable to how CPS organized recognition of states.

- **Natural deduction rules for ATMS-based focused problem solver** *(p.542-543)*: CONDITIONAL ELIMINATION becomes `(rule :IMPLIED-BY (show (implies ?p ?q) :RAR ?r!) (test (not (load (living ?r))) (assume (show ?q)) (reagent! (trigger-for ?q)))`. AND INTRODUCTION, OR ELIMINATION, show-all, and show-any operations for handling conjunctive/disjunctive subgoals.

## Data Structures

- **CLTMS clause `datastructure`** *(p.501)*: Contains `clause-status` field with values `:SUBSUMED`, `:QUEUED`, `:DIRTY`, `:NOT-INDEXED`. Additional fields: `clauses` (list of all clauses), `queue` (pending clauses sorted by integer length with shortest first), `conses` (cons cells for temporary use), `delay-sat` (boolean flag), `cons-size` (integer count).

- **ATRE struct fields** *(p.518)*: `in-rules` (rules executed when triggers are jointly believed in focus), `contradiction-rules` (contradiction handlers associated with particular environments), `imp-rules` (rules executed when triggers are implied by the current focus). Additional new fields beyond basic TRE: `focus` (an environment which is the current focal point of ATRE's processing).

- **Planning problem struct** *(p.532)*: Fields include `title` (string for printing), `atre` (the ATRE it is associated with), `basis-set` (collection of choice sets defining the specific scenario), `operators` (list of operators for the domain), `plist` (property list for caching intermediate results).

- **Operator representation** *(p.530-531)*: Uses `defoperator` keyword with defining form, preconditions, delete-list, add-list. Example operators: `Pickup`, `Putdown`, `Stack`, `Unstack` — each with TEST fields for optional precondition checks.

- **Blocks World predicates** *(p.523)*: `(on ?block1 ?thing1)` — block is on top of thing; `(above ?thing1 ?thing2)` — thing1 is somewhere above thing2; `(clear ?thing)` — thing has space for a block on top; `(holding ?thing)` — being held by agent; `(hand-empty)` — canonical agent not grasping a block.

- **Choice sets for Blocks World** *(p.526)*: Three kinds needed: (1) for each block, what is it on top of; (2) for each block, what is on top of it; (3) what, if anything, is the gripper holding.

## Key Claims and Properties

- **BCP incompleteness** *(p.500)*: The CLTMS example shows CLTMS behavior on simple clauses that show BCP incompleteness — two clauses that individually don't trigger BCP but after consensus yield a singleton clause.

- **Consensus discarding optimization** *(p.504)*: Most clauses built by `simplify-consensus` are immediately discarded because they fail the subsequent subsumption check. Therefore, using temporary cons cells is efficient since most results are throwaway.

- **Prime implicates as canonical form** *(p.509)*: An implicant of a set of formulas Σ is a conjunction of literals containing no pair of complementary literals such that it is a prime implicant of Σ if no proper subset of it is an implicant. The prime implicates of Σ are the negations of the prime implicants of ¬Σ, treating Σ as a conjunction of formulas. *(p.509)*

- **ATMS operations are the same regardless of strategy** *(p.514)*: Whether using many-worlds or focused strategy, the underlying ATMS operations are the same; if the given environment can be consistently combined with the statement's label, then the extension is consistent. In focused problem solving, each union is a candidate for its focus. *(p.514)*

- **Two caveats for computing relative consistency** *(p.514)*: (1) Nogoods discovered later on may render new environments inconsistent; (2) new environments are only complete relative to the set of assumptions in force when the label is examined — adding new assumptions may enable yet more possible extensions.

- **Many-worlds strategy drawback** *(p.513)*: The cost of propagating complex labels versus repeatedly propagating simple labels — many-worlds labels grow exponentially, making the ATMS itself a liability rather than an ally. *(p.525)*

- **Focused strategy leverages ATMS** *(p.513)*: The ATMS extracts its price in advance by making the initial propagation of an assumption more expensive, but then switching becomes free (in contrast to expanding sets of assumptions that forced JTMS/LTMS to use stack-oriented techniques).

- **Clear statements are local effects of actions** *(p.525)*: The `clear` predicate points out a subtle problem — negative instances of `above` lead to the same problem as trying to define `clear`. The solution uses closed-world representation via facts that characterize the set of things above each block. *(p.525)*

- **ATMS is neutral with respect to planning strategy** *(p.525)*: By propagating information across contexts early, an ATMS provides a reusable, efficient context for world modeling.

- **Envisioning reduces planning to graph search** *(p.536)*: Once all possible situations are enumerated and linked by actions, the problem of planning is reduced to graph search.

- **Envisioning is the prototypical many-worlds ATMS problem solving strategy** *(p.536)*: It works well for small problems and can be very useful in gaining insight into the structure of a domain, but quickly becomes unmanageable for larger domains.

- **Backtracking is actually rare in focused ATMS problem solving** *(p.525)*: Only a small fraction of the potential search space ends up being explored.

- **Natural deduction rules are structurally small but impactful** *(p.544)*: Although the necessary changes to rules are syntactically small, their impact should not be underestimated. In the FTRE-based system, the use of a stack to manage assumptions was forced into a depth-first reasoning strategy.

- **Contradiction handling becomes localized and distributed** *(p.544)*: By associating contradiction handlers directly with environments, localized, distributed contradiction processing is provided (versus stack-oriented mechanisms in previous TMS architectures).

## Design Decisions and Rationale

- **Compile-formula vs. add-formula** *(p.503)*: `compile-formula` transforms at compile time (faster execution but slower compilation). `add-formula` transforms at runtime. For a few very large formulas, the compile-time approach exploits the efficiency of consensus on small sets. *(p.503)*

- **Temporary cons cells for consensus** *(p.504)*: Most consensus results are discarded after subsumption check. Using an advanced Common Lisp implementation that allows reuse of cons cells avoids allocation overhead. *(p.504)*

- **Connection graph maintenance** *(p.505)*: Clauses stored in `true-false-clauses` and `true-node-false-clauses` sorted by descending length. Indexing by length enables the shortest-first resolution strategy. *(p.505)*

- **Discrimination tree simplification** *(p.506)*: The `add-to-trie` procedure is noted as actually less sophisticated than described because unless the problem is relatively large, the additional checking of canonical order is not worth the overhead — most Common Lisps implement `member` well enough. *(p.506)*

- **Many-worlds vs. focused strategy** *(p.512-513)*: Many-worlds works in all consistent contexts at once. Focused strategy works in a single context. The leverage provided by the ATMS comes from the ease with which the focus can be changed — making initial propagation more expensive but switching free. Each strategy imposes different requirements on the inference engine and on the ATMS interface. *(p.513)*

- **Implied-by rule execution strategy** *(p.515)*: The most restrictive strategy, forcing the combinations to be implied by the current focus, and hence presumably relevant to the problem solver's current activities. In all but a few cases, a mixture of these strategies provides the most efficiency. *(p.515)*

- **Contradiction handling in ATMS-based systems** *(p.516-517)*: In non-ATMS systems, contradiction handling is defined in terms of the belief states of nodes in the dependency network. In ATMS, the current context is implicitly represented by the global state; contradiction handling must be performed through a global mechanism. Decentralizing contradiction handling creates a tight link between task processing and TMS operations. *(p.516)*

- **ATRE contradiction rules as localized handlers** *(p.518)*: Contradiction handlers associated with particular environments. The procedure `change-focus` does reality checks on the ATRE and the proposed focus environment, and if sound, makes the change. *(p.518)*

- **Choice of causally primitive predicates for Blocks World** *(p.524-525)*: `on`, `holding`, `hand-empty` are causally primitive; `above` is derived (transitive closure of `on`); `clear` is problematic — negative instances of `above` lead to complexities. Solution: use ATRE rules as constraints rather than deriving `clear` from negation. *(p.524-529)*

- **STRIPS formulation for ATMS** *(p.529-530)*: Operators use the STRIPS pattern (defining form, preconditions, delete-list, add-list) but with ATRE rules for the domain laws. The preconditions do not need to be causally primitive predicates, but delete and add list entries must be. *(p.530)*

- **defoperator with TEST field** *(p.531)*: Operators `attack` and `unstack` need optional TEST field preconditions — e.g., the block being released or placed must be on top of another block and not on the table. This supplements the precondition list. *(p.531)*

- **Focused problem solving with natural deduction** *(p.541-543)*: Adapting the FTRE-based KM* system from Chapter 8 as a starting point. The data, procedures, and inference modes of Janet were based on JTRE. The global control structure was an AND/OR tree of tasks, built from suggestions created by pattern-directed rules.

- **show-all and show-any operations** *(p.543)*: `show-all` introduces conjunctive subgoals (all must succeed); `show-any` introduces disjunctive subgoals (one must succeed). These make the set of problem solver operations extensible to allow domain-specific constructs. *(p.543)*

- **Tasks as irrelevant when not executing** *(p.544)*: Tasks covering which kind of configurations suddenly become irrelevant should be represented in a way that marks them as irrelevant even when not executing. Discussed in Section 14.4. *(p.517)*

## Implementation Notes

- **CLTMS queue structure** *(p.501)*: Queue is a list of lists, where each component list consists of an integer length followed by clauses of that length. Top-level list sorted so shortest clauses always occur first. Example: `((2 #<Clause 3> #<Clause 2>) (4 #<Clause 5>) (7 #<Clause 9> #<Clause 10)))`. *(p.501)*

- **Performance of consensus on temporary LTMS** *(p.503)*: Creating a temporary LTMS instance seems like overhead, but has two advantages: (1) unnecessary to have a distinct algorithm to convert formula to prime implicates, (2) when the database is expressed as a few very large formulas, the procedure exploits the efficiency of the consensus algorithm. *(p.503)*

- **Connection graph lazy construction** *(p.505)*: The connection graph is not consulted in the inner loop of BPA, and many of the clauses constructed there are immediately subsumed. Therefore, construction is delayed — `insert-clause` inserts the clause into the connection graph when needed (the indexing right after the clause is created in `bcp-add-clause` is suppressed). *(p.505)*

- **ATRE focus mechanism** *(p.518)*: New fields beyond basic TRE: `in-rules` (executed when triggers are jointly believed in focus), `contradiction-rules` (handlers for particular environments), `imp-rules` (executed when triggered combinations are implied by current focus). The procedure `change-focus` does reality checks. The `focus-manipulation` procedure ensures rule scheduling tracks changes in focus environments. *(p.518)*

- **ATRE assumptions** *(p.519)*: Since assumptions are permanent in the ATMS, there is no distinct notion of enabling or disabling an assumption. Instead, `assume!` provides an interface to the ATMS procedure `assume-node`, which permanently makes the node associated with the fact into an assumption. Procedures `already-assumed?` and `assume-if-needed` allow procedures to avoid redundancy in assuming facts. *(p.519)*

- **Nogood representation in ATRE ATMS** *(p.519)*: The syntax of `nogood!` is close to what was used for Horn justifications, namely an informant followed by a list of antecedents. For an N-queens puzzle solver, the consistency rule might look like: `(rule :IF (Queen ?column1 ?row1) :VAR ?Q1 (on ?column2 ?row2) :VAR ?Q2 :TEST (set (or (= ?column1 ?column2) ...))` *(p.519)*

- **ATRE rule system changes (arules.lisp)** *(p.520-521)*: Major changes stem from tracking trigger conditions during rule definition and execution. Global variables `*in-nodes*` and `*imps*` are used as registers. Rules stored in `active-in-rules` field and rules with `:IMPLIED-BY` triggers accumulated from surrounding environment. The syntax simplified by moving trigger condition outside the list of antecedents. *(p.520-521)*

- **STRIPS operators via defoperator** *(p.531-532)*: The `defoperator` macro produces a datastructure representing that operator, extracts possible fields from the given form, distributes information appropriately among fields of a new operator struct, creates an ATRE rule which detects combinations of preconditions, and justifies the appropriate `applicable` assertion for each set of statements that satisfies the operator's preconditions. *(p.532-533)*

- **apply-operator substitution via unify** *(p.533-534)*: Uses `unify` to make substitutions of values for variables. The effect of the action is implemented by transforming the list of assumptions of the initial situation. *(p.534)*

- **plan-a.lisp forward-chaining search** *(p.539)*: Contains a forward-chaining search program that does not require an envisioner in order to operate. Reduces the search space by 12 percent on a simple three-blocks scenario. *(p.539)*

- **Focused problem solving adaptation** *(p.541-542)*: Section 14.4 outlines the mechanisms needed for tight control: the organization of focus as an ATMS environment, the implied-by strategy which only executes rules implied by the current focus, and contradiction handlers which decentralize the choice of action to take when inconsistencies arise. *(p.541)*

## Quotes Worth Preserving

- "The leverage provided by the ATMS comes from the ease with which the focus can be changed. In a traditional TMS, changing assumptions requires retracting some subset of the old assumptions and enabling new ones, which can force label propagation through substantial portions of the dependency network. The ATMS extracts its price in advance, by making the initial propagation of an assumption more expensive (due to the need for label updating and maintaining the nogood database), but then switching becomes free." *(p.513)*

- "Envisioning is the prototypical many-worlds ATMS problem solving strategy. It works well for small problems, and can be very useful in gaining insight into the structure of a domain. However, it quickly becomes unmanageable." *(p.536)*

- "The ATMS is neutral with respect to the planning strategy employed. By propagating information across contexts early, an ATMS provides a reusably efficient context for world modeling." *(p.525)*

- "Although the necessary changes to the rules are syntactically small, their impact should not be underestimated. In the FTRE-based system, the use of a stack to manage assumptions forced us into a depth-first reasoning strategy. In that system, each new assumption had to be completely explored (within the given resource limitations) before exploring a different assumption. The reification of contexts as environments facilitates the use of other control strategies." *(p.544)*

- "The same vocabulary of control predicates (i.e., open, solved, and failed) used in JANET can be adapted to describe the properties and relationships of tasks in the new system. That is, unless a contradiction is being sought, any task with a contradictory focus environment can be said to have failed." *(p.544)*

- "While most ATMSs have been implemented on serial computers, the parallel nature of its computations suggests that parallel processing could be very effective... How would you organize an ATMS to most efficiently operate in a scalable multiprocessor architecture, where there may be as few as two processors and as many as one hundred?" *(p.549)*
