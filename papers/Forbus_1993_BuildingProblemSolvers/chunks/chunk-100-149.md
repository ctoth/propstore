## Key Definitions

- **TRE (Tiny Rule Engine)**: A pattern-directed inference system consisting of 5 files: `tinter.lisp` (interface), `tdata.lisp` (database), `rules.lisp` (rule system), `unify.lisp` (unification), `treex.lisp` (tests). *(p.100)*

- **dbclass (database class)**: Each database class has an associated name, the symbol it corresponds to, and the assertions and rules that belong to it (facts and rules fields respectively). The dbclass-table provides a backpointer to the TRE. *(p.102)*

- **fetch**: Operates in two stages. First it gets a candidate set by retrieving assertions associated with the database class of the pattern. Then it uses `unify` on each candidate to determine if it indeed matches the pattern. `unify` returns a set of bindings (possibly empty) when it succeeds, or the distinguished symbol `:FAIL` to indicate the match was unsuccessful. *(p.103-104)*

- **Rule structure**: Implemented via `defstruct`-defined entities. The `id` field is an integer used to distinguish rules when printing. Has several advantages: unique identity, easy comparison, debugging aid, rule creation timestamp tracking. *(p.105)*

- **KM* system**: A natural deduction system adapted from Kalish and Montague. Three advantages: (1) better organized than most, (2) natural formalism for computer-oriented formulation including assumption handling, (3) includes explicit control information. *(p.109)*

- **Proof format in KM***: A numbered set of lines. Each line has three parts: `(lineNumber) (Statement) (Justification)`. Justifications take two forms: a tag indicating the inference rule used plus line numbers referenced; or a special tag (`premise`, `ass`). *(p.110)*

- **FTRE (Fast TRE)**: Extended version of TRE adding: rule compilation, open-coding unification, context mechanism for assumptions. Consists of files: `finter.lisp`, `fdata.lisp`, `frules.lisp`, `funify.lisp`, `unify.lisp`. *(p.140)*

- **Discrimination tree**: A tree structure where leaves are assertions and rules. Each vertex represents choosing some aspect of the structure. First node discriminates on the car of the assertion, second level on the cadr, and so forth. Used for efficient rule/assertion retrieval. *(p.131)*

- **Generalized hashing**: An indexing strategy where, given a pattern, features (car, caddr, etc.) are extracted and stored in a hash table. Each feature has a value which describes a known aspect of the pattern's structure. Assertions and triggers matching a given pattern can be retrieved by computing the features, fetching the sets associated with each feature, and intersecting them. *(p.132-133)*

- **Context (in FTRE)**: The value of a variable depends on the environment in which it is evaluated. Different variable binding disciplines correspond to different environment structures and means of manipulating them. FTRE uses a stack model for logical environments. *(p.137-138)*

- **Assumption stack**: A stack providing local context in which variables can take on distinct values for a time, and then revert to their old values when the computation is finished. *(p.138)*

## Algorithms and Pseudocode

### TRE Inference Rule Encodings *(p.118-120)*
TRE encodings of KM* inference rules as Lisp `rule` forms:
- **Not Elimination**: `(rule (not (not ?p)) (assert! ?p) :not-elimination)`
- **And Elimination**: `(rule (and ?p ?q) (assert! ?p) (assert! ?q) :and-elimination)`
- **Or Elimination**: encoded via backward chaining with two sub-goals
- **Conditional Elimination**: `(rule ?a (rule ?b (assert! (implies ?a ?b)) :conditional-introduction))`
- **Biconditional Elimination**: encoded similarly
*(p.118-120)*

### Forward/Backward Chaining in KM* *(p.119-120)*
- **AND-Introduction**: `(rule (show ?a ?b) :and-introduction ...)`
- **OR-Elimination**: uses `show` to structure backward chaining
- **Back-chaining on CE** (Conditional Elimination): `(rule (show ?g) (assert! (show (implies ?a ?g))) ...)`

### FTRE Context Operations *(p.141-142)*
- `seek-in-context`: Begins by checking current depth against maximum depth. If it has not been exceeded, it pushes a new context by bumping the local state information (local-rules, local-data, normal-queue, and assumption-queue). Triggered rules in the containing environments are inherited by new environments. *(p.141)*
- `try-in-context`: Encapsulates the operations needed to prove something within a context. Returns a result indicating whether the goal was proven. *(p.142)*
- `seek-in-context` must return a result indicating whether or not the goal was proven, accomplished by setting `result` to the result of calling `fetch` on the goal before popping the context. *(p.142)*

### FTRE Rule Compilation *(p.143-148)*
- Rules are compiled into Lisp procedures for efficiency. Each rule now includes a match procedure and a body procedure.
- `generate-match-procedure`: creates the match procedure from trigger patterns. It analyzes the trigger to determine which pattern variables must be bound by the match procedure. *(p.148-149)*
- `build-rule`: arranges the indexing form, match and body procedures. *(p.147)*
- `fully-expanded-body`: walks through the body of a procedure, recursively macro-expanding it. *(p.147)*

### Speeding Up Rule Retrieval *(p.131)*
- Discrimination trees organize the database as a tree whose leaves are assertions and rules. Vertex branching is based on structural aspects (car, cadr, etc.).
- The database class technique is a one-level discrimination tree. *(p.131)*

## Data Structures

### TRE Structure *(p.101)*
- Global variable `*TRE*` holds the current TRE instance
- Fields: `title` (string for identification), `dbclass-table` (hash table for database classes), `debugging` flag, `rule-counter` (unique identifier for rules), `rules-run` (statistics counter)
- Interface procedures: `run`, `assert!`, `fetch`, `rule`, `uassert!`, `show-data`, `show-rules`, `referent` *(p.101-102)*

### Database Class (dbclass) *(p.102)*
- Fields: `name`, `symbol` (correspondence), `facts` (assertions), `rules` (rules belonging to it)
- `dbclass-table`: hash table mapping symbols to dbclass instances

### Rule Structure *(p.105)*
- Fields: `id` (integer), `dbclass` (pointer to database class of trigger), `trigger` (rule pattern), `body` (Lisp code for procedural import), `environment` (bindings from rule's defining context)

### FTRE Structure *(p.140)*
- Superset of TRE structure
- Additional fields: `depth` (current stack depth), `max-depth` (maximum assumptions allowed)
- Two queues: `normal-queue` (rules triggered that do not make assumptions), `assume-queue` (rules that make assumptions)
- `local-rules` and `local-data` for context-local storage *(p.140-141)*

### FTRE Rule Structure *(p.143-147)*
- Additional field: `assumption?` — non-nil if the rule makes an assumption *(p.144)*
- `matcher` and `body` fields hold the rule's match procedure and body procedure respectively *(p.143)*
- `*rule-procedures*` global variable accumulates procedure definitions during rule evaluation *(p.145)*

## Key Claims and Properties

- **TRE is only partial for KM* implementation**: TRE does not provide a mechanism for deleting an assumption from the database, and the assumption retraction problem is significant. If we delete an assertion, we would need to find all assertions derived from that assumption and retract them too. Multiple derivations make this "extremely difficult." A better solution involving truth maintenance systems is introduced in the next chapter. *(p.117-118)*

- **Efficiency of TRE's simple implementation**: "The only virtue of this implementation is its simplicity. It is very efficient, since translating between the two different notions of value requires creating a new pair of code every time a rule is executed." *(p.107)*

- **Pattern-directed inference systems as flexible representational medium**: "Assertions are a flexible representational medium, providing a powerful inference mechanism via pattern matching. How well does the PDIS model satisfy the desiderata for AI programming set out in Section 2.1?" *(p.121)*

- **PDIS model ranks high on explicitness**: "because the module of assertions can express advice as well as data. It ranks high on flexibility, since rules will run whenever their triggers match, rather than within some complex control structure as in a traditional system." *(p.121)*

- **One drawback of the PDIS model is efficiency**: The extra step of pattern matching increases the cost of reference. Also lacks coherence since rules can run in any order, and the "pencil and paper version of KM*" implementation does not have justifications. *(p.121)*

- **Discrimination trees vs. generalized hashing**: Discrimination trees are effective but require maintaining the tree as assertions are added. Generalized hashing is "very flexible" — for instance, fetching all Left-Of and Right-Of assertions about a specific object is straightforward. *(p.131-133)*

- **Stack-based context constraints**: Three constraints for FTRE's stack-based logical environments: (1) when making an assumption, must push a new context; (2) assertions must be stored in the most recently created context; (3) execution of rules should be arranged so conclusions are drawn in the simplest logical environments (fewest assumptions) possible. *(p.121-122)*

- **Resource bounds guarantee**: "Resource bounds guarantee that the system won't waste too much time trying to prove something it cannot, but unfortunately also ensure that there are conclusions it could in principle prove but in actuality will not." *(p.140)*

- **Rule compilation advantage**: "We can execute rules much more efficiently if we arrange for the body of a rule to be a separate procedure." Creating a procedure for each rule body has two advantages: (1) counting up an expression at rule execution time is no longer required; (2) the body procedure can simply be called with the current values of pattern variables as arguments. *(p.135-136)*

## Design Decisions and Rationale

- **Unifier returns `:FAIL` symbol**: Rather than returning two values (success flag + bindings), return a special symbol `:FAIL` to indicate failure of a match. "Since unifiers tend to be defined recursively this would lead to considerable extra work." *(p.100)*

- **Queue design is order-independent**: "We have not specified any particular order of execution for rules, only that all matching rule/assertion pairs must eventually be executed. Any strategy, including last-in first-out or first-in first-out (LIFO or FIFO), satisfies these conditions." *(p.100)*

- **Using Lisp assertions to represent propositions**: "We solve the representation problem in this context by using TRE assertions to represent propositions in a ground and by using TRE rules to encode KM*'s laws of inference. This solution is only partial, since it does not provide the explanation capabilities found in KM*'s use of line numbers and justifications." *(p.117)*

- **Why assumptions need TMS**: "This version of TRE does not provide any mechanism for solving the assumption problem. Suppose we attempted to model retraction by deleting an assumption from the database. We would still need to find a way to delete all the assertions derived from that assumption." Multiple derivations of an assertion create critical dependencies between internal events in the database. *(p.117-118)*

- **Three properties for language design** *(p.128)*:
  1. **Support common usage**: Every language has idioms, patterns of constructs used over and over again. Such patterns should be simple.
  2. **Support conciseness**: Generally the easier it is to do something, the fewer chances there are for mistakes.
  3. **Remain close to the host language**: It is all too easy to squander time and treasure trying to make the "optimal" syntax.

- **Class indexing for retrieval**: "we continue using class indexing for retrieval, albeit allowing clever organization of assertions/rules to reduce the size of the candidate set." *(p.135)*

- **Two queues in FTRE**: Normal-queue for rules that don't make assumptions; assume-queue for rules that do. "This is accomplished by using two queues: assume-queue holds triggered rules that make assumptions and normal-queue holds triggered rules that do not make assumptions." The depth field indicates current stack depth, incremented when a context is pushed, decremented when popped. *(p.140)*

- **Why nested rules need different treatment than top-level rules**: "The reason is that nested rules must only be added to the database whenever the rule that contains them is triggered. Furthermore, where they are indexed often depends on the bindings provided by the containing triggers." Top-level rules are defined before the indexing form. *(p.144-145)*

## Implementation Notes

- **TRE consists of 5 files**: `tinter.lisp` (organizing data structure and context mechanism), `tdata.lisp` (database system), `rules.lisp` (rule system), `unify.lisp` (unifier), `treex.lisp` (test examples). *(p.100)*

- **FTRE consists of 5 files**: `finter.lisp`, `fdata.lisp`, `frules.lisp`, `funify.lisp`, `unify.lisp` (same as TRE). *(p.140)*

- **Database class retrieval implementation**: When finding the database class of an assertion or pattern, it is "slightly complicated." The basic operation is to compute the database class of an assertion, use the database class table to retrieve its struct (if already created), and build it otherwise. The rename thunk gets complicated as there are two notions of binding in TRE. *(p.103)*

- **`fetch` two-pass approach**: First retrieves candidate set by database class matching; then applies `unify` to each candidate for exact matching. *(p.103-104)*

- **Open-coding unification** (`funify.lisp`): The macro `quotize` receives an expression which, when evaluated, will reproduce the constant parts of the input expression while replacing pattern variables with their bindings. `rllet` provides a means of binding pattern variables within the scope of a rule — Lisp variables and pattern variables are closely aligned. *(p.149)*

- **Pattern variable convention in FTRE**: Variables that begin with "?" are pattern variables. When creating procedures for rules, pattern variables become Lisp variables. The form returned by `rllet` is in fact a `let` expression, but with the value expressions being the pattern match bindings. *(p.149)*

- **Rule compilation creates procedures**: `generate-match-procedure` creates a `generate-match-body` function; this analyzes the trigger to determine which pattern variables must be newly bound vs. already known, and generates optimized matching code. *(p.148-149)*

- **The `run-rules` procedure**: After asserting a new assertion, runs matching rules. The procedure `try-rule-on` checks whether a rule could be triggered on an assertion, following set design in assuming that the match procedure returns a flag signaling whether or not the match is good plus a list of bindings it requires. *(p.131)*

- **Assumption-retraction problem foreshadows TMS**: The difficulty of retracting assumptions and their consequences is explicitly identified as a motivation for truth maintenance systems in the next chapter. *(p.117-118)*
