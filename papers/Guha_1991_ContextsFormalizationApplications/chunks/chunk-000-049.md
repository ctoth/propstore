### Chapter 2: The Logic of Contexts and Lifting (p.17-40)

**2.1.1 Desired Properties** (p.17-19)
1. Contexts are first-class objects; ist formulas are first-class formulas
2. Different contexts can use different languages (vocabulary depends on context)
3. FOL with extensions should suffice internally
4. Symbols referentially opaque in ist
5. Nested ist: `ist(c_i, ist(c_j, alpha))` allowed
6. Lifting rules transfer facts between contexts

**2.1.2 Syntax and Semantics** (p.20-24)
- Total Vocabulary = set of constant symbols *(p.20)*
- Language = subset of total vocab with designated constants (predicate/function/individual) and fixed arities *(p.21)*
- Context Structure CS: assigns to each context C a Language L(C) and set of structures CM(C) *(p.21)*
- Formula F meaningful in context C iff every symbol not within ist is part of L(C) *(p.21)*
- Satisfaction: `<U, CS, S>` satisfies F in C iff U in CM(C), F meaningful in C under CS, standard FOL truth recursion *(p.22)*
- ist satisfaction: `ist(ck, P)` satisfied iff Ck is context AND for every U_j in CM(Ck), P satisfied in Ck *(p.22)*
- Entailment: F entails G in C iff every satisfying tuple also satisfies G *(p.23)*
- `presentIn(c, A)` - object A exists/is present in context c *(p.23)*
- `corefer(C1, A, C2, B)` - A in C1 denotes same object as B in C2 *(p.23)*
- `value(c, A) = z` - the denotation of A in context c *(p.23)*
- CM defines DAG of contexts: `forall ci: not presentIn(ci, ci)` *(p.23)*
- Nonmonotonicity via circumscription; Cyc uses Argumentation Based Default Reasoning *(p.23)*
- Non-Axioms: `ist(c, p v q) => ist(c,p) v ist(c,q)` NOT valid *(p.24)*
- `not ist(c,p) => ist(c, not p)` NOT valid (but valid if theory is complete) *(p.24)*
- Barkan formula `ist(c (forall x p(x))) <=> (forall x ist(c p(x)))` does NOT hold *(p.24)*
- Assumption: domain of outer context larger than inner contexts *(p.24)*

**2.1.3 Proof Theory** (p.25-27)
- Proof = finite sequence with context sequences [C1, C2, ..., Cn] *(p.25)*
- Enter pushes context, Exit pops *(p.25)*
- Inference rules (a)-(l):
  - (a) All FOL rules for non-ist formulas
  - (b) Modus ponens, modus tollens, propositional rules
  - (c) Logical connective combination within contexts
  - (d) Modus ponens from outside
  - (e) Conjunction splitting
  - (f) Consistency: `ist(ci, a)` / `not ist(ci, not a)`
  - (g) Skolemization
  - (h) Modified Universal Instantiation (using value/corefer)
  - (i) Universal export from context
  - (j) Implication export
  - (k) Universal import to context (with domain assumption)
  - (l) Implication import with domain assumption

**2.2 Example Axioms** (p.28-29)
1. Dropping argument to predicate: `forall x ist(C1 Tall(x)) <=> ist(C2 Tall(x Person))` *(p.28)*
2. Dropping argument to function: cost unary in C1, binary in C2 *(p.28)*
3. Domain assumption: fixtures assumed installed in C1 but not C2 *(p.28)*
4. Fixing time: `fix-time(ci, ti)` = context snapshot at time ti *(p.29)*
5. Indexicals: `corefer(C1, Now, C2, T1)` for temporal lifting *(p.29)*

**2.3-2.4 Sample Proofs and Lifting** (p.29-34)
- Detailed proof showing Enter/Exit mechanics with Tall(Fred) example *(p.29-30)*
- Domain restriction proof with fixtures/installation *(p.30-31)*
- **Lifting** = relative decontextualization - using formulas in one context to conclude formulas in another *(p.31)*
- Two components of lifting framework *(p.32-33)*:
  - (a) **Default Coreference Rule (DCR)**: as a default, meaning of symbol doesn't change between contexts
  - (g) **Term DCR**: to state term A in context c_i denotes same thing B in c_j: `corefer(c_i, A, c_j, B)`
  - **Predicate/Function DCR**: if system knows P(x) is true in c_i, as default it would like to lift to c_j
  - `(DCR-P) forall c_i c_j P X: not ab-dcp(P c_i c_j X) => (ist(c_i P(X)) <=> ist(c_j P(X)))`
  - `ab-dcp(c_i, c_j, s, p)` is true if default violated for c_i, c_j, s, and p
  - `(DCR-F)` similar for functions
- Compositional lifting: non-atomic formulas lifted via compositional rule (CR) *(p.34)*

**2.5 Example Proof with Lifting** (p.35)
- Basketball/Tall example showing DCR-based lifting between C1 and C2 *(p.35)*
- When C2 needs more sophisticated vocabulary (professional vs backyard basketball), lifting rules adapt *(p.35)*

**2.6 Model Theoretic Account of Lifting** (p.36-38)
- Context Dependence of Utterance: theory associated with each context *(p.36)*
- Background theory B restricts models *(p.36-37)*
- **Relative Decontextualization**: determining what speaker claims - eliminate models from background theory that are inconsistent *(p.37)*
- **Restriction of a model U by a domain D**: formalized definition *(p.38)*
  - For each relation/function R: U_i contains relation iff RUD with tuple in RU and objects in tree are D
  - Model U_i eliminated from T_j if there is a model U_i' in T_j such that correspondence function CF maps relations/functions appropriately

**2.7 Discussion** (p.39-40)
- Containment as biconditional: `ist(c_i, F)` in C_i states same as formula F in C_i *(p.39)*
- Containment is stronger than biconditional *(p.40)*
- Biconditionals inadequate - use default biconditionals with circumscription *(p.40)*

### Chapter 3: The Applications of Contexts (p.41-49)

**3.1 Different Uses of Contexts** (p.41-46)
1. **Referring to group of related assertions** (Microtheories / Mts) *(p.41)*
   - (a) Theory of some topic - different microtheories make different assumptions
   - (b) Problem Solving Contexts (PSC) - tailored for specific problem
   - (c) "Very slightly decontextualized" representations (Discourse Contexts)
2. **Bundling assertions into named object** *(p.42)*
   - Assumptions reasonable only for limited domains
   - Collecting relevant subset of KB into bundle for problem solver
   - By naming theories, allow statements about utility, reliability, etc.
   - Keeping different theories distinct transforms global consistency to local consistency
3. **Combining different theories** via lifting *(p.42)*
   - Assertions not automatically available across contexts - need lifting rules
   - **The Context of a Query**: KB like Cyc includes contexts, each with own assumptions *(p.42)*
   - Query posed in Problem Solving Context that captures assumptions of question
   - Process: enter context, answer query within it, lift answer back out
4. **Limiting scope of a theory** *(p.43-44)*
   - Common-sense theories often mutually contradictory; contexts limit scope
   - "Factoring in" = selectively entering context based on current need
   - Concept of a theory being adequate for a problem (not just true/false)
5. **Natural language representation** *(p.45-46)*
   - NL utterances are contextually very compact
   - Context of utterance set up by elaborate system of parameters
   - Parameters range from coarse (cultural setting, socioeconomic background) to fine (immediately preceding utterance)
   - NL front end should parse and determine contextual aspects, assert into discourse context
   - Two requirements: (i) front end should make subset of contextual aspects explicit without inappropriate assertion; (ii) system should be able to deduce more significant assumptions
6. **For topics with polymorphous nature** *(p.46)*
   - Example: "seller" has many closely related but distinct meanings
   - Different theories with same predicate but different denotations
   - Context mechanism allows different meanings to coexist

**3.2 Lifting Rules** (p.47-48)
- One main goal of introducing contexts: simplify KB construction *(p.47)*
- **Default Coreference Rule (DCR)**: as default, same term denotes same object across contexts *(p.47)*
- **Term DCR**: `(DCR-T) forall c_i c_j: not ab-dct(S c_i c_j) => (value(c_i S) = value(c_j S))` for each term S *(p.48)*
- **Predicate/Function DCR**: similar default lifting *(p.48)*
- `ab-dcp(c_i, c_j, s, p)` true if default violated *(p.48)*
- **Compositional lifting** (CR): `forall c_i costate(c_i, c_j, q_1, q_2) ^ costate(c_i, c_j, r_1, r_2) => costate(c_i (q_1 LC r_1) c_j (q_2 LC r_2))` *(p.48)*
- Default lifting applies only to atomic formulas, not arbitrary formulas *(p.48)*
- Lifting not blindly applied: restricted from Microtheories to Problem Solving Contexts (such as CSPSC) *(p.48)*
- Conditions where lifting blocked: (a) representation of p in c_k could be different, (b) object denoted by term 'a' in c_k not in domain of c_k *(p.48)*

**3.3 Problem Solving with Contexts** (p.49)
- Two strategies for using contexts in problem solving *(p.49)*:
  - **Lift and Solve**: lift assertions from other contexts into Problem Solving Context, use conventional solver
  - **Shift and Solve**: switch into existing context, solve there, lift answer back
- Conventional problem solver = first-order solver (possibly with extensions for default reasoning) *(p.49)*

## Rules/Algorithms

### Enter/Exit Protocol (p.25)
1. Start at null context sequence []
2. Enter ci: push ci onto context sequence -> [ci]
3. Within context: apply FOL inference rules
4. Exit: pop context from sequence
5. On exit from ci with derived formula F: conclude ist(ci, F)

### Lifting Algorithm (p.31-34, 47-48)
1. Given formula F in source context c_i
2. Apply DCR-T: check if each term has same denotation in target c_j (default: yes)
3. Apply DCR-P/DCR-F: check if each predicate/function has same meaning (default: yes)
4. If ab-dcp/ab-dct not asserted, lift atomic formulas directly
5. Use CR (compositional rule) to lift non-atomic formulas
6. Block lifting when: (a) representation differs, (b) object not in target domain

### Problem Solving Strategy (p.49)
1. Receive query Q
2. Create/enter Problem Solving Context (PSC)
3. Strategy A (Lift and Solve): lift needed assertions into PSC, solve conventionally
4. Strategy B (Shift and Solve): shift into existing context, solve, lift answer back
5. Exit PSC with answer
