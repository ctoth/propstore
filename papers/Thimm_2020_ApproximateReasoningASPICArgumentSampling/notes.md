# Approximate Reasoning with ASPIC+ by Argument Sampling

## One-Sentence Summary
Presents two approximation algorithms for ASPIC+ that avoid constructing the full argument graph by sampling a subset of arguments, yielding large runtime improvements with only marginal losses in answer accuracy under grounded semantics. *(p.1-2, p.8-11)*

## Problem Addressed
Classical ASPIC+ reasoning first constructs all arguments derivable from a knowledge base and then computes acceptable arguments from the resulting abstract argumentation framework. The paper targets cases where this construction step is the bottleneck because the number of rules is large enough that enumerating every constructible argument is impractical. *(p.1-2)*

The authors explicitly care about settings where correctness is still important, but exact completeness is lower priority than getting fast, roughly correct answers together with a subgraph of arguments that can still support explanation. *(p.2)*

## Key Contributions
- Introduces the first approximation algorithms for ASPIC+ reasoning that operate by sampling arguments instead of exhaustively generating the full argumentation framework. *(p.1-2, p.11)*
- Defines an **independent sampler** `RAND_I` / `ISAMP` that samples constructible arguments without reference to a query. *(p.5-6)*
- Defines a **directional sampler** `RAND_D` / `DSAMP` that starts from the query and expands along attacker links. *(p.6-7)*
- Shows experimentally on three benchmark families that the approximate solvers retain at least 99% accuracy while often reducing runtime substantially. *(p.9-11)*
- Argues that the general sampling idea is not tied to ASPIC+ and may transfer to other structured argumentation frameworks such as ABA, DeLP, or deductive argumentation. *(p.11-12)*

## Methodology
The paper keeps the abstract argumentation semantics layer fixed and approximates only the framework-construction phase. It starts from a simplified propositional ASPIC+ setting, defines how arguments and attacks generate an abstract AF, and then replaces exhaustive argument construction with one of two sampling procedures. Acceptability of a literal is then computed on the sampled AF rather than on the complete AF. *(p.2-7)*

## Formal Background

### Abstract Semantics
An argumentation framework is a pair `(A, ↝)` where `A` is a set of arguments and `↝` is the binary attack relation. The paper recalls the usual conflict-free, admissible, complete, grounded, preferred, and stable semantics. *(p.2-3)*

An argument is acceptable with respect to a knowledge base `K` under semantics `σ` when there exists an argument `A` with `Conc(A) = l` that is accepted under `σ` in the AF corresponding to `K`. *(p.4-5)*

### Simplified ASPIC+ Instance
The paper uses a minimal propositional ASPIC+ instantiation with a finite set of propositions `L` and literals `L^c = L ∪ {¬a | a ∈ L}`. A knowledge base is:

$$
K = (K_s, K_d)
$$

where `K_s` is a set of strict rules `φ_1, ..., φ_n → ψ` with `n > 0`, and `K_d` is a set of defeasible rules `φ_1, ..., φ_n ⇒ ψ` with `n > 0`. For practical reasons the authors exclude axiom-like rules with empty bodies from the formal setup and instead treat expressions such as `a` or `¬a` as shorthand for rules with empty antecedent. *(p.3)*

### Argument Construction
Arguments are built recursively:
- A literal `φ` is an argument if `φ ∈ K`. *(p.3)*
- If `φ_1, ..., φ_n → ψ ∈ K` and there are arguments concluding each `φ_i`, then `(A_1, ..., A_n, ψ)` is a strict argument. *(p.3-4)*
- If `φ_1, ..., φ_n ⇒ ψ ∈ K_d` and there are arguments concluding each `φ_i`, then `(A_1, ..., A_n, ψ)` is a defeasible argument. *(p.3-4)*

For every argument `A`, the paper uses the standard ASPIC+-style bookkeeping functions `Prem(A)`, `Conc(A)`, `Sub(A)`, `DefRules(A)`, and `TopRule(A)`. *(p.3-4)*

### Attack Relation
The simplified framework considers only **rebuts**. If `B' ∈ Sub(B)` and `Conc(A) = ¬Conc(B')`, then `A` attacks `B`. This yields an AF:

$$
AF_K = (A_K, ↝)
$$

where `A_K` is the set of arguments generated by knowledge base `K`, and `↝` is the attack relation induced by rebuttal. *(p.4)*

### Running Example
The example knowledge base

$$
K_s = \{a \to c,\; c \to ¬d,\; d \to b\}, \quad K_d = \{a,\; b,\; c \Rightarrow d\}
$$

produces six arguments `A_1` through `A_6` and attacks `A_3 ↝ A_5`, `A_3 ↝ A_6`, and `A_6 ↝ A_4`. Under grounded, complete, stable, and preferred semantics, the accepted arguments are `{A_1, A_2, A_3, A_4}`. *(p.4)*

## Baseline Reasoning Approaches
- **Naive** exhaustively constructs every possible argument from the knowledge base by bottom-up generation from axioms and assumptions, then repeatedly combining existing arguments to build larger ones. *(p.5, p.9)*
- **MB** is a relevance-based complete baseline that prunes to a neighbourhood around the query literal or rule. It assumes that sets of arguments with distinct relevant elements do not attack each other if no attack is present between the sets, so irrelevant sets can be ignored. The paper notes this relies on the `cns*-resistance` property, which fails for stable semantics. *(p.5, p.9)*
- **DIR** is the complete version of the paper's directional idea, i.e. directional construction with sampling probability fixed to 1. *(p.9)*

## Approximation Algorithms

### Independent Sampling: `RAND_I` / `ISAMP`
The first approach samples constructible arguments independently from the query. It is parameterized by:

| Name | Symbol | Units | Default | Range | Page | Notes |
|------|--------|-------|---------|-------|------|-------|
| Maximum sampled arguments | `maxArgs` | arguments | 500 | positive integer | 5, 9 | Upper bound on retained sampled arguments |
| Maximum duplicate samples before termination | `maxDuplicates` | duplicate draws | 50 | positive integer | 5-6, 9 | Stops search if sampling keeps revisiting old arguments |

Algorithm 1 has three layers. `GENRAND_I(K, maxArgs, maxDuplicates)` repeatedly calls `SAMPLEARGUMENT(K)` until either `maxArgs` arguments are collected or duplicate sampling reaches `maxDuplicates`. *(p.6)*

`SAMPLEARGUMENT(K)` chooses random literals until one can be expanded into a valid argument. It then calls `CONSTRUCTARGUMENT(K, ψ, {ψ})`. *(p.6)*

`CONSTRUCTARGUMENT(K, ψ, acc)` chooses a random rule concluding `ψ` whose premises do not intersect the accumulated set `acc`, recursively constructs subarguments for each premise, and fails if no suitable rule exists. The accumulator prevents cycles by excluding premises already on the construction path. *(p.6)*

The sampled result is not an arbitrary bag of literals: it is an induced AF over the sampled arguments, with the attack relation recomputed on the sampled set. *(p.5-6)*

### Directional Sampling: `RAND_D` / `DSAMP`
The second approach samples arguments relative to the query. It starts from all arguments with the queried conclusion and probabilistically expands to attackers, attackers of attackers, and so on. *(p.6-7)*

The directional sampler uses one explicit parameter:

| Name | Symbol | Units | Default | Range | Page | Notes |
|------|--------|-------|---------|-------|------|-------|
| Attacker-expansion probability | `p` | probability | 0.9 | `[0,1]` | 6-7, 9 | Probability used by `FLIP(p)` when deciding whether to explore attackers |

Algorithm 2 maintains:
- `args`: currently collected arguments,
- `argsDone`: arguments whose attackers have already been processed,
- `concsDone`: conclusions already expanded to avoid regenerating the same attackers repeatedly. *(p.7)*

Operationally, for each unexpanded argument `A`, the algorithm flips a biased coin `FLIP(p)`. If the flip succeeds, it computes `ATTACKINGCONCS(A) \ concsDone`, adds those conclusions to `concsDone`, and adds all arguments with those conclusions to the working set. *(p.7)*

### Semantic Constraint
Directional sampling relies on **directionality**. The paper states that this condition is satisfied for all semantics considered except stable semantics, which is why the experiments evaluate all solvers under grounded semantics. *(p.7, p.9)*

## Experimental Setup

### Datasets
The paper uses three benchmark families. *(p.8-10)*

1. **Random**: randomly generated propositional ASPIC+ knowledge bases built from parameters `n` (propositions), `m` (rules), `l` (maximum number of literals in rule bodies), and `s` (fraction of defeasible rules). The evaluated values are `n ∈ {5,10,15}`, `m ∈ {10,20,30}`, `l ∈ {2,3}`, and `s ∈ {0.2,0.4,0.6}`, producing 5400 random instances. *(p.8)*
2. **Animals**: knowledge bases mined from the "Animals with Attributes" dataset using Apriori to produce minimal-confidence association rules, with one randomly selected animal fixed as the query. Parameters include `c ∈ {0.6,0.65,0.7,0.75,0.8,0.85,0.9,0.95}` and `s ∈ {0.6,0.65,0.7,0.75,0.8,0.85,0.9,0.95}`; the final dataset contains 1920 instances. *(p.8)*
3. **MaxSAT**: knowledge bases generated from incomplete weighted MaxSAT instances from Eval2010 by turning hard clauses into strict rules and soft clauses into assumptions and defeasible rules. After removing the seven largest instances, the final dataset contains 1490 instances. *(p.8-9)*

### Solvers Compared
- `Naive`
- `MB`
- `DIR`
- `ISAMP`
- `DSAMP` *(p.9)*

### Evaluation Protocol
- Grounded semantics was used for every solver for simplicity. *(p.9)*
- Runtime measure `RT` is the average runtime in milliseconds over solved instances. *(p.9-10)*
- Solved percentage `S` is the fraction of instances solved within a 5-minute timeout. *(p.9-10)*
- Accuracy `Acc` for the approximate solvers is measured on the subset of instances solved by at least one complete baseline (`Naive`, `MB`, or `DIR`) and records how often the approximate solver returns the same answer. *(p.9)*

## Results Summary
- All approximate methods achieved at least **99% accuracy** on every dataset. *(p.10-11)*
- On the **Random** dataset, `ISAMP` had the best runtime among the reported solvers at **336.8 ms**, compared with **476.4 ms** for the best complete solver `DIR`, while maintaining **99.1%** accuracy. *(p.10)*
- On the **Animals** dataset, `ISAMP` had the strongest runtime improvement, with **455.9 ms** average runtime versus **1881.1 ms** for baseline `MB`, again with near-perfect accuracy. *(p.10)*
- On the **MaxSAT** dataset, `DSAMP` slightly outperformed `DIR` in runtime, achieving **122.6 ms** average runtime with **100.0%** accuracy. *(p.10-11)*
- The authors attribute the MaxSAT result to the dataset's much higher fraction of strict rules and correspondingly low fraction of defeasible rules, which means most arguments have few or no attackers; in that regime, the directional attacker-driven search is especially effective. *(p.10-11)*
- The paper concludes that runtime gains are generally larger when instances contain more defeasible rules and fewer strict rules. *(p.11)*

## Implementation Details
- Keep the acceptability semantics unchanged; only approximate the **argument-generation phase**, not the semantics computation itself. *(p.5)*
- Build the sampled AF from sampled arguments and recompute attacks only within that sampled set via `ATTREL(args)`. *(p.5-7)*
- In independent sampling, use an accumulator `acc` when recursively expanding premises so the construction procedure does not loop back through already visited literals. *(p.6)*
- In independent sampling, stop early once duplicate draws exceed `maxDuplicates`; this avoids wasting time when the constructible space has effectively been exhausted. *(p.5-6)*
- In directional sampling, track both processed arguments (`argsDone`) and already expanded conclusions (`concsDone`) to prevent repeated regeneration of the same attacker frontier. *(p.7)*
- Use grounded semantics if you rely on directional expansion, because the paper's justification for the neighbourhood-style pruning depends on directionality, which the authors say does not hold for stable semantics. *(p.7, p.9)*
- If query-focused explanation matters, the directional sampler is the more natural fit because it preserves a query-centred attacker neighbourhood instead of spending budget on globally irrelevant arguments. *(p.7, p.11)*

## Limitations
- The theoretical and empirical treatment is restricted to a **simplified ASPIC+** instantiation that uses only rebuttal as the attack relation. *(p.3-4)*
- The experiments use **grounded semantics only**, so the results do not establish similar behaviour for preferred or stable semantics. *(p.7, p.9)*
- Directional sampling depends on the directionality property and is therefore not justified for stable semantics in the way used here. *(p.7)*
- The evaluation is performed on synthetic or transformed benchmark datasets rather than on full real-world deployed argumentation systems. *(p.8-10)*
- The paper gives approximation guarantees empirically, not formal error bounds on the quality of the sampled answer relative to the full AF. *(p.10-11)*

## Arguments Against Prior Work
- Exhaustive ASPIC+ reasoning is too expensive when the number of constructible arguments explodes; the paper explicitly presents complete graph construction as the practical bottleneck. *(p.1-2, p.5)*
- Existing ASPIC+ tools such as **TOAST** were not included as fair runtime baselines because they are only available as online/web services and their runtime cannot be measured comparably. *(p.9)*
- Relevance-only pruning around the query is not enough by itself for the authors' purposes; the directional sampler instead uses stochastic expansion over attacker structure to control runtime. *(p.5-7)*

## Design Rationale
- **Why sample arguments rather than change the semantics?** Because the authors want a cheap approximation of the expensive construction stage while preserving the familiar semantics layer on the sampled AF. *(p.5)*
- **Why have two samplers?** Independent sampling gives a query-agnostic baseline, while directional sampling biases effort toward arguments likely to matter for the queried conclusion. *(p.5-7)*
- **Why use `maxDuplicates` in `RAND_I`?** To stop once random sampling mostly rediscovers already seen arguments, which is a practical signal that continued search is low value. *(p.5-6)*
- **Why use probability `p` in `RAND_D`?** To trade coverage against runtime while still expanding attacker chains often enough to keep the sampled AF relevant to the query. *(p.7, p.9)*
- **Why grounded semantics in evaluation?** It keeps the comparison focused on the sampling strategy and avoids confounding the experiments with semantics that require more expensive computation or lack the relevant directionality property. *(p.7, p.9)*

## Testable Properties
- If `RAND_I` returns, the sampled AF is a subset of the full AF generated from the knowledge base, not an altered semantics on the full graph. *(p.5-6)*
- `RAND_I` terminates once `maxArgs` arguments have been sampled or the duplicate counter reaches `maxDuplicates`. *(p.5-6)*
- `RAND_D` only expands attacker conclusions not yet in `concsDone`, so the same attacker frontier should not be generated twice. *(p.7)*
- Under the paper's assumptions, the directional approximation should only be used for semantics satisfying directionality; stable semantics is excluded. *(p.7)*
- In the reported experiments, approximate methods should achieve at least 99% answer agreement with the complete baselines on the evaluated solved-instance subsets. *(p.10-11)*

## Relevance to Project
This paper is directly relevant to any propstore path that wants structured-argumentation answers and explanations without paying the full cost of exhaustive ASPIC+ construction. The key architectural pattern is to keep the semantics layer stable while making the argument-construction layer query-aware and approximate. *(p.5-7, p.11)*

For propstore specifically, `DSAMP` is the more interesting design because it preserves arguments near the query's attacker neighbourhood. That makes it a plausible fit for interactive reasoning or explanation-serving systems where the user asks about one conclusion at a time. *(p.7, p.11)*
